Divide-and-Conquer (Merge Sort), Reduction (Sum), and Amdahl's Law

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:
Data-parallelism is the technique of partitioning an independent, divisible computation (sorting, summation) across N worker threads mapped roughly to N CPU cores, executing the partitions concurrently, then performing a strictly sequential combination step — with the achievable speedup mathematically capped by however much of the algorithm cannot be parallelized (Amdahl's Law).

The Architecture Analogy

Think of a tax-processing office. You can hire 16 clerks (cores) and give each a stack of 1/16th of the forms (chunking) — that part scales beautifully. But at the end, one supervisor still has to walk through all 16 completed stacks in order to file the final consolidated report (the sequential merge/reduce step). No matter how many clerks you hire, that supervisor's walk-through time is a hard floor on how fast the whole office can finish. Amdahl's Law is just the formula for "how much does hiring more clerks actually help, given that fixed supervisor bottleneck."

How the JVM Sees It

  • Each new Thread(...).start() call causes the JVM to request a native OS thread (1:1 mapping on the platform-thread model you've been using so far) — the OS scheduler, not the JVM, actually distributes these across physical cores.
  • Runtime.getRuntime().availableProcessors() doesn't ask the JVM — it queries the OS/container cgroup for the number of logical cores visible to the process. This is why the number can lie in containerized environments (a container capped at 2 CPUs on a 64-core host may still report 64 unless cgroup-aware JVM flags are set).
  • thread.join() under the hood parks the calling thread (removing it from the OS run queue) and relies on the JMM's happens-before guarantee: everything the joined thread did before terminating is visible to the joining thread after join() returns — no extra volatile/synchronization needed for the merge step to safely read the sorted sub-arrays.
  • Recursive calls (sort(low, middle), sort(middle+1, high)) consume stack frames — each frame holds low, high, middleIndex as local variables. This is why deep, naive recursion on huge arrays without a base-case cutoff (e.g., insertion sort under N=10) risks StackOverflowError territory, though in practice O(log N) depth for merge sort is very shallow (a 1-billion-item array is only ~30 levels deep).

2. Course Transcript Content (The "What")

Key Concepts & Definitions

  • Merge Sort: divide-and-conquer, comparison-based, stable, O(N log N), not in-place — requires O(N) auxiliary memory (the temp array). Invented by John von Neumann (1945).
  • Divide phase: recursively split the array at the midpoint until each sub-array has a single element (sorted by definition) — optionally switch to insertion sort below a threshold (~5–10 elements) to cut down recursive call overhead, since insertion sort runs near-linear on small/near-sorted inputs.
  • Conquer/Merge phase: linear-time merge of two sorted sub-arrays using three indices (i for left, j for right, k for result) — copy the smaller head element, then flush whatever remains in either side.
  • Parallel Merge Sort strategy: create only as many threads as availableProcessors() (not one thread per sub-array) — halve the thread budget at each recursive level (numThreads / 2) until it drops to ≤1, at which point fall back to sequential merge sort for that sub-array. The merge step itself is always sequential.
  • Parallel Sum strategy: split the array into ceil(length / numThreads)-sized contiguous chunks, hand each chunk to a ParallelWorker thread that computes a partial sum, join() all workers, then sequentially add up the partial sums.
  • Amdahl's Law: Speedup(N) = 1 / ((1 - P) + P/N), where P = parallelizable fraction, N = processor count. The non-parallelizable fraction (1 - P) creates a hard asymptotic ceiling on speedup as N → ∞.

Code Snippets (from Transcript) — annotated

Sequential merge — the core combine step:

private void merge(int low, int middleIndex, int high) {
    for (int i = low; i <= high; i++) {
        tempArray[i] = nums[i];              // O(N) copy into scratch buffer
    }

    int i = low, j = middleIndex + 1, k = low;

    while (i <= middleIndex && j <= high) {
        if (tempArray[i] < tempArray[j]) {
            nums[k] = tempArray[i];
            i++;
        } else {
            nums[k] = tempArray[j];
            j++;
        }
        k++;
    }

    while (i <= middleIndex) { nums[k] = tempArray[i]; k++; i++; }   // flush left remainder
    while (j <= high)        { nums[k] = tempArray[j]; k++; j++; }   // flush right remainder
}
  • JVM/memory-level view: tempArray[i] = nums[i] and the reverse write-back are plain heap array accesses — no synchronization is needed here within a single thread's recursive call chain because each recursive invocation owns a disjoint [low, high] range; there's no shared mutable state being raced on inside one thread's subtree.
  • The while loops are branch-heavy — on real CPUs this creates a branch-prediction-dependent bottleneck; that's part of why an efficient, well-tuned quicksort often edges out merge sort in practice despite equal Big-O, since quicksort's partition step is more branch-predictable and cache-friendly on random data.

Parallel merge sort — thread creation and halving the budget:

private Thread createThread(int lowIndex, int highIndex, int numThreads) {
    return new Thread(() -> parallelMergeSort(lowIndex, highIndex, numThreads / 2));
}

public void parallelMergeSort(int lowIndex, int highIndex, int numThreads) {
    if (numThreads <= 1) {
        mergeSort(lowIndex, highIndex);   // fall back to sequential — no more cores to hand off to
        return;
    }

    int middleIndex = (lowIndex + highIndex) / 2;

    Thread leftSorter  = createThread(lowIndex, middleIndex, numThreads);
    Thread rightSorter = createThread(middleIndex + 1, highIndex, numThreads);

    leftSorter.start();
    rightSorter.start();

    try {
        leftSorter.join();
        rightSorter.join();
    } catch (InterruptedException e) {
        e.printStackTrace();
    }

    merge(lowIndex, middleIndex, highIndex);   // sequential — must wait for both children
}
  • Critical JVM mechanic: leftSorter.join() followed by rightSorter.join() is what creates the happens-before edge. Without it, merge() could read nums[] positions that another thread hasn't finished writing — a textbook data race and a visibility bug, not just a logic bug. The JMM guarantees that everything written by leftSorter before it terminates is visible to the current thread after join() returns, without needing volatile on nums.
  • Real cost you're paying: every single call to createThread spins up a brand-new native OS thread, which involves a syscall, a fresh stack allocation (default ~512KB–1MB per thread), and OS scheduler registration. This transcript's design (recursively halving numThreads) caps total thread count at roughly 2 * availableProcessors() - 1, which is fine — but it's fragile: if you forgot the numThreads <= 1 base case, you'd recursively spawn threads all the way down to single-element sub-arrays, potentially creating millions of OS threads for a 10M-item array and crashing with OutOfMemoryError: unable to create new native thread.

Parallel sum — chunking with Math.ceil, and the off-by-one the instructor hit live:

class ParallelWorker extends Thread {
    private final int[] nums;
    private final int lowIndex;
    private final int highIndex;
    private int partialSum = 0;

    ParallelWorker(int[] nums, int lowIndex, int highIndex) {
        this.nums = nums;
        this.lowIndex = lowIndex;
        this.highIndex = highIndex;
    }

    @Override
    public void run() {
        for (int i = lowIndex; i < highIndex; i++) {
            partialSum += nums[i];
        }
    }

    public int getPartialSum() { return partialSum; }
}

public int sum(int[] nums) {
    int size = (int) Math.ceil((double) nums.length / numThreads);

    for (int i = 0; i < numThreads; i++) {
        int lowIndex = i * size;
        int highIndex = Math.min(nums.length, (i + 1) * size);   // guards last-chunk overrun
        workers[i] = new ParallelWorker(nums, lowIndex, highIndex);
        workers[i].start();
    }

    try {
        for (ParallelWorker worker : workers) worker.join();
    } catch (InterruptedException e) {
        e.printStackTrace();
    }

    int total = 0;
    for (ParallelWorker worker : workers) total += worker.getPartialSum();
    return total;
}
  • The bug the instructor hit: (i + 1) * size for the last chunk can exceed nums.length (e.g., 6 items / 4 threads → size=2, last chunk's high = 8, which is out of bounds). Math.min(nums.length, ...) is the fix — a small but classic off-by-one in chunk boundary math, extremely common when hand-rolling data partitioning instead of using a library like Fork-Join that handles this for you.
  • Visibility note: partialSum is a plain int field, not volatile, and that's fine here — because the only reader (getPartialSum()) is called from the main thread strictly after worker.join() returns. The happens-before edge from join() makes an explicit volatile redundant. If you ever read partialSum from a different, non-joining thread while the worker might still be running, you'd have a genuine visibility bug and would need volatile or an AtomicInteger.

Important Caveats/Warnings (explicitly from the transcript)

  • Merge sort is not in-place — O(N) extra memory is a real, named disadvantage versus heap sort.
  • The merge/reduce step is always sequential — this is stated repeatedly and is the seed for Amdahl's Law later in the same section.
  • Do not create one thread per sub-array/chunk — only create as many threads as availableProcessors(), because OS context-switching between threads is expensive (state save/restore).
  • Parallel sum is slower than sequential sum for small-to-medium N (demonstrated at 100 items, 10M items, even 100M items) — thread creation/coordination overhead dominates until you reach roughly the billion-item scale, where parallel finally wins (~2x). This is a direct, measured illustration of Amdahl's Law/overhead trade-offs, not a hard rule of thumb to memorize as a fixed threshold.
  • Integer overflow bites at the 1-billion-item sum (32-bit int accumulator) — a correctness bug, not a concurrency bug, but worth flagging since it appeared in the same demo (fix: long accumulator).

3. Senior-Level Practical Implementation & Examples (The "How")

Real-World Use Cases

  1. Financial risk aggregation / EOD batch jobs: Summing exposures, P&L deltas, or position sizes across millions of trade records partitioned by instrument or desk — structurally identical to the parallel-sum pattern, just swapping int sums for BigDecimal/double aggregations with proper rounding semantics.
  2. Log/metrics pre-aggregation before a reduce stage: Splitting a large in-memory batch of log lines or metric samples across worker threads to compute partial counts/sums/histograms before merging — the same divide → independent work → sequential combine shape as both examples here, and the direct conceptual ancestor of what Stream.parallel() and Fork-Join do for you automatically later in this course.
  3. Image/computer-vision feature pipelines (explicitly mentioned in transcript): summing pixel-intensity regions for Haar-like features in classical CV pipelines — latency-sensitive, high-volume numeric reduction, exactly the kind of workload where the crossover point favoring parallelism (as demonstrated at the ~1 billion mark) actually matters in production.

Complete Code Implementation

A production-hardened version of the parallel sum, addressing gaps the lecture left open (overflow, no thread-pool reuse, no defensive checks):

import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicLong;

public class ProductionParallelSum {

    private final int numThreads;

    public ProductionParallelSum(int numThreads) {
        if (numThreads < 1) throw new IllegalArgumentException("numThreads must be >= 1");
        this.numThreads = numThreads;
    }

    /**
     * Uses a bounded, reusable ExecutorService instead of raw Thread creation —
     * avoids the OS thread-creation cost on every call, and gives us
     * structured shutdown + timeout handling.
     */
    public long sum(int[] nums) throws InterruptedException, ExecutionException {
        if (nums.length == 0) return 0L;

        ExecutorService pool = Executors.newFixedThreadPool(numThreads);
        int size = (int) Math.ceil((double) nums.length / numThreads);
        var futures = new java.util.ArrayList<Future<Long>>();

        try {
            for (int i = 0; i < numThreads; i++) {
                int lowIndex = i * size;
                int highIndex = Math.min(nums.length, (i + 1) * size);
                if (lowIndex >= highIndex) continue; // fewer items than threads

                futures.add(pool.submit(() -> {
                    long partial = 0L;                    // long, not int -> avoids overflow
                    for (int idx = lowIndex; idx < highIndex; idx++) {
                        partial += nums[idx];
                    }
                    return partial;
                }));
            }

            long total = 0L;
            for (Future<Long> f : futures) {
                total += f.get(30, TimeUnit.SECONDS);      // bounded wait, not an unbounded join()
            }
            return total;

        } finally {
            pool.shutdown();                               // always release pool resources
            if (!pool.awaitTermination(5, TimeUnit.SECONDS)) {
                pool.shutdownNow();                         // graceful, then forceful cleanup
            }
        }
    }
}

Why these changes matter at a senior level:

  • ExecutorService + Future<Long> replaces raw Thread/join() — reusable pool, proper timeout via Future.get(timeout, unit) instead of an unbounded join() that can hang forever if a worker deadlocks or infinite-loops.
  • long accumulator directly fixes the overflow bug the instructor's own demo hit at 1 billion items.
  • finally { pool.shutdown(); ... } — a raw-Thread implementation has no equivalent leak risk (threads terminate on their own), but an ExecutorService will leak non-daemon threads forever if you forget to shut it down — a very common production incident source.

Testing Concurrent Code

  • Correctness under concurrency: run the parallel and sequential implementations on the same randomly generated array (as the instructor did) and assertEquals the results across dozens of randomized trials and array sizes (0, 1, small, and multi-million element cases) — this catches boundary bugs like the Math.min chunk overrun.
  • Timeout-guarded execution: wrap the parallel call in assertTimeoutPreemptively(Duration.ofSeconds(5), () -> ...) (JUnit 5) so a hung worker thread fails the test suite fast instead of blocking the build indefinitely.
  • Stress/fuzz for chunk-boundary math: parametrize tests across numThreads values that don't evenly divide the array length (e.g., array of 7 items over 3 threads) — this is exactly where off-by-one chunking bugs like the one shown live in the transcript.
  • Deterministic thread-count control: don't rely on availableProcessors() in your unit tests — inject the thread count explicitly (as this implementation's constructor does) so tests are reproducible across CI machines with different core counts.

4. Deep Dive: Tradeoffs, Architecture, and Performance (The "Why")

Performance Implications

  • Thread creation cost dominates at small N. The lecture's own numbers make this concrete: at 100 and even 10M items, parallel sum was slower than sequential; only at ~1 billion items did parallel pull ahead (~2x). Each new Thread().start() is a real OS-level cost (stack allocation + scheduler registration) that a tight sequential loop simply doesn't pay.
  • Context-switching overhead compounds this: with numThreads capped at availableProcessors(), you get low switching overhead; if you'd instead spawned one thread per array element (the naive-but-tempting approach), the OS scheduler would burn most of its time swapping thread state in and out rather than doing sum/compare work.
  • GC pressure differs meaningfully between the two algorithms in this section: merge sort's tempArray is a single O(N) allocation reused across the whole sort — low GC churn. Sum's ParallelWorker objects are small, short-lived, and proportional to numThreads (not N) — negligible GC impact either way. If you'd naively created one worker object per chunk of chunks or per array element, you'd generate significant young-gen churn.
  • Cache locality: contiguous chunking (as both implementations do) is deliberately cache-friendly — each thread's for-loop walks sequential memory, maximizing L1/L2 cache-line reuse. A naive strided partition (thread i takes every i-th element) would be far worse for cache performance, even though it's "equally parallel" on paper.

Tradeoff Analysis Matrix: Sequential vs. Hand-Rolled Parallel (this lecture's style) vs. ExecutorService-based

Criteria Sequential Hand-rolled Thread/join() (as taught) ExecutorService-based
Throughput (large N, e.g. >100M items) Low High High (comparable)
Throughput (small/medium N) Best — no overhead Worse — thread creation dominates Better than raw threads (pool reuse), still worse than sequential at tiny N
Latency predictability Very predictable Variable — OS scheduler dependent More predictable via Future.get(timeout)
Memory overhead Minimal One stack per thread, created & destroyed per call Stacks amortized — pool reused across calls
Ease of use / Risk Trivial, no race risk Manual join() ordering, easy to forget error handling, thread-count bugs Slightly more boilerplate, but structured lifecycle (shutdown, timeouts) reduces risk

Common Pitfalls (Senior Level)

  • Visibility/Ordering: The correctness of both algorithms in this section hinges entirely on join()'s happens-before guarantee. If a future refactor replaced join() with, say, a shared boolean done flag checked in a busy-wait loop without volatile, you'd introduce a genuine JMM visibility bug — the reading thread could spin forever on a stale cached value of done, or read partially-written sub-array data.
  • Deadlock/Livelock/Starvation: These specific implementations are structurally deadlock-free (no thread waits on a resource another thread it's not related to holds — it's a simple fork/join tree), but the pattern is fragile: if you introduced any shared lock inside ParallelWorker.run() (e.g., to update a shared counter), you'd need to audit lock-acquisition order across all worker threads to avoid the classic circular-wait deadlock condition.
  • Resource Exhaustion: The single biggest risk in the taught implementation — uncapped/uncareful recursive thread creation. If the numThreads <= 1 base case in parallelMergeSort were removed or miscalculated (e.g., always passing the same numThreads instead of halving), you'd recursively spawn threads down to single-element sub-arrays: for a 10M-item array that's up to ~2×10M thread creations, guaranteed OutOfMemoryError: unable to create new native thread well before that.
  • Chunk-boundary arithmetic bugs: The exact bug the instructor hit live (ArrayIndexOutOfBoundsException from (i+1) * size overrunning nums.length) is one of the most common real-world bugs in hand-rolled data partitioning — always guard the last chunk explicitly, or better, use a library primitive that does it for you.

Architectural Impact

  • This lecture's "N threads = N cores, sequential combine" pattern is the conceptual seed of the Fork-Join framework you'll formalize soon — Fork-Join automates exactly this halving-recursion-and-join shape via a work-stealing pool, removing the manual thread-count-halving and join()-ordering bookkeeping you did by hand here.
  • At a system-design level, this pattern only pays off for CPU-bound, in-memory, embarrassingly-parallel workloads on a single JVM/machine. It does not generalize to distributed batch processing (multiple machines) — for that scale, the same divide/conquer/reduce shape reappears as MapReduce/Spark-style architectures, but with network partitioning, fault tolerance, and shuffle costs replacing simple Thread.join().
  • Amdahl's Law is the reason horizontal scaling strategies increasingly favor reducing the sequential fraction of a pipeline (e.g., replacing a single sequential merge/reduce step with a tree-reduction across workers, or batching the reduce step) rather than simply adding more compute — a core consideration in designing any high-throughput microservice or data pipeline.

5. Interview Questions & Answers

Basic

Q1: Why is merge sort described as "not in-place," and what's the practical cost of that?
A: Merge sort's merge step needs to compare elements from two already-sorted sub-arrays and write them into a separate buffer in sorted order — you can't safely do this by overwriting the original array in place while still reading from it. This costs O(N) auxiliary memory (the temp array), which is the standard trade-off cited against heap sort (same O(N log N) time, O(1) space) or quicksort (in-place, typically faster in practice on random data).

Q2: Why does the parallel implementation cap thread count at availableProcessors() instead of one thread per sub-array/chunk?
A: Because OS thread creation and context-switching are expensive — each thread needs a native stack, kernel bookkeeping, and scheduler registration. Beyond the number of physical/logical cores, additional threads don't add real parallelism (there's nowhere new to run them); they only add scheduling overhead. Matching thread count to core count keeps every thread doing real work rather than waiting in a run queue.

Q3: In the parallel sum implementation, why is partialSum allowed to be a plain (non-volatile, non-atomic) int field?
A: Because it's only written by its own worker thread and only read by the main thread after calling worker.join(). The JMM guarantees a happens-before relationship across join() — everything the worker wrote before terminating is visible to the joining thread afterward. No additional synchronization is required for this specific access pattern.

Intermediate

Q4: The lecture showed parallel sum being slower than sequential sum at 10 million items, but faster at 1 billion. Why does that crossover exist, and how would you find it for a real workload instead of guessing?
A: At smaller N, the fixed cost of creating and joining threads (each a syscall + stack allocation) outweighs the actual per-element work saved by parallelizing a simple += loop, which is already extremely cheap per element. As N grows, the per-thread payoff (real CPU-bound work) eventually dwarfs the fixed thread-management overhead. Rather than guessing a static threshold, you'd benchmark with something like JMH across a range of N values on target hardware, and/or make the algorithm choose sequential vs. parallel dynamically based on N and availableProcessors() — a technique the Fork-Join framework's default thresholding does automatically.

Q5: Compare this hand-rolled Thread+join() approach to using an ExecutorService for the same parallel-sum task. When would you actually prefer raw threads?
A: An ExecutorService amortizes thread-creation cost across many calls (pool reuse), gives you bounded waits via Future.get(timeout) instead of unbounded join() that can hang forever if a worker deadlocks or infinite-loops, and centralizes lifecycle management (shutdown()/awaitTermination()). Raw threads are reasonable only for a one-shot, short-lived program (like this classroom demo) where pool reuse buys nothing and the extra ExecutorService boilerplate isn't worth it. In any long-running service, the pool-based approach is the correct default.

Q6: The instructor hit an ArrayIndexOutOfBoundsException from (i + 1) * size overrunning the array on the last chunk. What class of bug is this, and how do you defend against it systematically rather than patching one instance?
A: It's an off-by-one/boundary bug in manual data partitioning — extremely common whenever array length isn't evenly divisible by thread count. Beyond the immediate Math.min(nums.length, ...) fix, defend against it systematically by writing parametrized tests that specifically use array lengths not evenly divisible by the thread count (e.g., 7 items / 3 threads), and by preferring a well-tested partitioning utility (or the Fork-Join framework's automatic splitting) over hand-rolled chunk math wherever the codebase already has access to one.

Advanced

Q7: Walk through exactly which JMM guarantee makes it safe for the main thread to call merge() immediately after leftSorter.join(); rightSorter.join(); without any explicit synchronized block or volatile field on nums.
A: Thread.join() has a specified happens-before relationship in the JMM: all actions performed by a thread before it terminates happen-before any other thread successfully returns from a join() on that thread. So every write leftSorter and rightSorter made to nums[] (within their respective [low, high] ranges) is guaranteed visible to the main thread once both join() calls return — without this, per-core CPU caches could hold stale copies of nums[] and the merge could read garbage/half-written values. This is exactly analogous to how volatile writes happen-before subsequent volatile reads, just scoped to thread lifecycle instead of a single field.

Q8: The parallel merge sort recursively halves numThreads at every level. What happens if you remove the numThreads <= 1 base case (or introduce a bug where it's always passed the same value instead of halved), and how would this manifest differently for a 100-item array versus a 100-million-item array?
A: Without the base-case cutoff, parallelMergeSort would keep spawning two new native threads at every recursion level, all the way down to single-element sub-arrays — for an array of length N that's roughly 2N thread creations. For 100 items, you'd likely just see terrible performance (thread overhead massively dwarfing the trivial sort work) but probably not crash. For 100 million items, you'd almost certainly hit OutOfMemoryError: unable to create new native thread well before completion, since the JVM/OS has a hard ceiling on live native threads (typically low thousands depending on stack size and available virtual memory) — a textbook resource-exhaustion failure mode distinct from a logic bug.

Q9: Given Amdahl's Law, if profiling shows a pipeline is 90% parallelizable, what's the theoretical maximum speedup as core count → ∞, and what senior-level architectural lever would you actually pull to do better than that ceiling?
A: With P = 0.9, as N → ∞, Speedup → 1 / (1 - P) = 1 / 0.1 = 10x — no matter how many cores you throw at it, you cannot exceed a 10x speedup while that 10% sequential fraction remains fixed in size. The correct lever isn't "add more cores" — it's reducing the absolute size of the sequential portion itself (e.g., replacing a single flat sequential reduce/merge with a parallel tree-reduction across a hierarchy of workers, batching I/O in the sequential segment, or restructuring the algorithm so more of the combine step can also run concurrently, as merge sort partially allows by parallelizing merges at different tree levels rather than only the leaves). This reframes Amdahl's Law from a hard limit into a target for architectural redesign, which is also the direction Gustafson's Law pushes back on (worth mentioning if the interviewer probes further: Gustafson's Law argues that as problem size grows, the parallelizable fraction tends to grow too, softening Amdahl's pessimism for scalable workloads).

6. Mastery Checklist

  • I can explain why merge sort requires O(N) auxiliary memory and why that specifically rules out true in-place sorting for this algorithm.
  • I can trace, frame by frame, how merge sort's recursive divide phase uses the call stack, and where/why the base case triggers unwinding and merging.
  • I can design a thread-count-bounded parallel divide-and-conquer algorithm (capping at availableProcessors(), halving the budget at each recursive level) instead of naively spawning one thread per sub-problem.
  • I can explain precisely which JMM happens-before guarantee (Thread.join()) makes it safe to read another thread's writes without volatile or explicit synchronization, and identify when that guarantee does not apply (e.g., non-joining readers).
  • I can identify and fix off-by-one chunk-boundary bugs in manual data-partitioning code (Math.min guard on the last chunk).
  • I can articulate why hand-rolled parallel algorithms underperform sequential ones at small/medium N, and can migrate a raw-Thread implementation to a reusable, timeout-bounded ExecutorService-based one.
  • I can state Amdahl's Law from memory, compute a theoretical speedup ceiling given a parallelizable fraction P, and explain — at a senior/architectural level — why "reduce the sequential fraction" is the correct lever rather than "add more cores" once you're near that ceiling.
