Java Multithreading: Threads Manipulation

Principal Architect Notes — Production-Grade Mastery Series

Syllabus Coverage: Lectures 10–19 (Sequential Processing · Runnable · Thread Class · join() · JVM Threads · Speed Analysis · Daemon Threads · Thread Priority)

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:

Thread manipulation in Java is the act of explicitly creating, starting, coordinating, and classifying threads — using Runnable, the Thread class, join(), sleep(), daemon flags, and priority hints — to move a program from its default sequential behavior into controlled concurrent execution.

The Architecture Analogy

Imagine a film production set. The director (main thread) kicks off the day by calling "action." In a low-budget sequential shoot, the director personally does everything: set up camera, act, do lighting, edit — one after another, nothing moves until the previous task is done.

A professional production assigns crew members (worker threads) to specialized roles: the camera operator runs independently, the sound engineer runs independently, the lighting technician runs independently. The director shouts "action" (thread.start()) for each crew member and they all work concurrently.

But here's the catch: the director cannot call "cut" and move to the next scene (join point) until the camera operator finishes their shot. That "waiting for the camera operator" is exactly what thread.join() does — it makes the calling thread wait until the target thread finishes.

Daemon threads are like the catering crew — they're there only to support the main production crew. The moment the last actor (user thread) goes home, the catering crew packs up immediately too. Nobody waits for them. If you stay to set them up before the actors arrive (setDaemon(true) before start()), they automatically leave when the last actor does.

Thread priority is like a studio memo saying "give the lead actor scheduling preference." In theory, the lead gets priority. In practice, if the union rules (OS scheduler) override the memo, the extra actor might get called first anyway.

How the JVM Sees It

Creating a thread: When new Thread(runnable) is called, the JVM creates a Java Thread object on the heap. At this point, no OS thread exists — it's just a Java object in NEW state.

Starting a thread: thread.start() calls the native method Thread.start0(), which triggers pthread_create() on Linux / CreateThread() on Windows. The OS allocates a new kernel thread, assigns it a kernel thread ID, and places it in the OS scheduler's ready queue. The JVM allocates a private stack for it (default 512KB–1MB). The start() call returns immediately on the calling thread — it does not wait.

run() vs start(): Calling run() directly bypasses all of the above. It's a plain Java method call on the current thread. Zero OS thread creation. Zero concurrency. This is the single most common beginner mistake.

join(): Under the hood, join() uses OS-level thread synchronization — specifically a wait/notify mechanism on the Thread object's monitor. The calling thread parks itself (WAITING state, zero CPU) until the target thread transitions to TERMINATED, at which point the OS wakes the waiting thread and returns it to RUNNABLE.

sleep(ms): Calls the native Thread.sleep0(), which tells the OS to park the current thread for the specified duration. The thread enters TIMED_WAITING state — zero CPU consumption. After the duration, the OS scheduler returns the thread to RUNNABLE. Critically, sleep() does not release any locks the thread holds (important for later lectures on synchronization).

Daemon flag: Tracked as a boolean field on the Thread object in the JVM. At JVM shutdown decision time, the JVM counts only non-daemon (isDaemon() == false) threads. If that count reaches zero, the JVM calls System.exit() internally, terminating all remaining daemon threads immediately — mid-execution, no cleanup.

2. Course Content: Key Concepts & Definitions

The Three Thread Creation Approaches

Java gives you three syntactically different ways to define what a thread executes. They all result in the same thing: a Runnable passed to a Thread object.

Approach 1: Implement Runnable (Preferred)

// Step 1: Define the task
class Runner implements Runnable {
    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            System.out.println("Runner: " + i);
        }
    }
}

// Step 2: Wrap in Thread and start
Thread t = new Thread(new Runner());
t.start();

Runnable is a functional interface — it has exactly one abstract method (run()). This means it can be replaced with a lambda or anonymous inner class. The @Override annotation is not mandatory but is strongly recommended — it lets the compiler catch typos where you accidentally write Run() instead of run().

Why this approach is preferred: Your class can still extend another class. Java doesn't allow multiple inheritance of classes, so if you extend Thread, you've spent your one inheritance slot. Implementing Runnable keeps the class hierarchy clean — the task (what to do) is decoupled from the execution mechanism (the Thread).

Approach 2: Anonymous Inner Class

Thread t = new Thread(new Runnable() {
    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            System.out.println("Anonymous: " + i);
        }
    }
});
t.start();

Useful for short, one-off tasks defined at the point of use. IntelliJ will flag this as "can be replaced with lambda expression."

Approach 3: Lambda Expression (Concise, Modern)

// Runnable is a functional interface, so a lambda works directly
Thread t = new Thread(() -> {
    for (int i = 0; i < 10; i++) {
        System.out.println("Lambda: " + i);
    }
});
t.start();

// Or store the Runnable first, then pass it
Runnable task = () -> {
    for (int i = 0; i < 10; i++) {
        System.out.println("Stored lambda: " + i);
    }
};
new Thread(task).start();

Lambda is the idiomatic modern Java style. The compiler infers that the lambda implements Runnable.run(). Zero class boilerplate.

Approach 4: Extend Thread Class (Use Sparingly)

class Runner extends Thread {
    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            System.out.println("Extended Thread: " + i);
        }
    }
}

// Now the runner IS a thread — no wrapping needed
Runner r = new Runner();
r.start();

Since Runner is a Thread, you call start() directly on it. The Thread class itself implements Runnable — its default run() does nothing, which is why you must override it.

When to avoid: You've used your one inheritance slot. If Runner later needs to extend Vehicle or AbstractService, you can't. Prefer implements Runnable.

thread.start() vs thread.run() — The Most Critical Distinction

Thread t = new Thread(() -> {
    System.out.println("Executing on: " + Thread.currentThread().getName());
});

t.run();    // Output: "Executing on: main"
            // ↑ No new thread. run() is a plain method call on the current thread.
            // Zero OS thread creation. Sequential. Indistinguishable from a regular method call.

t.start();  // Output: "Executing on: Thread-0"
            // ↑ New OS thread created. pthread_create() called internally.
            // The lambda executes on the new thread. True concurrency.

This will never throw an exception if you call run() by mistake. The code looks like it works. The output looks normal. Only the thread name reveals the bug. Under load, the "work" you thought was concurrent is actually sequential — a latent performance bug.

Thread.sleep(ms) — Pausing Execution

class Runner extends Thread {
    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            System.out.println(Thread.currentThread().getName() + ": " + i);
            try {
                Thread.sleep(1000);   // pause THIS thread for 1 second
                                      // Other threads continue running unaffected
            } catch (InterruptedException e) {
                // Thread was interrupted while sleeping
                // ALWAYS restore the interrupt flag — covered below
                Thread.currentThread().interrupt();
                return;  // exit cleanly
            }
        }
    }
}

What sleep() does at the OS level: The current thread is moved from RUNNING to TIMED_WAITING. The OS scheduler removes it from the run queue for the specified duration. Zero CPU. After the duration, the thread is returned to RUNNABLE and waits for a CPU slot.

Why InterruptedException must be handled: Any thread can call interrupt() on a sleeping thread to signal it should stop (used for graceful shutdown — covered in later lectures). InterruptedException is the notification. Always either re-throw it or call Thread.currentThread().interrupt() to preserve the interrupt signal for callers. Never silently swallow it with an empty catch block.

thread.join() — Waiting for Another Thread to Finish

public static void main(String[] args) throws InterruptedException {
    Thread downloader = new Thread(() -> downloadImages(), "image-downloader");
    Thread processor  = new Thread(() -> processImages(), "image-processor");

    downloader.start();
    // processor.start() called immediately after — but processImages()
    // requires the download to be done first! This would be a bug:
    processor.start();  // ← starts before download finishes

    // FIX: wait for downloader before starting processor
    // (shown in full example below)
}

The dependency problem: start() returns immediately. The main thread does not wait. If Task B depends on Task A's output, blindly starting both threads simultaneously produces incorrect results.

join() solves this:

public static void main(String[] args) {
    Thread downloader = new Thread(() -> downloadImages(), "image-downloader");
    downloader.start();

    try {
        downloader.join();   // main thread parks here (WAITING state, zero CPU)
                             // until downloader thread reaches TERMINATED state
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return;
    }

    // Only reaches here after downloader has finished
    Thread processor = new Thread(() -> processImages(), "image-processor");
    processor.start();
}

Joining multiple threads:

t1.start();
t2.start();
t3.start();

// Main thread waits for ALL three before continuing
t1.join();
t2.join();
t3.join();

System.out.println("All threads done. Continuing main.");
// This line only executes after t1, t2, AND t3 have all terminated

Key behavior: join() can be called from any thread on any other thread — not just from main. Thread A can join() on Thread B; Thread C can join() on Thread A. The join point creates an explicit ordering guarantee.

The Main Thread

Every Java program starts with exactly one thread: the main thread, assigned automatically by the JVM to execute the main(String[] args) method. You cannot run a Java application without it.

public static void main(String[] args) {
    System.out.println("Thread: " + Thread.currentThread().getName());    // "main"
    System.out.println("Priority: " + Thread.currentThread().getPriority()); // 5
    System.out.println("Daemon: " + Thread.currentThread().isDaemon());   // false
}

The main thread is a user thread with default priority (5). Child threads spawned from it inherit its daemon status by default (i.e., they are also non-daemon unless explicitly set). The JVM will not exit while the main thread or any other user thread is still alive.

Threads Inside the JVM — What's Actually Running

Use Thread.getAllStackTraces().keySet() to see all threads the JVM is managing:

for (Thread t : Thread.getAllStackTraces().keySet()) {
    System.out.printf("%-30s | %-15s | daemon=%-5s%n",
        t.getName(), t.getState(), t.isDaemon());
}

Typical output for a simple two-thread application:

main                           | RUNNABLE        | daemon=false
Thread-0                       | RUNNABLE        | daemon=false
Thread-1                       | RUNNABLE        | daemon=false
Notification Thread            | RUNNABLE        | daemon=true
Finalizer                      | WAITING         | daemon=true
Reference Handler              | RUNNABLE        | daemon=true
Common-Cleaner                 | WAITING         | daemon=true

The JVM background daemon threads:

Thread Name Purpose
Finalizer Runs finalize() methods on objects before GC reclaims them
Reference Handler Processes WeakReference, SoftReference, PhantomReference objects
Common-Cleaner Modern replacement for finalizers — processes Cleaner actions
Notification Thread JVM internal event notification (JMX, management)

All of these are daemon threads. They exist only to support user threads. When all user threads exit, these are killed immediately — which is exactly correct behavior. You don't want the JVM to stay alive just because the garbage collector is mid-cycle.

The insight: You can never run a Java application without threads. Even "single-threaded" code runs on the main thread with several daemon threads active in the background.

Daemon Threads vs. User Threads

Dimension User Thread Daemon Thread
Default? Yes — all threads are user threads by default Must be set explicitly before start()
JVM exit behavior JVM waits for all user threads to finish JVM terminates daemon threads immediately when no user threads remain
Typical role Core application work: request handling, computation, business logic Background support: GC, monitoring, logging, housekeeping
API Default (setDaemon(false)) setDaemon(true) — must be called before start()
Examples Your Runner threads, web server request handlers JVM's Finalizer, Reference Handler, timer callbacks 
Thread worker = new Thread(() -> {
    while (true) {
        monitor();           // background monitoring loop
        try { Thread.sleep(1000); } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            break;
        }
    }
}, "health-monitor");

worker.setDaemon(true);   // MUST be called before start()
worker.start();

// When all user threads finish, the JVM will kill "health-monitor" immediately
// No cleanup code in the daemon thread's loop will run after this point

Critical rule: setDaemon(true) must be called before start(). Calling it after start() throws IllegalThreadStateException. This is because the daemon status is baked into the OS thread at creation time.

Practical implication for daemon thread design: Because daemon threads can be killed at any moment without warning when the JVM exits, they must never:

  • Hold file handles or database connections that need explicit closing
  • Be in the middle of writing to a file (partial writes)
  • Hold locks (locks would be released abruptly, leaving shared state inconsistent)

Use daemon threads only for stateless or best-effort background work where abrupt termination is acceptable.

Thread Priority

// Priority range: Thread.MIN_PRIORITY (1) to Thread.MAX_PRIORITY (10)
// Default: Thread.NORM_PRIORITY (5)

Thread low    = new Thread(task, "low-priority-thread");
Thread normal = new Thread(task, "normal-priority-thread");
Thread high   = new Thread(task, "high-priority-thread");

low.setPriority(Thread.MIN_PRIORITY);    // 1
normal.setPriority(Thread.NORM_PRIORITY); // 5 (default — same as not setting it)
high.setPriority(Thread.MAX_PRIORITY);   // 10

low.start();
normal.start();
high.start();
// Actual execution order: non-deterministic despite the priorities

Why priority is a hint, not a guarantee: Java's thread scheduler is not a part of the JVM — it is the OS scheduler. The JVM passes priority hints to the OS, but different OSes interpret them differently:

  • Windows: Maps Java's 1–10 scale to Windows thread priority classes. Honors priorities more than other OSes — you may see some effect.
  • Linux: The default scheduler (CFS — Completely Fair Scheduler) largely ignores Java thread priorities. All threads get fair CPU time regardless.
  • macOS: Similar to Linux — priorities are suggestions at best.

Thread starvation: If high-priority threads always run, low-priority threads might never execute — called starvation. Java's scheduler design intentionally avoids pure priority scheduling to prevent this. Even a low-priority thread will eventually get CPU time.

The practical rule: Do not design correctness or ordering guarantees around thread priority. If thread A must run before thread B, use join(), not priority. Priorities are only useful as a loose performance hint to the OS — and even then, test your assumption on the target OS before relying on it.

Is Multithreading Faster Than Sequential? (The Real Answer)

This is one of the most common interview questions and the most misunderstood topic for beginners.

On a single-CPU machine:

Sequential:
 CPU: [===Task A===][===Task B===]
 Total time = time(A) + time(B)

Multi-threaded (same single CPU, time-slicing):
 CPU: [=A=][=B=][=A=][=B=][=A=][=B=]
           ↑     ↑     ↑
      context context context
       switch  switch  switch
 Total time = time(A) + time(B) + (n × context_switch_cost)

Conclusion for single CPU: multithreading is always slower than sequential due to context-switch overhead. The CPU does the same total work, plus extra overhead on every thread switch.

On a multi-CPU machine (what modern hardware actually is):

Sequential (still uses only 1 core):
 Core 0: [===Task A===][idle        ]
 Core 1: [idle        ][===Task B===]

Multi-threaded (OS assigns threads to separate cores):
 Core 0: [===Task A===]
 Core 1: [===Task B===]
 Total time ≈ max(time(A), time(B))   — parallel speedup!

Conclusion for multi-CPU: multithreading can be dramatically faster — wall-clock time approaches the duration of the longest task, not the sum of all tasks.

But even on multi-core hardware, multithreading isn't always faster:

Scenario Multithreading helps? Why
Tasks are independent, long-running, CPU-bound ✅ Yes Each task runs on its own core
Tasks are tiny (microseconds each) ❌ No Thread creation + context-switch overhead dominates
Algorithm is inherently sequential (each step depends on previous) ❌ No Can't parallelize — no independent sub-tasks
I/O-bound tasks (network, disk waits) ✅ Yes Waiting threads park (zero CPU), other threads run
Too many threads for available cores ❌ Diminishing returns Context-switch overhead increases; performance curves down

The correct mental model: Multithreading's value is not always raw speed — it's also concurrency (responsiveness: keeping the UI alive while background work runs) and resource utilization (keeping CPU busy during I/O waits). These benefits exist even on a single CPU.

3. Senior-Level Practical Implementation

Complete Working Example: All Thread Creation Patterns

package com.example.threads;

public class ThreadCreationPatterns {

    // Pattern 1: Named class implementing Runnable (preferred for complex tasks)
    static class DataProcessor implements Runnable {
        private final String dataSource;

        DataProcessor(String dataSource) {
            this.dataSource = dataSource;
        }

        @Override
        public void run() {
            for (int i = 0; i < 5; i++) {
                System.out.printf("[%s] Processing batch %d%n",
                    Thread.currentThread().getName(), i);
                try {
                    Thread.sleep(200);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();   // restore interrupt flag
                    System.out.println(Thread.currentThread().getName() + " interrupted — stopping");
                    return;
                }
            }
        }
    }

    public static void main(String[] args) throws InterruptedException {

        // Pattern 1: Class implements Runnable
        Thread t1 = new Thread(new DataProcessor("db-source"), "db-processor");

        // Pattern 2: Lambda expression (for simple tasks)
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 5; i++) {
                System.out.printf("[%s] Step %d%n",
                    Thread.currentThread().getName(), i);
                try { Thread.sleep(150); } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                }
            }
        }, "lambda-worker");

        // Pattern 3: Extends Thread (less preferred)
        Thread t3 = new Thread("extending-thread") {
            @Override
            public void run() {
                System.out.println("[" + getName() + "] Running");
            }
        };

        System.out.println("=== Starting threads ===");
        t1.start();
        t2.start();
        t3.start();

        // Join: wait for all to complete before printing summary
        t1.join();
        t2.join();
        t3.join();

        System.out.println("=== All threads finished ===");
    }
}

Real-World Use Case: Image Pipeline with Thread Coordination

The course's download-then-process example, fully implemented with correct join() coordination:

import java.util.ArrayList;
import java.util.List;

public class ImagePipeline {

    // Shared state (will be protected by synchronization in later lectures;
    // for now we use join() to ensure no concurrent access)
    static List<String> downloadedImages = new ArrayList<>();

    public static void main(String[] args) {

        // Stage 1: Download images (worker thread)
        Thread downloader = new Thread(() -> {
            System.out.println("[downloader] Starting download...");
            for (int i = 1; i <= 5; i++) {
                try {
                    Thread.sleep(300);   // simulate network latency per image
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                }
                downloadedImages.add("image_" + i + ".jpg");
                System.out.println("[downloader] Downloaded image_" + i + ".jpg");
            }
            System.out.println("[downloader] All images downloaded.");
        }, "image-downloader");

        // Stage 2: Process images — MUST wait for download to complete
        Thread processor = new Thread(() -> {
            System.out.println("[processor] Starting processing...");
            for (String image : downloadedImages) {   // safe: only runs after join()
                System.out.println("[processor] Applying filters to: " + image);
                try { Thread.sleep(100); } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                }
            }
            System.out.println("[processor] Processing complete.");
        }, "image-processor");

        downloader.start();

        // ← Critical join point. Without this, processor starts immediately
        //   and downloadedImages is still empty.
        try {
            downloader.join();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return;
        }

        processor.start();   // safe to start now — download is guaranteed done

        try {
            processor.join();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return;
        }

        System.out.println("[main] Pipeline complete. " + downloadedImages.size() + " images processed.");
    }
}

Real-World Use Case: Daemon Thread for Health Monitoring

public class ApplicationServer {

    public static void main(String[] args) throws InterruptedException {

        // Daemon: background health monitor — should not prevent JVM shutdown
        Thread healthMonitor = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                System.out.println("[health-monitor] Heap used: " +
                    (Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory()) / 1024 / 1024 + "MB");
                try {
                    Thread.sleep(2000);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
        }, "health-monitor");

        healthMonitor.setDaemon(true);   // Must be before start()
        healthMonitor.start();

        // User thread: core application work
        Thread requestHandler = new Thread(() -> {
            System.out.println("[request-handler] Handling requests...");
            try {
                Thread.sleep(5000);   // simulate 5 seconds of request handling
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
            System.out.println("[request-handler] Done.");
        }, "request-handler");

        requestHandler.start();
        requestHandler.join();   // main thread waits for user work to complete

        System.out.println("[main] All user threads done. JVM will now exit.");
        // health-monitor daemon thread is killed here automatically — no action needed
    }
}

Listing All Active JVM Threads (Diagnostic Utility)

public class ThreadDiagnostics {

    public static void printAllThreads() {
        System.out.printf("%-35s | %-15s | %-10s | %s%n",
            "Thread Name", "State", "Priority", "Daemon");
        System.out.println("-".repeat(80));

        Thread.getAllStackTraces().keySet()
              .stream()
              .sorted((a, b) -> Boolean.compare(a.isDaemon(), b.isDaemon()))
              .forEach(t -> System.out.printf("%-35s | %-15s | %-10d | %s%n",
                  t.getName(), t.getState(), t.getPriority(), t.isDaemon()));
    }

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            try { Thread.sleep(5000); } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }, "my-worker");
        worker.start();

        Thread.sleep(100);   // let threads initialize
        printAllThreads();

        worker.join();
    }
}

This is invaluable during development to understand what the JVM is actually managing, and in production as a first-pass diagnostic before reaching for jstack.

4. Deep Dive: Tradeoffs, Architecture & Performance

Performance: Context-Switch Cost in Detail

Every context switch involves the CPU:

  1. Saving current thread's PC register, stack pointer, and all general-purpose registers
  2. Updating the thread's state in the OS kernel
  3. Loading the next thread's saved registers
  4. Flushing/invalidating CPU pipeline
  5. L1/L2 cache cold-start: the new thread's working data must be fetched from RAM into cache

On modern x86 hardware with HotSpot JVM: ~1–10 microseconds per switch, plus cache miss penalties that can add another 100–300 nanoseconds per cache line miss. For tasks that take milliseconds, this overhead is negligible. For tasks that take microseconds (tight loops, simple arithmetic), context-switch cost can exceed the actual work cost — making multithreading strictly slower.

Tradeoff Matrix: Thread Creation Approaches

Approach Flexibility Code Clarity Best For
implements Runnable High (can extend other classes) Clear separation of task/thread Complex tasks, reusable logic
Lambda expression High Concise, modern Simple one-off tasks, small closures
Anonymous inner class High Verbose, dated Legacy code; use lambda instead
extends Thread Low (inheritance used up) Clear but limiting Simple scripts, never in production libs

The join() Guarantee: What It Does and Doesn't Provide

join() provides a strict happens-before guarantee per the JMM: everything a thread does before it terminates is visible to the thread that calls join() after join() returns.

List<String> results = new ArrayList<>();   // shared state

Thread worker = new Thread(() -> {
    results.add("computed value");  // write
});
worker.start();
worker.join();   // join() guarantees the write above is visible here
System.out.println(results.get(0));  // "computed value" — safe, no synchronization needed

This is one of the few ways in Java to safely share data between threads without explicit synchronization — by ensuring threads don't overlap in time (the writer finishes completely before the reader starts). This pattern breaks down the moment multiple threads access shared data concurrently — that's when you need synchronized, volatile, and locks (coming in the next section).

Common Pitfalls at This Stage

1. Calling run() Instead of start()

Thread t = new Thread(() -> System.out.println("On: " + Thread.currentThread().getName()));
t.run();    // prints "On: main" — no new thread, sequential execution
t.start();  // prints "On: Thread-0" — new OS thread created

Silent bug. No exception. Never do this.

2. Calling setDaemon() After start()

Thread t = new Thread(task, "monitor");
t.start();
t.setDaemon(true);  // IllegalThreadStateException — too late

Daemon status must be set before the OS thread is created. Always place setDaemon() before start().

3. Swallowing InterruptedException

// WRONG — kills the interrupt signal permanently
try { Thread.sleep(1000); } catch (InterruptedException e) { /* ignore */ }

// CORRECT — restore the flag so callers can observe the interruption
try { Thread.sleep(1000); } catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}

Swallowing InterruptedException prevents graceful shutdown mechanisms from working. This becomes critical once you reach thread pools and ExecutorService (lecture 45+), where interrupts are the primary shutdown signal.

4. Forgetting That Main Thread Doesn't Wait for Child Threads

public static void main(String[] args) {
    new Thread(() -> {
        try { Thread.sleep(2000); } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        System.out.println("Worker done");
    }).start();

    System.out.println("Main done");   // prints FIRST — main doesn't wait
    // JVM stays alive because the worker (user thread) is still running
}

If you need the main thread to wait, you must call join() explicitly. Main does not implicitly wait.

5. Depending on Thread Priority for Ordering

// WRONG: This is not a reliable ordering mechanism
highPriorityThread.setPriority(Thread.MAX_PRIORITY);
lowPriorityThread.setPriority(Thread.MIN_PRIORITY);
highPriorityThread.start();
lowPriorityThread.start();
// "highPriorityThread runs first" is not guaranteed — OS may ignore priorities entirely

If order matters, use join(). If coordination matters, use synchronization primitives (upcoming lectures).

Architectural Impact

Why this section matters for what comes next: Every concept introduced in the upcoming section (synchronization, locks, wait()/notify()) exists because of one fact established here: threads share the heap, and join() is the only coordination tool introduced so far. join() is too coarse for most real-world scenarios — it completely serializes two threads at a point in time. What you need are finer-grained coordination mechanisms that let threads cooperate while both running — that's exactly what synchronization provides.

The thread creation patterns (Runnable vs extends Thread) established here are permanent. Even when you move to thread pools (ExecutorService in lecture 45) and CompletableFuture (lecture 119), you're still ultimately submitting Runnable or Callable tasks — the abstraction just manages the Thread lifecycle for you.

5. Interview Questions & Answers

Basic

Q1: What are the three ways to create a thread in Java and which is preferred?
Three approaches: (1) Implement the Runnable interface and override run(), then pass the instance to new Thread(runnable). (2) Use a lambda expression since Runnable is a functional interface — new Thread(() -> { ... }). (3) Extend the Thread class and override run(), then call start() on the subclass instance. The implements Runnable approach (or lambda) is strongly preferred in production because it doesn't consume the class's single inheritance slot — your task class can still extend a domain class. Extending Thread should be reserved for cases where you genuinely need to customize the thread's behavior itself, not just the task it runs, which is rare.
Q2: What is the difference between thread.start() and thread.run()?
start() creates a new OS thread via pthread_create() (Linux) or CreateThread() (Windows), registers it with the JVM scheduler, and executes run() on that new thread. start() returns immediately on the calling thread — it does not wait. run() is a plain Java method call — it executes on the calling thread with zero OS thread creation and zero concurrency. The code appears to run but does so sequentially, making it functionally identical to calling any other method. This is a silent bug: no exception is thrown, output looks normal, and the mistake only reveals itself as a performance failure under load. Always use start().
Q3: What is a daemon thread and how do you create one?
A daemon thread is a background thread that the JVM will not wait for when shutting down. When all non-daemon (user) threads finish, the JVM terminates immediately and kills any remaining daemon threads — even if they are mid-execution. Create one by calling thread.setDaemon(true) before thread.start() — calling it after start() throws IllegalThreadStateException. Daemon threads are appropriate for background support tasks (monitoring, GC, logging) where abrupt termination is acceptable. They must never hold resources that need explicit cleanup (file handles, DB connections) because their cleanup code will not run on JVM exit. All JVM internal threads (Finalizer, Reference Handler, GC threads) are daemon threads.

Intermediate

Q4: You have two tasks where Task B must use the results of Task A. You run them as separate threads. What is wrong with this code and how do you fix it?
Thread a = new Thread(() -> computeResult());
Thread b = new Thread(() -> useResult());   // depends on computeResult()'s output
a.start();
b.start();  // BUG: b starts immediately, result may not exist yet
The bug is that start() returns immediately and both threads begin concurrently. Thread B may start reading the result before Thread A has written it, producing incorrect behavior — or in the best case, an empty/null result. Fix: call a.join() between starting A and starting B. join() parks the calling thread in WAITING state (zero CPU) until Thread A terminates. The JMM guarantees that everything Thread A wrote before terminating is visible to the thread that called join() after it returns. This makes it safe to start Thread B immediately after a.join() without any additional synchronization.
Q5: Is multithreading always faster than sequential execution? Justify your answer.
No — and the answer depends on three factors. On a single-CPU machine, multithreading is always slower than sequential because the same CPU does the same total work plus context-switch overhead (saving/restoring registers, cache invalidation) on every thread switch. On a multi-CPU machine, multithreading can be dramatically faster for independent, long-running tasks because they run truly in parallel on separate cores — wall-clock time approaches the duration of the longest task rather than the sum of all tasks. However, even on multi-core hardware, multithreading is slower when tasks are too small (context-switch cost dominates), too many threads compete for too few cores (scheduling overhead rises), or the algorithm is inherently sequential (no independent sub-tasks to parallelize). Multithreading's value is also not purely about speed — keeping a UI responsive while background work runs, or keeping a CPU busy during I/O waits, are valid reasons to use threads even when raw throughput doesn't improve.
Q6: Why is thread priority unreliable in Java and what should you use instead?
Thread priority in Java is a suggestion to the OS scheduler, not a command. The JVM translates Java's 1–10 scale into OS-level hints, but different operating systems interpret them differently: Windows honors Java priorities more directly, while Linux's CFS (Completely Fair Scheduler) largely ignores them — all threads receive approximately equal CPU time regardless of priority. Additionally, the JVM intentionally avoids pure priority scheduling to prevent starvation — a thread with priority 1 will eventually run even if priority-10 threads are present. The result is that execution order based on priority is non-deterministic across OSes and JVM implementations. For ordering: use join(). For mutual exclusion: use synchronized or ReentrantLock (upcoming lectures). Thread priority can serve as a loose performance hint on specific OSes but should never be the basis of correctness logic.

Advanced

Q7: Explain the join() happens-before guarantee and when it makes synchronization unnecessary.
The JMM specifies that a thread's termination happens-before the return of join() in any other thread. This means all writes performed by the joining thread — to any variable, shared or not — are guaranteed visible to the thread that called join(), without any synchronized, volatile, or atomic operations. Practically: if Thread A writes data and terminates, and Thread B calls a.join() before reading that data, Thread B is guaranteed to see the writes. This is safe specifically because there is no temporal overlap — A has finished before B reads. This guarantee breaks down entirely if both threads were running concurrently. The join() pattern effectively serializes threads at a point in time, creating a non-overlapping access window that eliminates data visibility concerns. This is why it's appropriate for pipeline-style threading (download then process) but insufficient for producer-consumer patterns where threads must genuinely run simultaneously and share data while both active — that requires synchronized or locks.
Q8: A production service runs 50 background monitoring daemon threads. The JVM is slow to start up. What is wrong and how would you diagnose and fix it?
Each thread, daemon or not, requires OS resources at creation: a kernel thread descriptor, stack memory (512KB–1MB by default), and initialization in the JVM's thread table. Fifty threads at startup means 50 pthread_create() calls, ~25–50MB of native stack memory, and significant OS scheduling overhead — all before the application has served a single request. Diagnosis: measure startup time before and after adding monitoring threads; use jstack on the running JVM to see 50 daemon threads all in TIMED_WAITING (sleeping in their monitoring loops) — this shows they're doing nothing useful at startup. Fix: (1) Consolidate multiple monitoring concerns into fewer threads — a single monitoring thread can check heap, CPU, connection counts, and log them together. (2) Delay daemon thread startup until after the application is warm — use a scheduled check that begins after a startup grace period. (3) Consider whether 50 independent threads are the right architecture at all — a single scheduled task running on a timer thread (itself a daemon) can poll multiple metrics in sequence. In production, creating threads has a cost that must be justified by necessity, not convenience.
Q9: What happens to daemon threads' state (open files, database connections, held locks) when the JVM kills them on exit?
The JVM terminates daemon threads by calling the underlying OS thread termination mechanism without running any further Java code in those threads. This means: (1) No finally blocks execute in the daemon thread — any resource cleanup code in finally is skipped. Open file streams, database connections, and network sockets become leaked from Java's perspective. The OS will reclaim the file descriptors when the process exits, but this bypass any application-level cleanup (flushing buffers, committing transactions, sending close handshakes). (2) Held locks are released abruptly — if a daemon thread holds a synchronized lock and is killed mid-critical-section, the monitor is released by the JVM but the shared data it was protecting may be in an inconsistent intermediate state. Any user thread that subsequently acquires that lock will see corrupted state. (3) Partial writes to shared data — a daemon thread in the middle of writing a multi-field update to a shared object is killed mid-update; readers see partial state. This is why daemon threads must be designed for stateless or best-effort work only. Never use a daemon thread for transaction management, file writing, or any operation where mid-stream termination produces corrupt state.

6. Mastery Checklist

After internalizing these notes, you should be able to:

  • Write a thread using all three approaches (implements Runnable, lambda, extends Thread) and correctly explain the JVM/OS operations triggered by each
  • Explain precisely what happens at the OS level when thread.start() is called vs. thread.run() — and why the latter is a silent bug that won't throw an exception
  • Implement Thread.sleep() correctly, including the mandatory InterruptedException handling pattern (restoring the interrupt flag, not swallowing it)
  • Use thread.join() to create explicit ordering guarantees between threads, explain the JMM happens-before guarantee it provides, and identify when join() makes additional synchronization unnecessary
  • Distinguish daemon threads from user threads on every axis: JVM exit behavior, appropriate use cases, when setDaemon() must be called, and what cleanup guarantees do NOT exist for daemon threads
  • Use Thread.getAllStackTraces() to list active JVM threads, identify the JVM's built-in daemon threads (Finalizer, Reference Handler, Common-Cleaner), and explain their purpose
  • Explain thread priority's 1–10 scale, why it is OS-dependent and non-deterministic, why setPriority() should never be used for ordering guarantees, and what to use instead
  • Answer the "is multithreading faster than sequential?" question correctly for both single-CPU and multi-CPU scenarios, naming the specific costs (context-switch overhead, cache invalidation) that make multithreading slower in certain conditions
  • Identify the three production pitfalls in this section (calling run() not start(), setDaemon() after start(), swallowing InterruptedException) and write the correct version of each
  • Name all threads in production code and articulate specifically what value named threads provide during a thread dump incident diagnostic
  • Explain why join() is insufficient for producer-consumer patterns where threads must share data while both are running, and identify that this is the motivation for the synchronization mechanisms in the next section
