Java Multithreading: Deadlock & Livelock

Principal Architect Notes — Production-Grade Mastery Series

Syllabus Coverage: Lectures 35–36 + Livelock Intro/Example (Deadlock Theory · Deadlock Example · Livelock Theory · Livelock Example)

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:
A deadlock is a permanent standstill where two or more threads each hold a lock the other needs and neither will ever release what it holds, while a livelock is the more deceptive sibling — threads stay active and keep changing state in response to each other, yet never make real progress.

The Architecture Analogy

Deadlock is two delivery trucks meeting nose-to-nose in a one-lane alley, each carrying cargo the other needs at its destination. Truck A holds Package 1, needs Package 2. Truck B holds Package 2, needs Package 1. Neither driver will back up. Both engines are off. Nothing moves, forever, until someone forcibly tows one truck away (kills one thread/process).

Livelock is the narrow hallway scenario the course describes precisely: Alice and Bob, walking toward each other, both politely try to let the other pass. Alice steps left, Bob steps right — they're now facing each other on the same side again. Both step back. Both step again. Engines are running, both people are actively moving, but neither has covered an inch of forward progress. The crucial difference from deadlock: nobody is stuck or blocked — they're both burning energy reacting to each other in an endless loop.

How the JVM Sees It

Deadlock at the JVM level: Two or more threads enter a BLOCKED state simultaneously, each waiting on a monitor (or ReentrantLock) held by the other. The JVM's thread scheduler has nothing to schedule for these threads — they remain in BLOCKED indefinitely, consuming zero CPU but permanently unable to proceed. The JVM has a built-in deadlock detector accessible via ThreadMXBean.findDeadlockedThreads() and visible in jstack output, which explicitly prints "Found one Java-level deadlock" with the full cycle of thread → lock → holder relationships.

Livelock at the JVM level: Threads remain in RUNNABLE/RUNNING state the entire time — they are actively executing instructions (calling tryLock(), checking conditions, releasing locks, retrying). CPU usage can actually be high during a livelock, which makes it more dangerous to diagnose than deadlock — jstack won't show threads stuck in BLOCKED; it'll show threads cycling through RUNNABLE repeatedly, which looks like "the application is working" until you notice zero actual task completion.

2. Course Content: Key Concepts & Definitions

Deadlock — Definition and Anatomy

A deadlock occurs when two or more threads are each waiting for a resource (typically a lock) held by another thread in the group, and none of them will ever release what they hold. All threads in the cycle are stuck permanently.

The classic two-thread, two-lock cyclic dependency:

// Thread 1 (worker1): acquires lock1, then tries lock2
// Thread 2 (worker2): acquires lock2, then tries lock1
// ↑ This is the cyclic dependency that causes deadlock
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class Deadlock {
    private final Lock lock1 = new ReentrantLock(true);
    private final Lock lock2 = new ReentrantLock(true);

    public void worker1() {
        lock1.lock();
        System.out.println("worker1 acquires lock1");
        try { Thread.sleep(300); } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        lock2.lock();                                    // ← worker1 waits for lock2
        System.out.println("worker1 acquires lock2");
        lock2.unlock();
        lock1.unlock();
    }

    public void worker2() {
        lock2.lock();
        System.out.println("worker2 acquires lock2");
        try { Thread.sleep(300); } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        lock1.lock();                                    // ← worker2 waits for lock1
        System.out.println("worker2 acquires lock1");
        lock1.unlock();
        lock2.unlock();
    }

    public static void main(String[] args) {
        Deadlock deadlock = new Deadlock();
        new Thread(deadlock::worker1, "worker1").start();
        new Thread(deadlock::worker2, "worker2").start();
    }
}

Execution trace:

worker1 acquires lock1
worker2 acquires lock2
[application hangs here forever]

worker1 holds lock1, sleeps 300ms, then tries to acquire lock2 — but worker2 already holds it. worker2 holds lock2, sleeps 300ms, then tries to acquire lock1 — but worker1 already holds it. Both threads are now permanently BLOCKED. The application does not crash, does not throw an exception — it simply hangs forever.

The Fix: Consistent Lock Ordering

public void worker2() {
    lock1.lock();                  // ← changed: acquire lock1 FIRST (same order as worker1)
    System.out.println("worker2 acquires lock1");
    try { Thread.sleep(300); } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
    lock2.lock();                  // ← then lock2 — matching worker1's order
    System.out.println("worker2 acquires lock2");
    lock2.unlock();
    lock1.unlock();
}

Output after the fix:

worker1 acquires lock1
worker1 acquires lock2
worker2 acquires lock1
worker2 acquires lock2

By making both methods acquire lock1 before lock2, the cyclic dependency is eliminated. The thread that gets lock1 first will be able to proceed to lock2 uncontested (since the other thread is still waiting for lock1). This is the single most important structural deadlock-prevention rule: always acquire multiple locks in a globally consistent order, everywhere in the codebase.

Livelock — Definition and Anatomy

A livelock occurs when two or more threads continuously change their state in response to each other's actions, but never converge to a state of actual progress. Unlike deadlock, the threads are not blocked — they are actively running, just stuck in an unproductive loop.

The hallway analogy mapped to code: Both threads try to be "polite" — if they can't get both locks they need, they release what they have and retry, rather than waiting indefinitely. If both threads do this in a perfectly synchronized rhythm, they can keep "stepping aside" for each other forever.

import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class LivelockExample {
    private final Lock lock1 = new ReentrantLock(true);
    private final Lock lock2 = new ReentrantLock(true);

    private void worker1() {
        while (true) {
            try {
                if (lock1.tryLock(50, TimeUnit.MILLISECONDS)) {
                    System.out.println("worker1 acquires lock1");
                    try {
                        if (lock2.tryLock(50, TimeUnit.MILLISECONDS)) {
                            System.out.println("worker1 acquires lock2");
                            lock2.unlock();
                            break;       // ← got both locks: done, exit the loop
                        } else {
                            System.out.println("worker1 cannot acquire lock2");
                            // releases lock1 and retries from scratch — this is the livelock trigger
                        }
                    } finally {
                        lock1.unlock();
                    }
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                return;
            }
        }
    }

    private void worker2() {
        while (true) {
            try {
                if (lock2.tryLock(50, TimeUnit.MILLISECONDS)) {
                    System.out.println("worker2 acquires lock2");
                    try {
                        if (lock1.tryLock(50, TimeUnit.MILLISECONDS)) {
                            System.out.println("worker2 acquires lock1");
                            lock1.unlock();
                            break;
                        } else {
                            System.out.println("worker2 cannot acquire lock1");
                        }
                    } finally {
                        lock2.unlock();
                    }
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                return;
            }
        }
    }

    public static void main(String[] args) {
        LivelockExample example = new LivelockExample();
        new Thread(example::worker1, "worker1").start();
        new Thread(example::worker2, "worker2").start();
    }
}

Why this livelocks: worker1 acquires lock1, tries lock2 — fails because worker2 holds it. worker1 releases lock1 and retries. Meanwhile worker2 acquires lock2, tries lock1 — fails because worker1 (re)acquired it. worker2 releases lock2 and retries. If both threads' timing stays roughly synchronized (each cycle taking about the same time), they can repeat this dance indefinitely — both consuming CPU, both "active," neither completing.

worker1 acquires lock1 → tries lock2 → fails → releases lock1
worker2 acquires lock2 → tries lock1 → fails → releases lock2
worker1 acquires lock1 → tries lock2 → fails → releases lock1
worker2 acquires lock2 → tries lock1 → fails → releases lock2
... repeats forever ...

Deadlock vs. Livelock — Side-by-Side Comparison

Dimension Deadlock Livelock
Thread state BLOCKED — frozen, not executing RUNNABLE/RUNNING — actively executing
CPU usage Zero (threads are parked) High (threads keep retrying/looping)
Symptom Application appears to hang silently Application appears to run but produces no output/progress
Cause Threads wait indefinitely for a lock held by another Threads keep releasing and retrying, never converging
Diagnostic signal jstack shows threads in BLOCKED, JVM explicitly reports "Java-level deadlock" jstack shows threads cycling through RUNNABLE, repeated log lines with no terminal state reached
Detection difficulty Easier — JVM has automatic deadlock detection Harder — looks like normal (if slow) operation; requires noticing zero forward progress over time

The Universal Fix for Both

Both deadlock and livelock share the same two structural fixes, plus one livelock-specific addition:

  1. Use ReentrantLock with tryLock() instead of indefinite blocking. This converts permanent deadlock (threads frozen forever) into a recoverable failure (threads can back off and retry) — but tryLock() alone, used naively, is exactly what causes livelock if both threads retry in lockstep.
  2. Avoid cyclic dependencies — enforce a consistent lock acquisition order across the entire codebase. If every thread that needs lock1 and lock2 always acquires lock1 first, the cyclic wait condition required for deadlock cannot occur. This is the single most reliable prevention technique and should be a hard rule in any codebase using multiple locks.
  3. For livelock specifically: introduce randomness into retry timing. Instead of both threads retrying after a fixed 50ms, use a random backoff (e.g., ThreadLocalRandom.current().nextInt(10, 100) milliseconds). This breaks the synchronized rhythm that causes both threads to repeatedly collide — eventually one thread's retry timing will desynchronize from the other's, allowing one to successfully acquire both locks while the other is still waiting.
// Livelock fix: random backoff breaks the synchronized retry rhythm
import java.util.concurrent.ThreadLocalRandom;

private void worker1() {
    while (true) {
        try {
            if (lock1.tryLock(50, TimeUnit.MILLISECONDS)) {
                try {
                    if (lock2.tryLock(50, TimeUnit.MILLISECONDS)) {
                        try {
                            // got both locks — do the work, then exit loop
                            return;
                        } finally {
                            lock2.unlock();
                        }
                    }
                } finally {
                    lock1.unlock();
                }
            }
            // Random backoff before retrying — breaks the lockstep pattern
            Thread.sleep(ThreadLocalRandom.current().nextInt(10, 100));
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return;
        }
    }
}

3. Senior-Level Practical Implementation

Real-World Use Case 1: Database Transfer Between Two Accounts (Deadlock Risk)

This is the textbook real-world deadlock scenario — transferring funds requires locking two accounts, and concurrent transfers in opposite directions create the exact cyclic dependency pattern:

public class Account {
    private double balance;
    private final Lock lock = new ReentrantLock(true);
    private final int id;

    public Account(int id, double balance) {
        this.id = id;
        this.balance = balance;
    }

    // DEADLOCK-PRONE: locks acquired in caller-determined order
    public static void transferUnsafe(Account from, Account to, double amount) {
        from.lock.lock();
        try {
            to.lock.lock();           // ← cyclic dependency risk if another thread
            try {                     //   calls transferUnsafe(to, from, ...) concurrently
                if (from.balance >= amount) {
                    from.balance -= amount;
                    to.balance += amount;
                }
            } finally {
                to.lock.unlock();
            }
        } finally {
            from.lock.unlock();
        }
    }

    // DEADLOCK-SAFE: always lock in a globally consistent order (by account ID)
    public static void transferSafe(Account a, Account b, double amount, boolean aToB) {
        Account first  = (a.id < b.id) ? a : b;   // always acquire lower ID first
        Account second = (a.id < b.id) ? b : a;

        first.lock.lock();
        try {
            second.lock.lock();
            try {
                Account from = aToB ? a : b;
                Account to   = aToB ? b : a;
                if (from.balance >= amount) {
                    from.balance -= amount;
                    to.balance += amount;
                }
            } finally {
                second.lock.unlock();
            }
        } finally {
            first.lock.unlock();
        }
    }
}

With transferUnsafe, Thread 1 calling transfer(accountA, accountB, 100) and Thread 2 simultaneously calling transfer(accountB, accountA, 50) creates the classic cyclic lock dependency — guaranteed eventual deadlock under load. transferSafe eliminates this entirely by always locking the lower-ID account first, regardless of transfer direction.

Real-World Use Case 2: Detecting Deadlock in Production with ThreadMXBean

import java.lang.management.ManagementFactory;
import java.lang.management.ThreadMXBean;
import java.lang.management.ThreadInfo;

public class DeadlockMonitor {

    /**
     * Periodically checks for deadlocked threads and logs full diagnostic info.
     * Run this on a scheduled background thread in production services.
     */
    public static void checkForDeadlocks() {
        ThreadMXBean threadBean = ManagementFactory.getThreadMXBean();
        long[] deadlockedIds = threadBean.findDeadlockedThreads();

        if (deadlockedIds != null) {
            System.err.println("=== DEADLOCK DETECTED ===");
            ThreadInfo[] threadInfos = threadBean.getThreadInfo(deadlockedIds, true, true);
            for (ThreadInfo info : threadInfos) {
                System.err.println("Thread: " + info.getThreadName());
                System.err.println("  State: " + info.getThreadState());
                System.err.println("  Locked on: " + info.getLockName());
                System.err.println("  Lock owner: " + info.getLockOwnerName());
                for (StackTraceElement element : info.getStackTrace()) {
                    System.err.println("    at " + element);
                }
            }
            // In production: trigger an alert, dump a heap snapshot, or restart the affected service
        }
    }
}

This is the programmatic equivalent of what jstack <pid> shows automatically — but callable from within the application itself for automated health checks.

4. Deep Dive: Tradeoffs, Architecture & Performance

The Four Necessary Conditions for Deadlock (Coffman Conditions)

A deadlock can only occur if all four of these hold simultaneously — understanding them tells you exactly which one to break:

Condition Description How to break it
Mutual exclusion At least one resource must be held in a non-shareable mode Use read-write locks where possible — allow concurrent readers
Hold and wait A thread holds at least one resource while waiting for another Acquire all needed locks atomically/upfront, or release held locks before waiting
No preemption Resources cannot be forcibly taken from a thread tryLock() with timeout effectively allows "preemption" via voluntary release on failure
Circular wait A cycle of threads exists where each waits for a resource held by the next Consistent lock ordering — the primary practical fix

In production Java code, breaking circular wait via consistent lock ordering is by far the most practical and most commonly applied fix, because it requires no runtime overhead and is purely a coding discipline.

Performance Implications

Deadlock has zero ongoing CPU cost — threads are parked in BLOCKED, consuming no cycles, but the application is functionally dead for those code paths. The cost is entirely in unavailability, not compute.

Livelock has real, ongoing CPU cost — threads are actively spinning through tryLock()/release/retry cycles. In a production system, a livelock can manifest as elevated CPU usage with degraded or zero throughput — a particularly nasty failure mode because monitoring dashboards showing "high CPU, system is working hard" can mask the fact that zero useful work is completing.

Common Pitfalls (Senior Level)

1. Assuming tryLock() Alone Prevents All Lock-Related Hangs

tryLock() converts deadlock risk into livelock risk if retry logic isn't carefully designed. Naively retrying on a fixed interval after a failed tryLock() is exactly the pattern that produces livelock when two threads retry in a synchronized rhythm.

2. Inconsistent Lock Ordering Across Different Code Paths

// Method A (in OrderService): 
synchronized(inventoryLock) { synchronized(orderLock) { ... } }

// Method B (in InventoryService), written by a different engineer:
synchronized(orderLock) { synchronized(inventoryLock) { ... } }
// ↑ Reverse order! This single inconsistency anywhere in a large codebase
//   reintroduces deadlock risk that's extremely hard to find via code review.

This is why large codebases benefit from a documented, enforced lock-ordering convention (e.g., "always acquire locks in alphabetical order of resource name") and, ideally, static analysis tooling that flags inconsistent acquisition order.

3. Forgetting That Deadlock Can Involve More Than Two Threads/Locks

Deadlock cycles can span any number of threads: Thread A waits for Thread B's lock, Thread B waits for Thread C's lock, Thread C waits for Thread A's lock — a three-way cycle. The Coffman conditions and the lock-ordering fix apply identically regardless of cycle length, but detection becomes harder to reason about manually — this is exactly why jstack's automated cycle detection is essential rather than optional in production.

5. Interview Questions & Answers

Basic

Q1: What is a deadlock and what are the two states it produces in the involved threads?

A deadlock occurs when two or more threads each hold a resource (typically a lock) that another thread in the group needs, and each is waiting for a resource held by another — forming a cycle where no thread can ever proceed. All threads in the cycle enter BLOCKED state permanently, consuming zero CPU but never completing their work. The application does not crash or throw an exception — it silently hangs for the affected code paths. The classic cause is a cyclic lock-acquisition order: Thread 1 acquires lock1 then waits for lock2; Thread 2 acquires lock2 then waits for lock1.

Q2: What is a livelock and how is it different from a deadlock in terms of thread state?

A livelock occurs when two or more threads continuously change their state in response to each other's actions — typically retrying after a failed lock attempt — but never converge to actual progress. The key difference from deadlock: livelocked threads remain in RUNNABLE/RUNNING state and are actively consuming CPU, executing real instructions (checking conditions, acquiring and releasing locks, retrying), whereas deadlocked threads are frozen in BLOCKED state with zero CPU usage. This makes livelock harder to detect — the application looks like it's "working" (high CPU, active threads) while producing zero useful output.

Q3: What is the single most reliable structural fix for preventing deadlock?

Enforcing a consistent, global lock acquisition order across the entire codebase. If every code path that needs both lock1 and lock2 always acquires lock1 first, the cyclic wait condition required for deadlock cannot occur — one thread will always be able to proceed to the second lock uncontested while others wait for the first. This eliminates the "circular wait" Coffman condition, which is the one most practically addressable in application code (the other three — mutual exclusion, hold-and-wait, no preemption — are typically inherent to the problem domain).

Intermediate

Q4: A production service occasionally hangs completely under load, with no exceptions logged. CPU usage drops to near zero for the affected threads. What is your diagnostic process?

This signature — silent hang, no exceptions, near-zero CPU for affected threads — is the classic deadlock fingerprint (not livelock, which would show elevated CPU). Diagnostic process: (1) Take a thread dump immediately: kill -3 <pid> or jstack <pid>. (2) The JVM automatically detects cycles and prints "Found one Java-level deadlock:" with the full thread-to-lock-to-holder chain — look for this section first. (3) If not flagged automatically (rare edge cases with certain lock types), manually trace: find threads in BLOCKED state, note which lock each is waiting for (waiting to lock <0xADDR>), then find which other thread holds that lock (locked <0xADDR> in another thread's trace), and follow the chain until it cycles back. (4) Once the cycle is identified, examine the code paths involved and check whether lock acquisition order is consistent across all of them — the fix is almost always reordering one path to match the others. (5) For ongoing prevention: integrate ThreadMXBean.findDeadlockedThreads() into a scheduled health check that alerts before a customer notices the hang.

Q5: Explain why naively converting synchronized to ReentrantLock.tryLock() with a fixed retry delay can introduce a NEW bug that wasn't present before.

synchronized blocks indefinitely — it cannot deadlock-avoid, but it also cannot livelock, because threads simply wait without taking any action. Converting to tryLock() with a fixed retry delay (e.g., "try for 50ms, on failure release everything and retry after exactly 50ms") introduces the possibility of livelock: if two threads enter this retry pattern at nearly the same time and their retry intervals stay synchronized, they can perpetually collide — each acquiring the lock the other needs just after the other releases it, in a repeating rhythm. The original synchronized code never had this failure mode because it had no retry logic at all — it just blocked. The fix when introducing tryLock()-based retry logic is to add random jitter to the retry delay (e.g., ThreadLocalRandom.current().nextInt(10, 100) ms instead of a fixed value) — this desynchronizes the threads' retry rhythms over a small number of iterations, allowing one to eventually win the race for both locks while the other waits.

Advanced

Q6: Walk through the Coffman conditions for deadlock and explain, for each one, whether it's typically practical to eliminate in a real Java application, and why lock ordering is usually the chosen fix.

The four Coffman conditions — all must hold for deadlock to occur. (1) Mutual exclusion: at least one resource is held non-shareably. This is usually inherent to the problem (you can't have two threads both writing to the same bank balance simultaneously) — eliminating it isn't generally practical, though read-write locks can reduce its scope for read-heavy workloads. (2) Hold and wait: a thread holds one resource while waiting for another. Theoretically eliminable by requiring threads to acquire all needed locks atomically (e.g., via a single coarser lock, or acquiring all locks before doing any work) — but this often kills concurrency by serializing unrelated operations, defeating the purpose of fine-grained locking. (3) No preemption: locks can't be forcibly taken from a thread. tryLock() with timeout offers a voluntary form of preemption — a thread can choose to give up and release what it holds — but true OS-level preemption of locks isn't available in the JVM and would be dangerous (leaving protected state in an inconsistent state). (4) Circular wait: a cycle exists in the wait-for graph. This is the only condition that can be eliminated with zero runtime cost and zero loss of concurrency — by enforcing a global, consistent lock acquisition order. This is why, in practice, virtually all deadlock-prevention strategies in real Java codebases focus on consistent lock ordering rather than attacking the other three conditions, which either aren't practical to eliminate or come with unacceptable performance costs.

Q7: A livelock is occurring between two services calling into a shared component with retry logic. Both services were independently developed with seemingly reasonable retry behavior. What systemic root cause typically produces this, and what's the architectural fix beyond random jitter?

The systemic root cause is usually that both services' retry logic was designed in isolation, each assuming it's the only actor under contention — neither anticipated a symmetric peer with identical retry behavior. When deployed together, their retry patterns can synchronize (a phenomenon related to the "thundering herd" problem) — both backing off and retrying at correlated intervals, repeatedly colliding. Random jitter is the tactical fix (breaks synchronization probabilistically) but doesn't address the systemic issue: uncoordinated retry policies across service boundaries. The architectural fix is to centralize contention resolution rather than relying on independent retry: (1) Use a fairness-guaranteeing lock (new ReentrantLock(true)) at the shared component so that whichever thread requested first is served first — removing the "race" entirely. (2) Introduce a queue-based access pattern instead of lock-and-retry — both services submit requests to a single-threaded executor or a message queue that processes them strictly in order, eliminating contention by design rather than managing it after the fact. (3) If genuine concurrent access is required, redesign the shared component to avoid needing both locks simultaneously — often achievable by restructuring the critical section to operate on a single combined resource rather than two separately-locked ones.

6. Mastery Checklist

After internalizing these notes, you should be able to:

  • Define deadlock precisely, identify the cyclic lock-acquisition pattern that causes it, and explain why threads in deadlock show zero CPU usage and never throw exceptions
  • Define livelock precisely, distinguish it from deadlock by thread state (RUNNABLE vs BLOCKED) and CPU usage (high vs zero), and explain why it's the harder failure mode to detect
  • Implement the consistent lock-ordering fix for deadlock and explain why it's the most practical of the four Coffman-condition fixes
  • Implement random-jitter backoff as the livelock fix and explain precisely why a fixed retry interval reproduces the synchronized-collision pattern
  • Use ThreadMXBean.findDeadlockedThreads() (or jstack) to diagnose a production deadlock, trace the lock-holder cycle from thread dump output, and identify the inconsistent acquisition order causing it
  • Recognize the deadlock "fingerprint" (silent hang, zero exceptions, near-zero CPU on affected threads) vs. the livelock "fingerprint" (active threads, elevated CPU, zero forward progress) when triaging a production incident
  • Explain why naively converting synchronized to tryLock()-based retry logic can introduce livelock risk that didn't exist in the original blocking code
  • Articulate all four Coffman conditions for deadlock and explain why circular wait (addressed via lock ordering) is the practically chosen target over the other three
