Java Multithreading: Thread Coordination

Principal Architect Notes — Production-Grade Mastery Series

Syllabus Coverage: Lectures 53–56 (Collections Framework Overview · Synchronized Wrappers · CountDownLatch · CyclicBarrier)

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:
Thread coordination is the discipline of controlling when threads proceed relative to each other — either by making shared data structures safe to access concurrently (synchronized wrappers), making one thread wait until others complete a task (CountDownLatch), or making multiple threads all wait for each other before any can proceed past a shared checkpoint (CyclicBarrier).

The Architecture Analogy

Synchronized wrappers are like hiring a single security guard to stand in front of a file cabinet — only one person can access any drawer at any time, even if they'd be opening completely different drawers that have nothing to do with each other. Safe, but slow.

CountDownLatch is a race starting pistol that won't fire until all runners are confirmed at the starting line. The starter (main thread) waits. Each runner (worker thread) signals "I'm ready" (countDown()). Only when the last runner signals does the starter fire the gun (await() returns) and the race begins. One thread waits for many — and the latch is single-use, like a pistol that fires once.

CyclicBarrier is a relay race checkpoint. All runners must reach the checkpoint before any of them can continue to the next leg. Thread 1 arrives at mile 5, waits. Thread 2 arrives, waits. Thread 3 arrives — now all three resume together. Crucially, the checkpoint can be reused for mile 10 and mile 15 — the latch cannot.

How the JVM Sees It

Synchronized wrappers (Collections.synchronizedList etc.) wrap the target collection in a decorator class where every method is a synchronized method on the wrapper object's intrinsic lock — one lock for the entire collection, all operations serialized.

CountDownLatch internally holds an int count managed by AQS (AbstractQueuedSynchronizer). countDown() is a CAS decrement on state. await() parks the calling thread in the AQS wait queue until state reaches zero — at which point the JVM unparks all waiting threads simultaneously. The count can never be reset (no reset() method) — it is a one-shot mechanism by design.

CyclicBarrier uses a ReentrantLock + Condition internally. Each thread calling await() acquires the lock, increments an arrival count, and calls condition.await() (releasing the lock). When the final thread arrives (arrival count == parties), it runs the optional barrier action Runnable, then calls condition.signalAll() to release all waiting threads — and resets the internal state automatically for reuse.

2. Course Content: Key Concepts & Definitions

The Collections Framework — Thread-Safety Status

Data Structure Thread-Safe? Notes
ArrayList, LinkedList ❌ No Most common — unsynchronized by default
HashMap, LinkedHashMap, TreeMap ❌ No Standard maps — not safe for concurrent use
HashSet, LinkedHashSet, TreeSet ❌ No Standard sets — not safe
Vector ✅ Yes Legacy synchronized list — obsolete, use CopyOnWriteArrayList
Stack ✅ Yes Extends Vector — also legacy
Hashtable ✅ Yes Legacy synchronized map — replaced entirely by ConcurrentHashMap
ConcurrentHashMap ✅ Yes Modern, efficient — segment/bin-level locking (covered in Group 3)
Blocking queues ✅ Yes ArrayBlockingQueue, LinkedBlockingQueue, etc. (covered in Group 2)

The overarching rule: everything in java.util (except Vector/Hashtable) is not thread-safe. Everything in java.util.concurrent is. The gap between them is the entire reason Collections.synchronizedXxx() and the concurrent collections exist.

Collections.synchronizedList() — The Blunt Instrument

import java.util.*;

List<Integer> rawList = new ArrayList<>();
List<Integer> safeList = Collections.synchronizedList(rawList);
// safeList is now thread-safe — every method is synchronized on the wrapper's monitor

// Without synchronization: ArrayIndexOutOfBoundsException under concurrent modification
// With Collections.synchronizedList: correct, but serialized — only one thread at a time

How it works: Collections.synchronizedList() returns a SynchronizedList decorator that wraps every method call in synchronized(mutex), where mutex is the wrapper object itself. This is identical to using the synchronized keyword on every method — which means it uses the same single intrinsic lock for add(), remove(), get(), size(), and every other operation.

The problem: one lock, all operations serialized. If Thread 1 is reading index 0 and Thread 2 is adding to index 999, they block each other — even though these are completely independent operations. This is the same single-lock bottleneck from the synchronized chapter, applied to a whole collection.

The other problem — iteration is NOT automatically synchronized:

List<String> safeList = Collections.synchronizedList(new ArrayList<>());

// BUG: even though safeList is synchronized, iteration is NOT
// Another thread can modify safeList between iterations of this loop
for (String item : safeList) {      // ← ConcurrentModificationException risk
    process(item);
}

// CORRECT: manually synchronize on the list for the entire iteration block
synchronized (safeList) {
    for (String item : safeList) {  // ← safe — lock held for entire traversal
        process(item);
    }
}

This is a critical, non-obvious gap: the individual method calls are synchronized, but compound operations like iteration (which internally calls iterator() then repeatedly calls next()) are not atomic as a unit. You must manually synchronize on the collection object for any multi-step operation that requires consistency across steps.

Available wrappers:

Collections.synchronizedList(list);    // for List implementations
Collections.synchronizedMap(map);      // for Map implementations
Collections.synchronizedSet(set);      // for Set implementations
Collections.synchronizedSortedMap(sortedMap);

CountDownLatch — One Thread Waits for Many

import java.util.concurrent.CountDownLatch;

A CountDownLatch is initialized with a count, and a thread calling await() blocks until that count reaches zero via repeated countDown() calls from other threads.

CountDownLatch(5)  →  count = 5
Worker 1 calls countDown()  →  count = 4
Worker 2 calls countDown()  →  count = 3
Worker 3 calls countDown()  →  count = 2
Worker 4 calls countDown()  →  count = 1
Worker 5 calls countDown()  →  count = 0  →  await() returns — main thread proceeds

Why CountDownLatch over thread.join():

Dimension thread.join() CountDownLatch
Thread tracking Must hold a reference to every Thread object No thread references needed — threads find the latch, not vice versa
Works with ExecutorService? ❌ No — ExecutorService hides thread references ✅ Yes — workers just call countDown() on a shared latch
Reusable? N/A (create new threads) ❌ No — once count reaches 0, latch is permanently open
Granularity "wait for this specific thread to finish" "wait until N operations have called countDown()" 
import java.util.concurrent.*;

class Worker implements Runnable {
    private final int id;
    private final CountDownLatch latch;

    Worker(int id, CountDownLatch latch) {
        this.id = id;
        this.latch = latch;
    }

    @Override
    public void run() {
        System.out.println("Thread " + id + " starts work");
        try {
            Thread.sleep(2000);   // simulate work
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            latch.countDown();    // ← MUST be in finally — even if work throws, latch must decrement
            System.out.println("Thread " + id + " done, latch count = " + latch.getCount());
        }
    }
}

public class CountDownLatchExample {
    public static void main(String[] args) throws InterruptedException {
        int workerCount = 5;
        CountDownLatch latch = new CountDownLatch(workerCount);

        ExecutorService service = Executors.newSingleThreadExecutor();
        for (int i = 0; i < workerCount; i++) {
            service.execute(new Worker(i, latch));
        }

        System.out.println("Main thread waiting for all workers...");
        latch.await();   // blocks until all 5 workers call countDown()
        System.out.println("All tasks finished — main thread continues");

        service.shutdown();
    }
}

The finally rule for countDown(): latch.countDown() must always be in a finally block. If a worker throws an uncaught exception before calling countDown(), the latch count never reaches zero, await() blocks forever, and the application hangs. finally guarantees the countdown happens regardless of success or failure.

What happens if the initial count is higher than the number of workers? await() blocks forever — no more workers will call countDown(), so the count never reaches zero. The initial count must always be ≤ the number of operations that will call countDown().

CyclicBarrier — Multiple Threads Wait for Each Other

import java.util.concurrent.CyclicBarrier;

A CyclicBarrier is initialized with a parties count (number of threads expected) and an optional barrier action Runnable that executes automatically when all threads arrive. Every thread calls await() — and all block until the last thread arrives, after which all are released simultaneously and the barrier resets for reuse.

Thread 1: await() → blocked
Thread 2: await() → blocked
Thread 3: await() → blocked
Thread 4: await() → blocked
Thread 5: await() → THIS is the 5th (= parties count)
           → barrier action runs ("All tasks finished")
           → ALL 5 threads released simultaneously
           → barrier RESETS automatically for next cycle
import java.util.concurrent.*;

class BarrierWorker implements Runnable {
    private final int id;
    private final CyclicBarrier barrier;
    private final Random random = new Random();

    BarrierWorker(int id, CyclicBarrier barrier) {
        this.id = id;
        this.barrier = barrier;
    }

    private void doWork() throws InterruptedException {
        System.out.println("Thread " + id + " starts work");
        Thread.sleep(random.nextInt(3000));   // random work duration 0–3s
    }

    @Override
    public void run() {
        try {
            doWork();
            System.out.println("Thread " + id + " reached the barrier — waiting for others");
            barrier.await();   // block until ALL parties have called await()
            System.out.println("Thread " + id + " continues after barrier");
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } catch (BrokenBarrierException e) {
            System.err.println("Barrier was broken — thread " + id + " aborting");
        }
    }
}

public class CyclicBarrierExample {
    public static void main(String[] args) {
        int threadCount = 5;

        // Optional barrier action: runs automatically when all parties arrive
        Runnable barrierAction = () -> System.out.println("=== All tasks complete — barrier released ===");

        CyclicBarrier barrier = new CyclicBarrier(threadCount, barrierAction);
        ExecutorService service = Executors.newFixedThreadPool(threadCount);

        for (int i = 1; i <= threadCount; i++) {
            service.execute(new BarrierWorker(i, barrier));
        }

        service.shutdown();
    }
}

BrokenBarrierException — a CyclicBarrier becomes "broken" if any waiting thread is interrupted while blocked in await(), or if the barrier action throws an exception. All other threads waiting on the same barrier receive BrokenBarrierException — a signal that the coordinated operation cannot proceed cleanly. Always handle both InterruptedException and BrokenBarrierException in production code.

Reusability in practice: once all parties call await() and are released, the barrier automatically resets its internal state. If you submit 5 more workers to the same executor using the same barrier object, they will form a new barrier cycle — the barrier doesn't need to be reinstantiated.

CountDownLatch vs. CyclicBarrier — Complete Comparison

Dimension CountDownLatch CyclicBarrier
Who waits Typically one (or more) "observer" threads call await(); worker threads call countDown() All participating threads call await() — everyone waits for everyone
Direction Asymmetric: watchers vs. workers Symmetric: all parties are peers
Reusability ❌ One-shot — once count hits 0, it stays open ✅ Reusable — resets after each cycle automatically
Barrier action ❌ Not supported ✅ Optional Runnable executes when barrier is triggered
Exception InterruptedException only InterruptedException + BrokenBarrierException
Typical use "Main thread waits for all workers to complete initialization" "All threads must complete phase 1 before any can start phase 2"
Relationship to join() Direct analog — but works with executor services No direct analog — unique to cyclic multi-party coordination

3. Senior-Level Practical Implementation

Real-World Use Case 1: Service Startup Orchestration with CountDownLatch

import java.util.concurrent.*;

public class ServiceStartup {

    public static void main(String[] args) throws InterruptedException {
        int serviceCount = 3;
        CountDownLatch readyLatch = new CountDownLatch(serviceCount);

        ExecutorService executor = Executors.newFixedThreadPool(serviceCount);

        executor.submit(() -> startService("DatabaseService", 2000, readyLatch));
        executor.submit(() -> startService("CacheService", 1000, readyLatch));
        executor.submit(() -> startService("MessageBroker", 1500, readyLatch));

        System.out.println("Waiting for all services to become ready...");
        boolean allReady = readyLatch.await(10, TimeUnit.SECONDS);   // timeout — don't wait forever

        if (allReady) {
            System.out.println("All services ready — starting HTTP server");
            startHttpServer();
        } else {
            System.err.println("Timeout: not all services ready in 10s — aborting startup");
            executor.shutdownNow();
        }

        executor.shutdown();
    }

    static void startService(String name, long startupMs, CountDownLatch latch) {
        try {
            System.out.println(name + " initializing...");
            Thread.sleep(startupMs);
            System.out.println(name + " READY");
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            System.err.println(name + " startup interrupted");
        } finally {
            latch.countDown();   // always signal, even on failure — prevents deadlock
        }
    }

    static void startHttpServer() { System.out.println("HTTP server accepting requests"); }
}

This is the textbook production use case for CountDownLatch — a service that must not begin serving requests until its dependencies (database, cache, message broker) have all completed initialization. The 10-second timeout on await() prevents the application from hanging indefinitely if one dependency fails to start.

Real-World Use Case 2: Parallel Data Processing with CyclicBarrier

import java.util.concurrent.*;

public class ParallelDataPipeline {

    static double[] partialResults = new double[4];

    public static void main(String[] args) {
        int threads = 4;

        // Phase-end action: aggregate partial results, then move to next phase
        CyclicBarrier barrier = new CyclicBarrier(threads, () -> {
            double total = 0;
            for (double r : partialResults) total += r;
            System.out.println("Phase complete — aggregated result: " + total);
        });

        ExecutorService executor = Executors.newFixedThreadPool(threads);

        for (int i = 0; i < threads; i++) {
            final int workerId = i;
            executor.submit(() -> {
                try {
                    // Phase 1: load data
                    partialResults[workerId] = loadDataChunk(workerId);
                    barrier.await();   // wait for all workers to finish loading

                    // Phase 2: process data (can start because all data is loaded)
                    partialResults[workerId] = processChunk(workerId, partialResults[workerId]);
                    barrier.await();   // wait for all workers to finish processing

                    // Phase 3: write results
                    writeResults(workerId, partialResults[workerId]);
                    barrier.await();   // final sync before shutdown

                } catch (InterruptedException | BrokenBarrierException e) {
                    Thread.currentThread().interrupt();
                }
            });
        }

        executor.shutdown();
    }

    static double loadDataChunk(int id) throws InterruptedException {
        Thread.sleep(500 + (long)(Math.random() * 500));
        return Math.random() * 100;
    }
    static double processChunk(int id, double data) throws InterruptedException {
        Thread.sleep(300);
        return data * 1.1;
    }
    static void writeResults(int id, double result) {
        System.out.printf("Worker %d wrote result: %.2f%n", id, result);
    }
}

This demonstrates the canonical CyclicBarrier pattern — a multi-phase parallel algorithm where each phase must complete across all threads before any thread can advance to the next phase. The same barrier object serves all three phase boundaries.

4. Deep Dive: Tradeoffs, Architecture & Performance

Why Synchronized Wrappers Are a Dead End for Production Use

The Collections.synchronizedXxx() wrappers are not wrong — they are correct. They are simply underperforming for any workload beyond trivial cases. The single intrinsic lock serializes all operations: read vs. write, add vs. get, two completely independent operations on non-overlapping indices. Under any significant concurrency, threads queue up behind the single lock, and the effective throughput is bounded by single-threaded performance — with the added overhead of lock acquisition and context switching. For production concurrent collection use, always prefer java.util.concurrent implementations: CopyOnWriteArrayList for read-heavy lists, ConcurrentHashMap for maps, ArrayBlockingQueue/LinkedBlockingQueue for queues.

CountDownLatch vs. CyclicBarrier — When Each Shines

CountDownLatch is the right tool when the coordination relationship is asymmetric — one thread (or a few) need to know when a set of other threads has finished some work. The waiting threads don't need to participate in the work — they only need to observe its completion. Classic uses: service startup gates, parallel initialization waiting, test synchronization.

CyclicBarrier is the right tool when all threads are peers and coordination is symmetric — every participating thread does a chunk of work, all must reach the checkpoint before any proceeds, and this pattern repeats across multiple phases. Classic uses: parallel algorithms with synchronization points (parallel merge sort, simulation time steps, matrix multiplication phases), where a thread that finishes its chunk early shouldn't race ahead into the next phase while others are still on the current one.

Common Pitfalls

1. Forgetting countDown() in a finally block

// WRONG: if doWork() throws, countDown() never executes → await() blocks forever
public void run() {
    doWork();
    latch.countDown();  // ← not in finally → hangs if doWork() throws
}

// CORRECT
public void run() {
    try { doWork(); }
    catch (Exception e) { /* log */ }
    finally { latch.countDown(); }   // ← always decrements, no matter what
}

2. Setting CountDownLatch count higher than number of countDown() callers

CountDownLatch latch = new CountDownLatch(10);  // expects 10 countdowns
for (int i = 0; i < 5; i++) {
    executor.execute(() -> { doWork(); latch.countDown(); });
}
latch.await();   // HANGS FOREVER — count can only reach 5, never 0

Always ensure new CountDownLatch(n) where n exactly equals the number of threads/tasks that will call countDown().

3. Not handling BrokenBarrierException in CyclicBarrier workers

// If any thread is interrupted mid-await(), ALL other waiting threads get BrokenBarrierException
// Without a catch for it, those threads crash silently
barrier.await();  // ← needs: } catch (BrokenBarrierException e) { ... }

4. Calling CyclicBarrier.reset() while threads are waiting

barrier.reset();   // while threads are blocked in await() → they all get BrokenBarrierException
                   // and abort immediately

reset() is only appropriate when no threads are currently waiting on the barrier — i.e., outside of normal execution cycles, typically in error-recovery code paths.

5. Iterating a synchronized wrapper without external synchronization

List<String> list = Collections.synchronizedList(new ArrayList<>());
// Thread A:
for (String s : list) { process(s); }   // ConcurrentModificationException if Thread B modifies during iteration

// Fix: synchronize on the collection for the entire iteration
synchronized(list) {
    for (String s : list) { process(s); }
}

5. Interview Questions & Answers

Basic

Q1: What is Collections.synchronizedList() and what is its main limitation?

Collections.synchronizedList() wraps an existing List in a decorator that synchronizes every method call on a single intrinsic lock — the wrapper object's monitor — making each individual operation (add, remove, get) thread-safe. Its main limitations are two-fold: (1) performance — a single lock serializes all operations including independent reads, making it effectively single-threaded under contention; (2) compound operation safety — individual method calls are synchronized, but multi-step operations like iteration are not atomic as a unit, requiring the caller to manually synchronize around the entire iteration block (synchronized(list) { for (item : list) { ... } }) to prevent ConcurrentModificationException. For production concurrent use, prefer CopyOnWriteArrayList (read-heavy) or ConcurrentLinkedQueue (write-heavy) over synchronized wrappers.

Q2: What is a CountDownLatch, how does it differ from thread.join(), and when would you use one over the other?

A CountDownLatch is initialized with a count and provides two operations: countDown() (decrements the count) and await() (blocks until the count reaches zero). It differs from thread.join() in two important ways: (1) join() requires a direct reference to the Thread object to join on; CountDownLatch requires no thread references — workers just call countDown() on a shared instance, making it compatible with ExecutorService where thread references are not exposed. (2) join() is coupled to a specific thread's lifecycle; CountDownLatch is coupled to a counter — multiple threads can each call countDown() multiple times, and the count can represent any logical event, not just thread completion. Use join() when you create threads manually and naturally hold references. Use CountDownLatch when using executor services, when coordination involves many threads, or when the "signal" is a logical event rather than thread completion.

Q3: What is the difference between CountDownLatch and CyclicBarrier?

Three differences: (1) Who waits: CountDownLatch is asymmetric — some threads wait (await()) while others decrement (countDown()); CyclicBarrier is symmetric — all threads call await() and everyone waits for everyone. (2) Reusability: CountDownLatch is single-use — once the count reaches zero, it stays open forever and cannot be reset; CyclicBarrier automatically resets after each cycle and can be reused indefinitely. (3) Barrier action: CyclicBarrier supports an optional Runnable that executes automatically when all parties arrive; CountDownLatch has no equivalent. Use CountDownLatch when one thread is waiting for many others to finish a task. Use CyclicBarrier when multiple threads must synchronize at a shared checkpoint before all proceeding together, especially when this pattern repeats.

Intermediate

Q4: You're using CountDownLatch with 5 worker tasks submitted to an ExecutorService. One of the workers throws an uncaught RuntimeException during its work. What happens to the await() call in the main thread, and how do you prevent it?

If countDown() is not called before the exception propagates, the latch count never reaches zero and await() in the main thread blocks indefinitely — a permanent hang. The ExecutorService absorbs the uncaught exception silently (it's recorded in the Future if submit() was used, but execute() merely calls the thread's uncaught exception handler and continues). The prevention is always placing countDown() in a finally block:

public void run() {
    try { doRiskyWork(); }
    catch (Exception e) { logError(e); }  // handle or log
    finally { latch.countDown(); }        // always decrements
}

Additionally, use latch.await(timeout, TimeUnit) instead of the unbounded await() in the main thread — a timeout provides a safety bound so even if something goes catastrophically wrong (a countDown() is missed despite finally), the waiting thread eventually unblocks with a false return value rather than hanging forever.

Q5: In a parallel data processing pipeline with 4 threads, you use a CyclicBarrier to synchronize phases. Thread 2 completes phase 1 but is interrupted while waiting in barrier.await(). What happens to the other three threads waiting at the barrier, and what exception do they receive?

When Thread 2 is interrupted while blocked in barrier.await(), the CyclicBarrier transitions to a "broken" state. All three other threads currently blocked in await() are immediately woken up and each receives a BrokenBarrierException — not an InterruptedException, which only Thread 2 itself receives. This "broken" state propagates to all subsequent calls to await() on the same barrier instance until barrier.reset() is called. This design is intentional: if one thread in a coordinated group cannot proceed, there's no meaningful way for the other threads to continue with the next phase without the missing thread's contribution. The BrokenBarrierException is the signal to abort the current coordinated operation, perform any necessary cleanup, and potentially restart from a known state. Always catch both InterruptedException and BrokenBarrierException in barrier-using code and include appropriate recovery logic.

Advanced

Q6: Explain why CountDownLatch is non-reusable at the JMM/AQS level, and describe the architectural pattern you'd use when you need a "resettable latch" — for example, a server that must wait for all pending requests to drain before accepting a new batch.

CountDownLatch's non-reusability is not arbitrary — it stems from AQS's state model for shared latches. Once the AQS state field reaches zero, all waiting threads are released and the latch transitions to a terminal "open" state. There is no reset() method because resetting a shared AQS latch while threads are queued would create a race condition: threads that just observed state = 0 and were about to return from await() could be "re-latched" mid-unpark, producing undefined behavior. CyclicBarrier avoids this by managing its own separate count variable inside a ReentrantLock and only resetting after all parties have been released — a strictly ordered teardown that AQS's simpler countdown model cannot guarantee. For a resettable latch scenario (drain-and-refill), the correct pattern is: (1) Use CyclicBarrier if the "new batch" cycle always involves the same set of fixed threads. (2) Use a Phaser (a generalization of both CountDownLatch and CyclicBarrier that supports dynamic registration and deregistration of parties, and is reusable by design) — the idiomatic solution for variable-participant, multi-phase barriers in production systems. (3) At a lower level, use AtomicInteger + a Condition from a ReentrantLock — manually decrement on each request completion, signal() when count reaches zero, and manually reset the counter to the new batch size under the lock before releasing.
