Java Multithreading: ConcurrentHashMap, CopyOnWriteArrayList & Exchanger

Principal Architect Notes — Production-Grade Mastery Series

Syllabus Coverage: Lectures 61–63 (ConcurrentHashMap · Slipping Conditions · CopyOnWriteArrayList · Exchanger)

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:
ConcurrentHashMap achieves thread-safe map operations at near-single-threaded read speed by eliminating locks on reads entirely and using the finest possible lock granularity (per-bucket) for writes; CopyOnWriteArrayList achieves thread-safe list reads at zero lock cost by making writers pay the full cost of copying the backing array; and Exchanger provides a rendezvous point where exactly two threads swap objects in a single atomic operation.

The Architecture Analogy

ConcurrentHashMap is a modern library with one rule for readers: "just walk in and read any shelf, no sign-in needed — volatile guarantees mean the shelf is always current." Writers have a more nuanced rule: "if you're writing to an empty shelf, use the self-checkout atomically (CAS). If you're writing to a shelf that already has something, lock just that one specific shelf — not the whole library, not a wing, just that one shelf." This is maximum read throughput (zero lock cost) with minimal write contention (one lock per bucket).

CopyOnWriteArrayList is a museum exhibit. Visitors (readers) walk through freely — no tickets, no guards, any number of visitors simultaneously. But if the museum wants to add or rearrange an exhibit (write), it temporarily creates an exact duplicate of the entire museum in a back room, makes the change there, then atomically swaps the public gallery to the new copy. Visitors in the original gallery see nothing change mid-walk — they're reading an immutable snapshot. Only after the swap do new visitors see the updated layout. Reads are free; writes are expensive.

Exchanger is a spy dead-drop rendezvous. Thread A arrives at the exchange point and waits. Thread B arrives and waits. The moment both are present, they simultaneously swap their packages and both walk away with the other's item. If only one arrives, it waits. There's no exchange without both parties present — and exactly two parties, never more.

How the JVM Sees It

ConcurrentHashMap reads (Java 8+): field values in map nodes are declared volatile, and the array itself is volatile. A read via get() involves no lock acquisition — the volatile reads establish happens-before with the prior writes that stored the values. The JVM's memory barrier instructions guarantee the reader sees the most recent writer's value without any explicit synchronization object.

ConcurrentHashMap writes: First insert into an empty bin: casTabAt() — a Compare-And-Swap on the array slot (atomic, lock-free). Subsequent inserts into an occupied bin: synchronized(binHead) — locks the first node of the linked list or TreeBin object (bin-level granularity, not segment, not table). Resize: locks the entire table briefly — all threads see MOVED nodes and help with rehashing.

CopyOnWriteArrayList writes: ReentrantLock serializes all write operations. Writer acquires lock, copies array (Arrays.copyOf()), makes modification on copy, then atomically sets the internal array reference via volatile write, releases lock. Readers always dereference the current volatile array reference — they read an immutable snapshot without any lock.

Exchanger: internally uses LockSupport.park()/unpark() and CAS on a Slot object. First thread to arrive CAS-stores its item and parks. Second thread CAS-swaps items and unparks the first thread. Both receive the other's item atomically.

2. Course Content: Key Concepts & Definitions

ConcurrentHashMap — Architecture and Internal Evolution

The evolution matters because Java 8 fundamentally changed how ConcurrentHashMap works:

Before Java 8 (segment-based): the array was divided into a fixed number of segments (16 by default), each with its own ReentrantLock. Writes to different segments could proceed in parallel; reads within a segment required the segment lock.

Java 8 and later (bin-level locking): segments are gone. The structure is a single flat array of bins (buckets). Reads are lock-free via volatile. Writes lock at the individual bin level — the smallest possible unit.

Array:
[bin 0] → null
[bin 1] → Node(key="A", val=1) → Node(key="Q", val=7)   ← linked list (collision)
[bin 2] → null
[bin 3] → TreeBin(root=Node...)                           ← red-black tree (≥8 collisions)
...

Read operations (lock-free):

ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// get() — NO lock, NO synchronized, NO volatile acquire beyond reading node.val
Integer value = map.get("key");   // safe to call from any number of threads simultaneously

Why reads are safe without locks: the node's value field and the next pointer are volatile. The volatile write that stored the value happens-before the volatile read that retrieves it — the JMM's happens-before chain ensures the reader sees the writer's value without any explicit synchronization.

Write operations (bin-level locking):

map.put("newKey", 42);
// Internally:
// 1. Hash "newKey" → compute bin index
// 2. If bin is empty: CAS (compare-and-swap) the null slot atomically — no lock
// 3. If bin has a node: synchronized(binHead) { ... traverse/insert ... }
//    — locks ONLY THIS ONE BIN, not the entire map

This is the key insight: two put() calls on different keys that hash to different bins proceed completely in parallel — they hold different locks, have no contention with each other whatsoever.

The Slipping Condition — Why Even ConcurrentHashMap Has Pitfalls

ConcurrentHashMap guarantees atomicity for individual operations (put, get, remove, containsKey). It does NOT guarantee atomicity for compound operations — sequences of multiple calls that must appear as a single atomic unit.

ConcurrentMap<String, String> map = new ConcurrentHashMap<>();

// SLIPPING CONDITION — looks reasonable, is a race condition:
if (!map.containsKey("key")) {          // Thread A checks: "key" absent — true
                                         // Thread B also checks: "key" absent — true (same time!)
    map.put("key", "value");            // Thread A inserts
                                         // Thread B ALSO inserts, overwriting A's value
}
// Both threads thought they were the first inserter — both executed put() — a lost update

This is called a slipping condition (also known as a "check-then-act" race condition): two threads both check, both see the same state, and both act — producing a result neither intended.

The fix: use atomic compound methods built into ConcurrentHashMap:

// putIfAbsent: atomically checks and inserts ONLY if key is absent
// Returns the existing value if key was already present, null if insertion succeeded
String existing = map.putIfAbsent("key", "value");
if (existing == null) {
    System.out.println("Successfully inserted");
} else {
    System.out.println("Key already existed: " + existing);
}

// computeIfAbsent: atomically compute and insert if absent
// More powerful — can use the key to compute the value
map.computeIfAbsent("user-123", key -> loadUserFromDatabase(key));

// compute: atomically update a value based on current value
map.compute("counter", (key, current) -> (current == null) ? 1 : current + 1);

// merge: atomically merge old and new values with a function
map.merge("count", 1, Integer::sum);   // add 1 to existing value (or put 1 if absent)

The key principle: any time your code does "check the map, then conditionally write to the map" — the gap between check and write is a slipping window where another thread can intervene. Always replace the two-step pattern with the corresponding atomic single-operation method from ConcurrentMap.

Why synchronizing on the map is wrong:

// "Solution" — but a bad one
synchronized (map) {
    if (!map.containsKey("key")) {
        map.put("key", "value");
    }
}

This works but defeats the purpose of ConcurrentHashMap entirely — you've reduced it to a single global lock on the whole map, the same bottleneck as Collections.synchronizedMap(). The atomic ConcurrentMap methods are the correct solution because they operate at bin-level granularity, not table-level.

ConcurrentHashMap — Internal Red-Black Tree Conversion

When a single bin accumulates more than 8 nodes (many hash collisions), the linked list is automatically converted to a red-black tree for O(log n) access instead of O(n):

Bin with 8+ collisions:
Before: Node → Node → Node → Node → Node → Node → Node → Node  (linked list, O(n))
After:          TreeBin (wraps red-black tree root)             (O(log n))

This introduces a subtle locking challenge: red-black tree rotations can change the root node. Locking on the root (as you would with a linked list's head node) would be unsafe — the root can change during rotation, meaning two threads might inadvertently hold different locks. The solution: ConcurrentHashMap wraps the tree in a TreeBin object that never changes. All threads synchronize on the TreeBin instance — a stable anchor — regardless of how the tree structure evolves internally.

CopyOnWriteArrayList — Read-Optimized List

import java.util.concurrent.CopyOnWriteArrayList;

// Implements List — drop-in replacement for ArrayList in concurrent read-heavy scenarios
List<String> list = new CopyOnWriteArrayList<>();
list.add("item1");   // safe from any thread — internally copies the array
list.add("item2");

How writes work (the "copy-on-write" mechanism):

// Conceptual implementation of add():
ReentrantLock lock = new ReentrantLock();
volatile Object[] array;   // the backing array — always immutable once visible

public void add(E element) {
    lock.lock();
    try {
        Object[] current = array;
        Object[] newArray = Arrays.copyOf(current, current.length + 1);  // O(n) copy
        newArray[current.length] = element;
        array = newArray;   // volatile write — atomically switches readers to new array
    } finally {
        lock.unlock();
    }
}

How reads work (lock-free):

public E get(int index) {
    return (E) array[index];   // reads the volatile 'array' reference — NO lock
}

A reader always sees either the old array or the new array — never a half-modified intermediate state. This is because the modification is made to a copy, and only the final volatile write switches the reference. Any reader that started reading the old array will complete its read on the old (immutable) array; any reader that starts after the volatile write sees the new array. There is no moment where the array is partially modified.

Iteration is always safe — reads a snapshot:

// This is completely safe even if another thread adds/removes items mid-iteration
for (String item : list) {
    process(item);
}
// The iterator holds a reference to the array snapshot at the moment iteration started
// New additions after iteration started are invisible to this iterator (not ConcurrentModificationException)

This is the reverse of Collections.synchronizedList — which is individually synchronized per method call but still throws ConcurrentModificationException on concurrent iteration if you don't manually synchronize the whole loop.

When to use vs. when not to:

Scenario CopyOnWriteArrayList?
Read-heavy: 1000 reads per 1 write ✅ Excellent — reads are free
Write-heavy: 1000 writes per 1 read ❌ Terrible — O(n) copy per write, high GC pressure
Iteration must be consistent mid-traversal ✅ Yes — iterator sees immutable snapshot
Large list with frequent writes ❌ Use ConcurrentLinkedQueue or LinkedBlockingQueue
Listener/observer lists (add once, iterate many times) ✅ Perfect use case

Exchanger — Two-Thread Object Swap

import java.util.concurrent.Exchanger;

// Generic — specify the type of object being exchanged
Exchanger<Integer> exchanger = new Exchanger<>();

An Exchanger provides a synchronization point at which two threads can exchange objects. Each thread calls exchange(myItem) — the call blocks until the other thread also calls exchange(). At that moment, both threads atomically receive the other's item.

class FirstThread implements Runnable {
    private final Exchanger<Integer> exchanger;
    private int counter = 0;

    FirstThread(Exchanger<Integer> exchanger) { this.exchanger = exchanger; }

    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                counter++;
                System.out.println("First thread: sending " + counter);
                counter = exchanger.exchange(counter);   // sends counter, receives other thread's counter
                System.out.println("First thread: received " + counter);
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}

class SecondThread implements Runnable {
    private final Exchanger<Integer> exchanger;
    private int counter = 0;

    SecondThread(Exchanger<Integer> exchanger) { this.exchanger = exchanger; }

    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                counter--;
                System.out.println("Second thread: sending " + counter);
                counter = exchanger.exchange(counter);  // sends counter, receives other thread's counter
                System.out.println("Second thread: received " + counter);
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}

public class ExchangerExample {
    public static void main(String[] args) throws InterruptedException {
        Exchanger<Integer> exchanger = new Exchanger<>();
        Thread t1 = new Thread(new FirstThread(exchanger), "first");
        Thread t2 = new Thread(new SecondThread(exchanger), "second");
        t1.start();
        t2.start();
        Thread.sleep(5000);
        t1.interrupt();
        t2.interrupt();
    }
}

Execution trace:

First thread: sending 1        (increments 0 → 1, calls exchange(1), blocks)
Second thread: sending -1      (decrements 0 → -1, calls exchange(-1), blocks)
[both arrive at exchange point]
First thread: received -1      (got second thread's value)
Second thread: received 1      (got first thread's value)

Critical constraints:

  • Exchanger works with exactly two threads — no more, no less. If only one thread calls exchange(), it blocks indefinitely waiting for a second thread that never comes
  • Both threads must call exchange() on the same Exchanger instance
  • The exchange is mutual and simultaneous — neither thread proceeds until both have called exchange()
  • Use exchanger.exchange(item, timeout, unit) (throws TimeoutException) rather than the unbounded exchange(item) to avoid indefinite blocking if the partner thread fails to arrive

Thread-local variable access: Exchanger solves the problem of sharing data between specific thread pairs without shared heap objects. Each thread keeps its data in local variables (on the stack — completely private), and the exchanger is the only crossing point — a clean, explicit, bidirectional data transfer.

3. Senior-Level Practical Implementation

Complete ConcurrentHashMap Example with Atomic Compound Methods

import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class ConcurrentMapExample {

    record UserProfile(String name, int loginCount) {}

    public static void main(String[] args) throws InterruptedException {
        ConcurrentMap<String, UserProfile> profiles = new ConcurrentHashMap<>();
        ConcurrentMap<String, AtomicInteger> loginCounts = new ConcurrentHashMap<>();

        ExecutorService executor = Executors.newFixedThreadPool(10);

        // Simulate 100 concurrent login events across 5 users
        for (int i = 0; i < 100; i++) {
            String userId = "user-" + (i % 5);
            executor.submit(() -> handleLogin(profiles, loginCounts, userId));
        }

        executor.shutdown();
        executor.awaitTermination(10, TimeUnit.SECONDS);

        // Print final login counts
        loginCounts.forEach((userId, count) ->
            System.out.printf("%s: %d logins%n", userId, count.get()));
    }

    static void handleLogin(ConcurrentMap<String, UserProfile> profiles,
                            ConcurrentMap<String, AtomicInteger> loginCounts,
                            String userId) {
        // computeIfAbsent: atomically create profile if it doesn't exist
        profiles.computeIfAbsent(userId, id -> new UserProfile(id, 0));

        // computeIfAbsent to create the counter, then increment — all thread-safe
        loginCounts.computeIfAbsent(userId, id -> new AtomicInteger(0))
                   .incrementAndGet();
    }
}

Real-World Use Case: CopyOnWriteArrayList for Event Listeners

import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;

public class EventBus {
    // Perfect use case: listeners added rarely, events fired frequently
    private final List<EventListener> listeners = new CopyOnWriteArrayList<>();

    @FunctionalInterface
    interface EventListener { void onEvent(String event); }

    // Called occasionally — adding one listener copies the whole list
    public void subscribe(EventListener listener) {
        listeners.add(listener);   // thread-safe, O(n) copy — infrequent, acceptable
    }

    public void unsubscribe(EventListener listener) {
        listeners.remove(listener);
    }

    // Called thousands of times per second — zero lock cost for reading the listener list
    public void publish(String event) {
        for (EventListener listener : listeners) {   // iterates immutable snapshot — never throws ConcurrentModificationException
            listener.onEvent(event);
        }
    }
}

Event bus/observer patterns are the textbook CopyOnWriteArrayList use case: listeners are added at startup or rarely during runtime, but events are dispatched at high frequency. The copy-on-write trade-off (O(n) write cost) is entirely acceptable when writes are rare.

Real-World Use Case: Exchanger for Ping-Pong Data Pipeline

import java.util.concurrent.*;
import java.util.ArrayList;
import java.util.List;

public class PingPongPipeline {

    public static void main(String[] args) throws InterruptedException {
        Exchanger<List<Integer>> exchanger = new Exchanger<>();

        // Producer fills a buffer, then exchanges it with the consumer's empty buffer
        // Consumer processes the full buffer while producer refills the returned empty one
        Thread producer = new Thread(() -> {
            List<Integer> buffer = new ArrayList<>();
            try {
                for (int cycle = 0; cycle < 5; cycle++) {
                    // Fill the buffer
                    for (int i = 0; i < 10; i++) buffer.add(cycle * 10 + i);
                    System.out.println("Producer filled: " + buffer);

                    // Exchange full buffer for the consumer's emptied buffer
                    buffer = exchanger.exchange(buffer);
                    System.out.println("Producer received empty buffer, size: " + buffer.size());
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }, "producer");

        Thread consumer = new Thread(() -> {
            List<Integer> buffer = new ArrayList<>();   // starts with an empty buffer
            try {
                for (int cycle = 0; cycle < 5; cycle++) {
                    // Exchange empty buffer for producer's full buffer
                    buffer = exchanger.exchange(buffer);
                    System.out.println("Consumer processing: " + buffer.size() + " items");
                    buffer.clear();   // process and empty the buffer
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }, "consumer");

        producer.start();
        consumer.start();
        producer.join();
        consumer.join();
    }
}

This "double-buffer exchange" pattern lets the producer and consumer work on separate buffers simultaneously — the producer fills one buffer while the consumer processes the other, then they swap. This avoids any copying of items (only the buffer references are exchanged) and allows both threads to work without interference.

Common Pitfalls (Senior Level)

1. Using containsKey() + put() instead of putIfAbsent() / computeIfAbsent()

// SLIPPING CONDITION — race between check and put
if (!map.containsKey(key)) { map.put(key, value); }

// CORRECT
map.putIfAbsent(key, value);
// OR for computed values:
map.computeIfAbsent(key, k -> expensiveComputation(k));

2. Using CopyOnWriteArrayList for write-heavy workloads

// Each add() copies the ENTIRE array — catastrophic under frequent writes
List<LogEntry> accessLog = new CopyOnWriteArrayList<>();
// In a high-throughput service logging 10,000 requests/sec, each add() copies the full list
// Use ConcurrentLinkedQueue or LinkedBlockingQueue instead for append-heavy patterns

3. Assuming ConcurrentHashMap iteration sees all in-flight changes

// ConcurrentHashMap's iterator is WEAKLY CONSISTENT — it may or may not reflect
// modifications made after the iterator was created
for (Map.Entry<String, Integer> entry : map.entrySet()) {
    // Another thread calling map.put() right now may or may not appear in this iteration
    // This is guaranteed NOT to throw ConcurrentModificationException — but completeness is not guaranteed
}

This is intentional — it's the price of lock-free reads. For a consistent snapshot, copy to a regular HashMap: new HashMap<>(concurrentMap) (briefly serializes the copy operation).

4. Calling Exchanger.exchange() from only one thread — permanent block

Exchanger<String> exchanger = new Exchanger<>();
// If Thread B never arrives at exchange(), Thread A blocks here forever
String received = exchanger.exchange("my-data");   // ← blocked indefinitely

// Production safety: always use the timeout overload
try {
    String received = exchanger.exchange("my-data", 5, TimeUnit.SECONDS);
} catch (TimeoutException e) {
    // Partner thread never arrived — handle gracefully
}

5. Expecting ConcurrentHashMap.size() to be exact under concurrency

int count = map.size();   // under concurrent modifications, this is an APPROXIMATE value
if (count > 1000) { ... }  // another thread may have added/removed items between size() and this check

size() is intentionally approximate in ConcurrentHashMap — maintaining an exact count would require table-wide locking. Use mappingCount() (Java 8+) for a long value that is also approximate but less prone to overflow. If an exact count is critical, you need external synchronization or a different design.

5. Interview Questions & Answers

Basic

Q1: Why is ConcurrentHashMap preferred over Hashtable and Collections.synchronizedMap()?

Both Hashtable and Collections.synchronizedMap() use a single lock on the entire map for every operation — reads and writes are serialized, meaning only one thread can access the map at a time regardless of what they're doing or which keys they're accessing. ConcurrentHashMap eliminates this bottleneck in two ways: (1) reads are entirely lock-free — node values are volatile, establishing happens-before with prior writes, so any number of threads can read simultaneously with zero synchronization overhead; (2) writes lock at the finest possible granularity — individual bins (array slots), not the whole table — so two writes to keys in different bins can proceed in parallel. Additionally, ConcurrentHashMap provides atomic compound methods (putIfAbsent, computeIfAbsent, compute, merge) that prevent the slipping condition race that raw containsKey() + put() creates. The only things ConcurrentHashMap doesn't support that HashMap does are null keys and null values — these are prohibited to eliminate ambiguity in concurrent contexts.

Q2: What is CopyOnWriteArrayList and what is its fundamental performance tradeoff?

CopyOnWriteArrayList is a thread-safe List implementation where every write operation (add, set, remove) creates a full copy of the backing array, makes the modification on the copy, and atomically replaces the internal array reference with the new copy. Reads never acquire any lock — they dereference the current (immutable) array reference, which is always a consistent snapshot. This produces the tradeoff: reads are extremely fast (zero lock cost, any number of concurrent readers), but writes are O(n) in both time (array copy) and memory (briefly holds old + new array simultaneously). It is appropriate when reads are overwhelmingly more frequent than writes — event listener lists, configuration registries, caches that are loaded once and read constantly. It is inappropriate for any workload with frequent writes, where the O(n) copy per write creates GC pressure and latency spikes.

Q3: What is a slipping condition in ConcurrentHashMap, and how do you fix it?

A slipping condition occurs when a "check-then-act" compound operation — a check (containsKey()) followed by a write (put()) — is not atomic. Between the check and the write, another thread can intervene, changing the state that the check observed. In ConcurrentHashMap, containsKey() and put() are each individually atomic, but the two-step compound operation is not: two threads can both call containsKey("key"), both see false (key absent), both proceed to put("key", value), and both execute the put — a double-insertion, with one thread's write silently overwriting the other's. The fix is to use the atomic compound methods that ConcurrentMap provides: putIfAbsent("key", value) atomically inserts only if the key is absent; computeIfAbsent("key", k -> value) atomically computes and inserts if absent; compute("key", (k, v) -> newV) atomically updates based on the current value. These methods execute the check-and-act as a single atomic operation at the bin level, with no window for another thread to intervene.

Intermediate

Q4: Explain ConcurrentHashMap's bin-level locking mechanism and why it is superior to segment-level locking from pre-Java 8.

Segment-level locking (pre-Java 8) divided the array into a fixed number of segments (default 16), each with a ReentrantLock. Up to 16 threads could write in parallel if they mapped to different segments, but all keys hashing to the same segment still serialized. The segment count was fixed at construction — if your load factor was higher than the initial segment count, effective parallelism was bounded. Java 8's bin-level locking eliminates segments entirely. The lock granularity is now a single array slot (bin) — a single key's "bucket." Two writes to different keys that hash to different bins use completely different synchronized objects and have zero mutual blocking. The number of "effective locks" equals the number of occupied bins — dynamically scaling with actual data density. Additionally, writes to empty bins use CAS (compare-and-swap, a lock-free atomic CPU instruction) rather than any lock at all — the absolute minimal synchronization cost for the common case of writing to a new key.

Q5: Two threads both call map.computeIfAbsent("user-123", k -> loadUserFromDB(k)) simultaneously on a ConcurrentHashMap. How many times is loadUserFromDB() called, and what value ends up in the map?

In Java 8+, computeIfAbsent acquires the bin lock before invoking the mapping function if the key is absent. This means only one of the two threads will find the bin in "absent" state and execute the mapping function — the second thread, when it acquires the bin lock, will find the key already present (inserted by the first thread) and return the existing value without calling the mapping function again. So loadUserFromDB() is called exactly once, and both threads see the same value in the map. This is the entire reason computeIfAbsent exists as an atomic operation — it prevents the "double computation" problem that would occur with the non-atomic containsKey() + put() pattern where both threads could each independently call loadUserFromDB() and potentially race to insert conflicting values.

Advanced

Q6: Explain why ConcurrentHashMap's iteration is "weakly consistent" rather than fail-fast, what this means in practice, and describe a production scenario where this distinction matters.

Fail-fast iterators (like HashMap's) maintain a modCount and throw ConcurrentModificationException if the count changes during iteration — a structural modification while iterating is always an error. This requires some form of detection that there's been a modification, which in a concurrent context would require locking the table during the entire iteration. ConcurrentHashMap takes the opposite approach — weakly consistent iterators: the iterator may or may not reflect structural modifications made after the iterator was created, but it never throws ConcurrentModificationException. Each next() call reads nodes using the same lock-free volatile reads as get() — if a node was inserted after the iterator passed that bin, it might not appear; if a node was deleted after the iterator already returned it, the iteration doesn't "un-return" it. In practice this matters for cache invalidation: if you're iterating the map to find expired entries while another thread might be inserting new entries, the weakly consistent iterator guarantees you'll see all entries that existed at iterator creation, but may also see some new entries and may miss some that were deleted mid-iteration. For use cases requiring a true consistent snapshot (e.g., taking a point-in-time inventory), copy the entire map atomically: new HashMap<>(concurrentMap) — this briefly acquires all bin locks during the copy operation, producing a fully consistent snapshot at that instant.
