Java Multithreading: Memory Model & Synchronization

Principal Architect Notes — Production-Grade Mastery Series

Syllabus Coverage: Lectures 20–25 (Memory Management · synchronized keyword · Intrinsic Locks · Synchronized Blocks · Object vs Class Locking)

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:
Synchronization in Java is the mechanism by which multiple threads coordinate access to shared heap memory — using the synchronized keyword to ensure that only one thread at a time can execute a critical section, enforced by an intrinsic lock that every Java object automatically carries.

The Architecture Analogy

Picture a shared office whiteboard that multiple employees (threads) need to update. The whiteboard is on the wall of the shared conference room (heap memory). Everyone's personal notepad (stack memory) is private — no one else can read or write to it.

The problem: if two employees simultaneously walk up and start updating the whiteboard, the result is a garbled mess. Neither change is correct.

The solution: a room key (intrinsic lock). Before updating the whiteboard, an employee must pick up the key from the hook by the door. While they hold the key, the door is locked — no other employee can enter. When they're done, they hang the key back up, and the next employee can enter.

The critical insight: Every office object — the whiteboard, the projector, the coffee machine — has exactly one key. If you put both the whiteboard and the projector behind the same key, then one employee updating the whiteboard blocks another from turning on the projector — even though those are completely independent operations. This is the core problem with naively synchronizing on this: all synchronized methods on an object share the same single key.

Synchronized blocks with custom lock objects are like installing separate keys for each piece of equipment. Now the whiteboard key and the projector key are independent, and both can be used simultaneously by different employees.

How the JVM Sees It

Every Java object has a header in heap memory that contains a monitor (also called the intrinsic lock or mutex). This is not something you instantiate — it's built into every Object by the JVM.

When a thread executes synchronized(obj) or enters a synchronized method:

  1. The JVM executes the monitorenter bytecode instruction on obj's monitor
  2. If the monitor is unowned, the thread acquires it (sets the owner to itself, count = 1)
  3. If the monitor is owned by this same thread (re-entrant), the count increments
  4. If the monitor is owned by a different thread, the current thread blocks in BLOCKED state — zero CPU — until the monitor is released
  5. On exit from the synchronized block/method, the JVM executes monitorexit, decrementing the count. When count reaches 0, the monitor is released and one waiting thread is chosen (non-deterministically) to acquire it

The JMM guarantee: The synchronized keyword establishes a happens-before relationship. Everything a thread does inside a synchronized block happens-before anything another thread does after acquiring the same lock. This gives you both mutual exclusion and memory visibility — writes inside the block are flushed to main memory on exit, and reads inside the block see the latest values from main memory on entry.

2. Course Content: Key Concepts & Definitions

Stack Memory vs. Heap Memory — The Root of the Problem

Property Stack Memory Heap Memory
What lives here Method calls, local variables, reference variables Objects and their instance/static fields
Scope Private per thread — completely invisible to other threads Shared across all threads in the same process
Lifetime Exists for the duration of the method call; automatically managed Exists until garbage collected
Size Small (512KB–1MB per thread) but fast Large but slower to access
Thread safety Inherently safe — no other thread can touch it Not safe by default — requires synchronization 
class Task implements Runnable {
    @Override
    public void run() {
        int number = 0;           // LOCAL VARIABLE → lives on THIS thread's stack
        String name = "Alice";    // LOCAL VARIABLE → lives on THIS thread's stack
        // These are 100% thread-safe. No other thread can see them.
    }
}

class DataPoint {
    private int value = 10;  // INSTANCE FIELD → lives on the heap
    // If two threads hold a reference to the same DataPoint object,
    // both can read and write 'value' — race condition without synchronization
}

The core rule: Local variables inside run() are on the thread's private stack — completely safe, no synchronization needed. Fields on heap objects shared between threads — always need protection.

The Race Condition — Why counter++ Is Not Atomic

A race condition occurs when the correctness of a program depends on the relative timing of multiple threads accessing shared data. The classic example:

private static int counter = 0;   // on the heap — shared between threads

public static void increment() {
    counter++;   // LOOKS like one operation. It is actually THREE:
    // 1. READ:  load the current value of counter from heap into a CPU register
    // 2. ADD:   increment the register value by 1
    // 3. WRITE: store the new value back to the heap
}

The race: Two threads can both execute step 1 before either executes step 3:

Thread 1: READ counter = 0
Thread 2: READ counter = 0    ← reads BEFORE Thread 1 writes back
Thread 1: ADD → 1
Thread 2: ADD → 1
Thread 1: WRITE counter = 1
Thread 2: WRITE counter = 1   ← overwrites Thread 1's write!

Result: counter = 1  (expected: 2) — one increment was lost

This is called a lost update and is the most common concurrency bug. Running 2 threads each calling increment() 1000 times produces a result less than 2000 — and the value differs every run because it depends on OS scheduling.

The synchronized Keyword — Method-Level

private static int counter = 0;

// Synchronized method: only one thread can execute this at a time
public static synchronized void increment() {
    counter++;   // now atomic from the caller's perspective
}

When a thread enters a synchronized method:

  • It acquires the intrinsic lock of the object (or the Class object for static methods)
  • All other threads trying to enter any synchronized method on the same object must wait in BLOCKED state
  • When the thread exits the method (normally or via exception), the lock is automatically released
  • The JVM picks one waiting thread to acquire the lock next (non-deterministic — no fairness guarantee)

Demo: race condition fixed vs unfixed:

// WITHOUT synchronized: result varies, always < 2000
// WITH synchronized: result is always exactly 2000
Thread t1 = new Thread(() -> {
    for (int i = 0; i < 1000; i++) increment();
});
Thread t2 = new Thread(() -> {
    for (int i = 0; i < 1000; i++) increment();
});
t1.start(); t2.start();
t1.join();  t2.join();
System.out.println("Counter: " + counter);  // always 2000 with synchronized

The Single-Lock Problem — When synchronized Methods Hurt Throughput

Every Java object has exactly one intrinsic lock. Synchronizing multiple methods on the same object means they all compete for that single lock — even if they are logically independent.

class SharedResource {
    private int counter1 = 0;
    private int counter2 = 0;

    public synchronized void incrementOne() { counter1++; }  // ← lock on 'this'
    public synchronized void incrementTwo() { counter2++; }  // ← SAME lock on 'this'
}

If Thread 1 calls incrementOne() and holds the lock, Thread 2 trying to call incrementTwo() must wait — even though counter1 and counter2 are completely independent variables. This serializes two operations that could safely run in parallel, cutting throughput in half for no reason.

Illustration:

Thread 1: [holds lock → incrementOne()] ────────────────────►
Thread 2: [blocked, waiting...         ] [lock free → incrementTwo()]►

Instead of:
Thread 1: [incrementOne()] ─────────────────────────────────►
Thread 2: [incrementTwo()] ─────────────────────────────────►
            ↑ both running in parallel — twice the throughput

Synchronized Blocks — Fine-Grained Control

synchronized blocks let you:

  1. Lock only the specific lines that need protection (not the entire method)
  2. Choose which object to lock on — enabling multiple independent locks
class SharedResource {
    private int counter1 = 0;
    private int counter2 = 0;

    // Two independent locks — one per counter
    private final Object lock1 = new Object();
    private final Object lock2 = new Object();

    public void incrementOne() {
        // Code before the block: no lock held, any thread can execute
        doSomeNonCriticalPreWork();

        synchronized (lock1) {          // ← acquires lock1 only
            counter1++;                 // protected by lock1
        }                               // ← lock1 released here

        doSomeNonCriticalPostWork();
        // Code after block: lock1 already released
    }

    public void incrementTwo() {
        synchronized (lock2) {          // ← acquires lock2 — INDEPENDENT of lock1
            counter2++;
        }
    }
}

Now Thread 1 executing incrementOne() and Thread 2 executing incrementTwo() run truly in parallel — they lock on different objects so neither blocks the other.

The golden rule: Only put the minimum necessary code inside a synchronized block. Every extra line inside the block is a line that serializes threads unnecessarily.

Object-Level Locking vs. Class-Level Locking

Object-level locking — synchronized(this) or synchronized instance method:

  • Acquires the intrinsic lock of the specific object instance
  • Two different instances of the same class have different locks — they don't block each other
class Counter {
    private int count = 0;

    public synchronized void increment() { count++; }
    // equivalent to:
    public void increment() {
        synchronized (this) { count++; }   // 'this' = this Counter instance
    }
}

Counter c1 = new Counter();
Counter c2 = new Counter();
// c1.increment() and c2.increment() can run in parallel — different locks

Class-level locking — synchronized(ClassName.class) or synchronized static method:

  • Acquires the lock of the Class object (one per class, shared by all instances)
  • All instances of the class share this single lock — they block each other
class Counter {
    private static int globalCount = 0;

    public static synchronized void increment() { globalCount++; }
    // equivalent to:
    public static void increment() {
        synchronized (Counter.class) { globalCount++; }   // single lock for all instances
    }
}

Counter c1 = new Counter();
Counter c2 = new Counter();
// c1.increment() and c2.increment() CANNOT run in parallel — same class-level lock

Demonstration of the difference:

class ObjectLocking {
    public synchronized void instanceMethod() {
        System.out.println(Thread.currentThread().getName() + " entered");
        Thread.sleep(2000);
        System.out.println(Thread.currentThread().getName() + " exited");
    }
}

// With object-level locking:
ObjectLocking obj1 = new ObjectLocking();
ObjectLocking obj2 = new ObjectLocking();
new Thread(obj1::instanceMethod, "T1").start();  // locks obj1's monitor
new Thread(obj2::instanceMethod, "T2").start();  // locks obj2's monitor — different!
// T1 and T2 run in parallel — output: both "entered" before either "exited"

// If instanceMethod were STATIC (class-level locking):
// T1 and T2 would serialize — T2 waits 2 seconds for T1 to finish

Guidance: Avoid class-level locking unless you genuinely need to protect static shared state that must be serialized across all instances. Class-level locks are the widest possible lock scope and kill concurrency across the entire class.

3. Senior-Level Practical Implementation

Complete Example: The Full Progression from Bug to Correct

package com.example.sync;

public class SynchronizationProgression {

    // ── Stage 1: THE BUG ───────────────────────────────────────────────────
    static class UnsafeCounter {
        private int count = 0;
        public void increment() { count++; }   // NOT atomic — race condition
        public int getCount() { return count; }
    }

    // ── Stage 2: OVER-SYNCHRONIZED (correct but slow) ─────────────────────
    static class OverSynchronizedResource {
        private int counter1 = 0;
        private int counter2 = 0;

        // Both methods share the same intrinsic lock — serializes independent work
        public synchronized void incrementOne() { counter1++; }
        public synchronized void incrementTwo() { counter2++; }
    }

    // ── Stage 3: FINE-GRAINED (correct AND fast) ──────────────────────────
    static class WellSynchronizedResource {
        private int counter1 = 0;
        private int counter2 = 0;

        private final Object lock1 = new Object();   // dedicated lock per resource
        private final Object lock2 = new Object();

        public void incrementOne() {
            synchronized (lock1) { counter1++; }     // only blocks incrementOne callers
        }
        public void incrementTwo() {
            synchronized (lock2) { counter2++; }     // completely independent of lock1
        }
        public int getCounter1() { return counter1; }
        public int getCounter2() { return counter2; }
    }

    // ── Demonstration ─────────────────────────────────────────────────────
    public static void main(String[] args) throws InterruptedException {
        WellSynchronizedResource resource = new WellSynchronizedResource();

        // Thread 1 increments counter1, Thread 2 increments counter2 — no contention
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 10_000; i++) resource.incrementOne();
        }, "counter1-worker");

        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 10_000; i++) resource.incrementTwo();
        }, "counter2-worker");

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        // Both must be exactly 10000 — fine-grained locking gives correctness + parallelism
        System.out.println("Counter1: " + resource.getCounter1());  // 10000
        System.out.println("Counter2: " + resource.getCounter2());  // 10000
    }
}

Real-World Use Case: Bank Account with Synchronized Block

A BankAccount where deposit and withdraw must be atomic, but getBalance read-only calls shouldn't unnecessarily block each other (note: reads still need synchronization for visibility — demonstrated here with the same lock for simplicity):

public class BankAccount {
    private double balance;
    private final Object balanceLock = new Object();   // dedicated lock
    private final String accountId;

    public BankAccount(String accountId, double initialBalance) {
        this.accountId = accountId;
        this.balance = initialBalance;
    }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Amount must be positive");
        synchronized (balanceLock) {
            // Critical section: only the balance update needs protection
            // Input validation above can run without the lock
            balance += amount;
            System.out.printf("[%s] Deposited %.2f. New balance: %.2f%n",
                Thread.currentThread().getName(), amount, balance);
        }
    }

    public boolean withdraw(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Amount must be positive");
        synchronized (balanceLock) {
            if (balance < amount) {
                System.out.printf("[%s] Insufficient funds. Balance: %.2f%n",
                    Thread.currentThread().getName(), balance);
                return false;
            }
            balance -= amount;
            System.out.printf("[%s] Withdrew %.2f. New balance: %.2f%n",
                Thread.currentThread().getName(), amount, balance);
            return true;
        }
    }

    public double getBalance() {
        synchronized (balanceLock) { return balance; }
    }

    public static void main(String[] args) throws InterruptedException {
        BankAccount account = new BankAccount("ACC-001", 1000.0);

        // Simulate concurrent deposits and withdrawals
        Thread[] threads = new Thread[10];
        for (int i = 0; i < 5; i++) {
            threads[i] = new Thread(() -> account.deposit(100.0), "depositor-" + i);
            threads[i + 5] = new Thread(() -> account.withdraw(80.0), "withdrawer-" + i);
        }
        for (Thread t : threads) t.start();
        for (Thread t : threads) t.join();

        System.out.printf("Final balance: %.2f%n", account.getBalance());
        // Without synchronization: balance is unpredictable and almost certainly wrong
        // With synchronization: balance is always exactly 1000 + (5×100) - (5×80) = 1100
    }
}

4. Deep Dive: Tradeoffs, Architecture & Performance

The monitorenter/monitorexit Bytecode Layer

The synchronized keyword compiles to two bytecode instructions:

  • monitorenter: acquire the monitor of the object on the operand stack
  • monitorexit: release the monitor — appears twice in compiled code (once for normal exit, once for exception path) to guarantee release even if an exception is thrown

This is why synchronized methods automatically release the lock on exception — the compiler generates the monitorexit in a synthetic finally block. You get this guarantee for free, which ReentrantLock does not give you automatically (you must use try/finally explicitly — covered in the next set of notes).

Performance Impact of Synchronization

Uncontended lock: Modern JVMs use "biased locking" and "lightweight locking" optimizations. An uncontended synchronized block on a modern HotSpot JVM costs ~2–5 nanoseconds — essentially negligible. The JIT compiler can even eliminate it entirely (lock elision) if it proves no other thread can reach the object.

Contended lock: When threads genuinely compete, the cost rises dramatically. The blocked thread enters BLOCKED state (OS thread parks), which involves a system call. When the lock is released, the OS must wake the waiting thread — another system call. This round trip is typically 1–10 microseconds, making contended synchronization ~1000x more expensive than uncontended.

The performance lesson: The cost of synchronized is not in acquiring an uncontended lock — it's in contention. Minimize contention by: (1) making critical sections as small as possible, (2) using fine-grained locks to reduce the scope of what's protected, (3) ensuring threads spend as little time inside synchronized blocks as possible.

Tradeoff Matrix: Synchronization Granularity

Approach Correctness Throughput Complexity Risk
No synchronization ❌ Race conditions High Low Data corruption
synchronized whole method ✅ Correct Low (over-serialized) Low Performance bottleneck
synchronized(this) block ✅ Correct Medium (still one lock) Medium Still serializes independent ops
synchronized on custom objects ✅ Correct High (parallel independent ops) Medium Must final the lock objects
Class-level lock (ClassName.class) ✅ Correct Very low (global serialization) Low Kills all concurrency for the class

Common Pitfalls at This Stage

1. Locking on a mutable or non-final object

// DANGEROUS: if lockObj is ever reassigned, the lock changes
private Object lockObj = new Object();
lockObj = new Object();  // now threads that checked the old object proceed simultaneously

// CORRECT: final ensures the lock reference never changes
private final Object lockObj = new Object();

Always declare custom lock objects final. The lock object itself is never modified — only the data it protects changes.

2. Synchronizing on a String literal or Integer

// VERY DANGEROUS: String literals are interned — shared across the entire JVM
private final String lock = "MY_LOCK";
synchronized (lock) { ... }  // could inadvertently share the lock with unrelated code
                               // using the same string literal anywhere in the JVM

// CORRECT: always use a dedicated new Object()
private final Object lock = new Object();

3. Making Lock Objects Public (Lock Escape)

// DANGEROUS: if the lock object is accessible externally, other code can synchronize
// on it too — creating hidden ordering dependencies and potential deadlocks
public final Object lock = new Object();  // ← exposed, can be used externally

// CORRECT: always private
private final Object lock = new Object();

4. Holding Locks During I/O or Other Long Operations

// BAD: lock held for entire duration including slow I/O
public synchronized void processAndSave(Data d) {
    process(d);             // fast — fine inside lock
    saveToDatabase(d);      // SLOW — holds lock during entire DB call
    // Other threads blocked for hundreds of milliseconds
}

// BETTER: do I/O outside the lock, only lock the in-memory state change
public void processAndSave(Data d) {
    process(d);
    byte[] serialized = serialize(d);  // prepare data outside lock
    synchronized (lock) {
        inMemoryCache.put(d.id(), d);  // only the fast memory write is locked
    }
    saveToDatabase(serialized);        // slow I/O outside the lock
}

5. Forgetting That synchronized on Different Objects Provides No Mutual Exclusion

// Thread 1 synchronizes on lock1, Thread 2 synchronizes on lock2
// These do NOT exclude each other — they're completely independent
synchronized (lock1) { counter++; }   // Thread 1
synchronized (lock2) { counter++; }   // Thread 2 — runs at same time, race condition!

// For mutual exclusion, both threads MUST synchronize on the SAME object
synchronized (sharedLock) { counter++; }   // Thread 1
synchronized (sharedLock) { counter++; }   // Thread 2 — correctly serialized

5. Interview Questions & Answers

Basic

Q1: What is a race condition and how does synchronized prevent it?

A race condition occurs when multiple threads access shared mutable data concurrently and the final result depends on the order of thread execution — which is non-deterministic. The classic example is counter++, which compiles to three non-atomic operations (read, increment, write). Two threads can both read the same value before either writes back, causing one increment to be lost. synchronized prevents this by ensuring only one thread at a time can execute the protected code block. Under the hood, it uses monitorenter/monitorexit bytecode instructions on the object's intrinsic monitor. The thread that acquires the monitor completes all three sub-operations atomically from other threads' perspectives — no other thread can enter a synchronized block on the same object until the monitor is released.

Q2: What is the difference between synchronizing a method and using a synchronized block?

Synchronizing a method (public synchronized void foo()) acquires the object's intrinsic lock for the entire duration of the method, protecting every line inside it. A synchronized block (synchronized(obj) { ... }) lets you specify exactly which lines need protection and — critically — which object to lock on. The block form gives two advantages: (1) you reduce the critical section to the minimum necessary, allowing more code to run without holding the lock, and (2) you can lock on different objects for different operations, enabling logically independent operations to run in parallel. The rule of thumb in production code: always prefer synchronized blocks over synchronized methods; only expose synchronized methods in the simplest cases where the entire method body genuinely needs serialization.

Q3: What is the intrinsic lock (monitor) and where does it come from?

Every Java object — literally every one, from new Object() to a complex domain class — has an intrinsic lock (also called a monitor) stored in the object's header in heap memory. This is not something you create; it's provided by the JVM for every object. When a thread executes synchronized(obj), the JVM executes a monitorenter bytecode instruction on that object, which atomically checks if the monitor is free and, if so, sets the thread as the owner. On exit, monitorexit decrements the ownership count and, if it reaches zero, releases the monitor, allowing a waiting thread to acquire it. The Class object itself also has a monitor, which is what static synchronized methods use.

Intermediate

Q4: You have two synchronized methods on the same class — incrementA() and incrementB() — operating on two completely independent counters. A code review flags this as a performance problem. Explain exactly why and propose a fix.

The problem is that both methods synchronize on this (the object's single intrinsic lock). Even though counterA and counterB are completely independent state, Thread 1 holding the lock in incrementA() prevents Thread 2 from entering incrementB(), even though those operations have zero logical dependency. In a high-throughput scenario, this halves the effective throughput — both operations are serialized into a single-threaded queue. Fix: replace method-level synchronization with synchronized blocks on dedicated lock objects, one per counter:

private final Object lockA = new Object();
private final Object lockB = new Object();

public void incrementA() { synchronized(lockA) { counterA++; } }
public void incrementB() { synchronized(lockB) { counterB++; } }

Now the two operations are protected by independent monitors — Thread 1 in incrementA() and Thread 2 in incrementB() run truly in parallel.

Q5: What is the difference between object-level locking and class-level locking? When is each appropriate?

Object-level locking (synchronized(this) or synchronized instance methods) acquires the monitor of the specific object instance. Two different instances of the same class have separate monitors — their synchronized methods can run in parallel. This is appropriate when protecting per-instance state (fields that belong to a specific object). Class-level locking (synchronized(ClassName.class) or static synchronized methods) acquires the monitor of the Class object itself, which is shared by all instances. This is appropriate for protecting static shared state that must be consistent across all instances (global counters, class-wide caches). The performance danger: class-level locking serializes all instances — if 100 objects exist, any synchronized static call from any of them blocks all others. Avoid it unless you genuinely need cross-instance serialization.

Q6: A colleague argues that adding synchronized to every method in a class will make it thread-safe. Is this correct? What subtle problem remains?

Incorrect — or at least, incomplete. Making individual methods synchronized gives you method-level atomicity: each individual method call is atomic. But if your class has compound operations — calling two methods in sequence where the result of the first affects the second — thread safety breaks at the compound level. Classic example:

// Both methods are synchronized individually
if (map.containsKey(key)) {      // synchronized check
    return map.get(key);         // synchronized read
    // BUG: another thread can REMOVE the key between these two calls
}

Each individual call is safe; the compound check-then-act is not. This is called a check-then-act race condition. The fix is to synchronize the compound operation as a unit. Additionally, synchronizing every method is often severe over-synchronization — it serializes all access to the object and destroys throughput. Thread safety must be designed at the operation level, not just the method level.

Advanced

Q7: Explain the JMM visibility guarantee provided by synchronized and why it matters beyond mutual exclusion.

synchronized provides two distinct guarantees, and the second is often overlooked. The first is mutual exclusion — only one thread can be inside a synchronized block on a given monitor at a time. The second is memory visibility: the JMM specifies that when a thread releases a monitor (exits a synchronized block), all writes it performed are flushed to main memory. When another thread acquires the same monitor (enters the corresponding synchronized block), it reads the latest values from main memory — it sees all writes that happened before the previous thread released the lock. This establishes a happens-before relationship: everything done inside a synchronized block by Thread A happens-before everything done inside a synchronized block on the same monitor by Thread B (if Thread B acquires it after Thread A releases). Without synchronized, a thread's writes to a shared variable might sit in a CPU register or L1 cache and never be visible to other threads — this is not a hardware bug, it's permitted JVM behavior for unsynchronized code. This is why you cannot fix a visibility problem by relying on synchronized on only one of the two threads — both the writer and the reader must synchronize on the same monitor.

Q8: Two threads are deadlocked. Thread 1 holds lockA and is waiting for lockB. Thread 2 holds lockB and is waiting for lockA. How would you detect this in production and what structural changes prevent it?

Detection: take a thread dump (jstack <pid> or kill -3 <pid>). The JVM's deadlock detector identifies cycles — you'll see output like "Thread-1" waiting to lock <0xABCD> (a java.lang.Object) which is held by "Thread-2" and "Thread-2" waiting to lock <0xEFGH> (a java.lang.Object) which is held by "Thread-1". The JVM explicitly prints "Java-level deadlock:" with the full cycle. Structural prevention: (1) Lock ordering — always acquire multiple locks in a globally consistent order. If all code acquires lockA before lockB, the cycle is impossible. (2) Lock timeout — use ReentrantLock.tryLock(timeout) instead of synchronized; if a lock cannot be acquired within the timeout, release all held locks and retry. synchronized has no timeout — it waits forever, making deadlock permanent. (3) Reduce scope — eliminate the need to hold multiple locks simultaneously by restructuring the data model. (4) Single lock — if two resources are always accessed together, protect them with a single shared lock instead of two separate ones. Note: this is a preview of the next section — ReentrantLock's tryLock() is the production-grade solution to this specific problem.

Q9: A high-throughput service uses a synchronized method that takes 100ms on average (it calls a database). You're seeing 99th-percentile latency of 800ms under load. Walk through your analysis.

100ms average in a synchronized method means every thread must wait for the previous to finish DB call. With 8 threads all calling this method, average queue depth at any moment: (arrival rate × service time). At throughput approaching 1/0.1s = 10 req/s per thread, queue begins building. 800ms p99 means some threads wait 7 additional DB call durations — about 7 threads deep in the wait queue. Root cause: holding the lock during I/O is the cardinal sin of synchronization. The lock is held during the full 100ms DB call when it only needs to be held for in-memory state updates (microseconds). Fix: (1) Extract the I/O out of the synchronized block — prepare the query, build the request outside the lock, execute the DB call outside the lock, then re-acquire the lock only to update the in-memory state with the result. (2) If the purpose of the lock is to deduplicate concurrent requests for the same resource (dog-piling), use a ConcurrentHashMap.computeIfAbsent() pattern to coalesce concurrent requests — this completely eliminates the synchronized method. (3) Measure: add metrics on lock contention time (time spent in BLOCKED state per thread) using JMX ThreadMXBean to confirm the diagnosis before rewriting.

6. Mastery Checklist

After internalizing these notes, you should be able to:

  • Explain the difference between stack memory and heap memory at the thread level, state what lives in each, and articulate why shared heap state requires synchronization while stack state never does
  • Demonstrate the race condition on counter++ by writing out the three-step (read-increment-write) non-atomic sequence and showing how two threads can interleave to produce a lost update
  • Apply the synchronized keyword correctly to both methods and blocks, and explain the monitorenter/monitorexit bytecode instructions it generates
  • Identify the single-lock bottleneck — when multiple synchronized methods on the same object serialize logically independent operations — and refactor to synchronized blocks on separate lock objects
  • Distinguish object-level locking (this) from class-level locking (ClassName.class), state what each serializes, and identify when class-level locking is inappropriately broad
  • Explain the two JMM guarantees of synchronized: mutual exclusion AND memory visibility (happens-before on lock release/acquire)
  • Apply the three rules for correct custom lock objects: always private, always final, always new Object() (never String literals or primitive wrappers)
