Java Multithreading: ReentrantLock, Conditions & Lock Interface

Principal Architect Notes — Production-Grade Mastery Series

Syllabus Coverage: Lectures 26, 31–34 (Re-entrancy · Lock Interface · ReentrantLock · try/finally pattern · Condition · Producer-Consumer with Locks · synchronized vs ReentrantLock)

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:
ReentrantLock is the explicit, programmable replacement for the synchronized keyword — it provides the same mutual exclusion guarantee but adds timeout-based locking, interruptible lock acquisition, fairness ordering, and multiple independent condition queues, eliminating every major shortcoming of synchronized.

The Architecture Analogy

synchronized is like a vending machine with a single built-in coin slot — every machine has one, it works automatically, you can't customize it, and once someone puts in a coin and the mechanism locks, anyone else who wants to use the machine just stands there indefinitely with no option to walk away.

ReentrantLock is like a custom key management desk at a hotel. You go to the desk and explicitly ask for a key (lock()). If someone else has it, you can: wait indefinitely (like synchronized), wait up to 30 seconds and leave if it's not available (tryLock(timeout)), ask to be told the moment a key is free (lockInterruptibly()), or request a fair queue so the longest-waiting guest gets the key next (new ReentrantLock(true)). You must explicitly return the key when done (unlock()). More importantly, the hotel can give you separate keys for different rooms — one key for the pool, one for the gym — unlike synchronized where the single intrinsic lock covers everything.

Condition objects are the room-specific intercom system. Instead of a single announcement speaker (the object's one wait set), each Condition is a dedicated channel: producers wait on the "buffer not full" intercom, consumers wait on the "buffer not empty" intercom. You signal exactly the right channel — no random wakeups.

How the JVM Sees It

Unlike synchronized (which is a JVM-level keyword compiled to monitorenter/monitorexit bytecode), ReentrantLock is a pure Java class in java.util.concurrent.locks, built on top of AbstractQueuedSynchronizer (AQS).

AQS maintains:

  • An int state field (for ReentrantLock: 0 = unlocked, >0 = locked, value = re-entry count)
  • A CLH queue (Craig, Landin, and Hagersten queue) of waiting threads — a doubly-linked list of nodes, each representing a parked thread
  • For fair mode: strict FIFO ordering of the CLH queue
  • For unfair mode: new arrivals can "barge in" and acquire the lock if it's momentarily free, even if others are queued

lock() uses compareAndSet(0, 1) on the state field (atomic CAS operation) to acquire. On failure, the thread is added to the AQS queue and parked via LockSupport.park(). unlock() decrements state and calls LockSupport.unpark() on the next queued thread.

Condition objects each have their own wait queue (a separate linked list of parked threads). await() moves the calling thread from "lock holder" to "condition wait queue." signal() moves the first thread from the condition queue back to the AQS lock queue — it must still re-acquire the lock before proceeding.

2. Course Content: Key Concepts & Definitions

Re-Entrancy — What It Means and Why It Matters

A lock is re-entrant if a thread that already holds the lock can acquire it again without blocking. Both synchronized and ReentrantLock are re-entrant.

public class ReentrantExample {

    public synchronized void outerMethod() {
        System.out.println("Entered outerMethod");
        innerMethod();   // ← calls another synchronized method on 'this'
        System.out.println("Exiting outerMethod");
    }

    public synchronized void innerMethod() {
        System.out.println("Entered innerMethod");
        // If locks were NOT re-entrant, the thread would deadlock here:
        // it already holds 'this' lock and is now trying to acquire 'this' lock again
        System.out.println("Exiting innerMethod");
    }
}

What happens under the hood: When a thread holding this's intrinsic lock calls innerMethod(), the JVM checks the monitor owner. Since the current thread IS the owner, it simply increments the re-entry count (from 1 to 2) and enters. On exit from innerMethod(), the count decrements to 1. On exit from outerMethod(), it decrements to 0 and the lock is released.

Without re-entrancy, any call from a synchronized method to another synchronized method on the same object would immediately deadlock — the thread would block waiting for itself to release the lock it already holds. This would make synchronized nearly unusable in practice.

ReentrantLock tracks re-entry the same way: each lock() call increments the hold count, each unlock() decrements it. The lock is only truly released when the hold count reaches 0.

ReentrantLock lock = new ReentrantLock();

lock.lock();           // hold count = 1
lock.lock();           // hold count = 2 (re-entrant — no block)
lock.unlock();         // hold count = 1
lock.unlock();         // hold count = 0 — lock ACTUALLY released

The Lock Interface and ReentrantLock

import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

// Lock is the interface — defines the contract
// ReentrantLock is the concrete implementation
Lock lock = new ReentrantLock();           // unfair (default) — higher throughput
Lock fairLock = new ReentrantLock(true);   // fair — FIFO ordering, prevents starvation

Key methods of the Lock interface:

Method Behavior synchronized equivalent
lock() Acquire lock; block indefinitely if held by another thread Entering a synchronized block
unlock() Release the lock Exiting a synchronized block
tryLock() Acquire if available NOW; return false immediately if not No equivalent
tryLock(time, unit) Acquire within timeout; return false if timeout expires No equivalent
lockInterruptibly() Acquire, but allow interruption while waiting No equivalent
newCondition() Create a Condition bound to this lock No equivalent (one wait set per object)

The Mandatory try/finally Pattern

ReentrantLock does NOT auto-release on exception (unlike synchronized). If code inside a lock() block throws and you haven't put unlock() in a finally block, the lock is held forever — causing all threads waiting for it to deadlock permanently.

lock.lock();
doSomething();     // if this throws, unlock() is never called → permanent deadlock
lock.unlock();

// ─── ALWAYS use try/finally ────────────────────────────────────────────────
lock.lock();
try {
    doSomething();    // even if this throws, finally block guarantees unlock
} finally {
    lock.unlock();    // ALWAYS executed — exception or not
}

This is the single most important operational rule for ReentrantLock. Every production use of lock() must be paired with unlock() in a finally block. No exceptions.

tryLock() — Non-Blocking and Timeout-Based Acquisition

Lock lock = new ReentrantLock();

// Non-blocking attempt: acquire if immediately available, return false otherwise
if (lock.tryLock()) {
    try {
        // Got the lock — do work
    } finally {
        lock.unlock();
    }
} else {
    // Lock not available — do something else instead of blocking
    System.out.println("Lock busy — skipping this cycle");
}

// With timeout: wait up to 500ms, then give up
try {
    if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
        try {
            // Got the lock within timeout
        } finally {
            lock.unlock();
        }
    } else {
        // Timeout expired — lock still held by another thread
        System.out.println("Timed out waiting for lock");
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();   // restore interrupt flag
}

tryLock() is the primary tool for deadlock prevention. If Thread 1 holds lockA and needs lockB, and Thread 2 holds lockB and needs lockA — a classic deadlock — using tryLock() with a timeout lets each thread detect it can't get both locks and back off, release what it holds, and retry.

Fairness: Fair vs Unfair ReentrantLock

// Fair lock — threads acquire in FIFO request order
Lock fairLock = new ReentrantLock(true);

// Unfair lock (default) — threads may "barge in" if lock becomes free
Lock unfairLock = new ReentrantLock(false);   // or just new ReentrantLock()
Property Fair (true) Unfair (false)
Acquisition order Strict FIFO — longest-waiting thread gets lock next Non-deterministic — a new arrival can take the lock before queued threads
Starvation Impossible — every thread eventually gets the lock Possible — a thread could be skipped repeatedly
Throughput Slightly lower — must maintain and check the queue Higher — avoids queue overhead when lock is immediately available
Context switches More — always wakes the queued thread Fewer — new arrival takes the lock without a wake-up cycle
Use when Long-running tasks, fairness is critical, preventing starvation Short critical sections, throughput matters more than order

synchronized is effectively non-deterministic (like unfair) with no way to opt into fairness.

Condition Objects — Targeted Thread Signaling

A Condition is obtained from a lock and provides its own wait/signal queue — completely separate from other Condition objects on the same lock.

Lock lock = new ReentrantLock();
Condition notFull  = lock.newCondition();   // producers wait here when buffer is full
Condition notEmpty = lock.newCondition();   // consumers wait here when buffer is empty
Condition method Equivalent to Behavior
await() Object.wait() Release lock, park thread on THIS condition's queue
signal() Object.notify() Wake one thread from THIS condition's queue
signalAll() Object.notifyAll() Wake all threads from THIS condition's queue

The critical advantage over notify(): With synchronized + notifyAll(), waking up all waiting threads means producers and consumers both wake up — only one proceeds, the rest re-evaluate and go back to waiting. With separate Condition objects, the producer calls notEmpty.signal() to wake exactly one consumer, and the consumer calls notFull.signal() to wake exactly one producer. Zero wasted wakeups, deterministic targeting.

Producer-Consumer with ReentrantLock + Condition

import java.util.LinkedList;
import java.util.List;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class LockBasedBuffer {
    private final List<Integer> buffer = new LinkedList<>();
    private final int capacity;

    private final Lock lock = new ReentrantLock(true);       // fair lock
    private final Condition notFull  = lock.newCondition();  // producers wait here
    private final Condition notEmpty = lock.newCondition();  // consumers wait here

    public LockBasedBuffer(int capacity) { this.capacity = capacity; }

    public void produce(int item) throws InterruptedException {
        lock.lock();
        try {
            while (buffer.size() == capacity) {
                System.out.println("[Producer] Buffer full — awaiting");
                notFull.await();   // releases lock; parks on notFull's queue
            }
            buffer.add(item);
            System.out.println("[Producer] Added: " + item);
            notEmpty.signal();     // wake exactly one waiting consumer
        } finally {
            lock.unlock();         // ALWAYS in finally
        }
    }

    public int consume() throws InterruptedException {
        lock.lock();
        try {
            while (buffer.isEmpty()) {
                System.out.println("[Consumer] Buffer empty — awaiting");
                notEmpty.await();  // releases lock; parks on notEmpty's queue
            }
            int item = buffer.remove(0);
            System.out.println("[Consumer] Removed: " + item);
            notFull.signal();      // wake exactly one waiting producer
            return item;
        } finally {
            lock.unlock();
        }
    }
}

Compare to the synchronized version: producers and consumers have separate waiting queues (notFull vs notEmpty). When a consumer signals, it calls notFull.signal() — guaranteed to wake a producer, not another consumer. No wasted wakeups, no starvation, deterministic behavior.

synchronized vs ReentrantLock — Complete Comparison

Feature synchronized ReentrantLock
Lock acquisition Automatic on block entry Manual — lock()
Lock release Automatic on block exit (even on exception) Manual — unlock() in finally
Timeout ❌ Waits indefinitely ✅ tryLock(time, unit)
Non-blocking attempt ❌ Not possible ✅ tryLock()
Interruptible waiting ❌ Thread blocked on synchronized cannot be interrupted ✅ lockInterruptibly()
Fairness ❌ Non-deterministic ✅ new ReentrantLock(true)
Multiple conditions ❌ One wait set per object ✅ Unlimited Condition objects via newCondition()
Manual unlock from elsewhere ❌ Must exit the block ✅ unlock() can be called from any method
Re-entrancy ✅ Yes ✅ Yes
Deadlock avoidance ❌ No mechanism ✅ tryLock() with timeout + back-off

Ease of use High — automatic management Lower — must remember try/finally
Risk if misused Moderate — over-synchronization High — forgotten unlock() = permanent deadlock

Bottom line: synchronized is appropriate for simple, short critical sections where auto-release on exception is a safety net. ReentrantLock is the tool for anything requiring timeouts, fairness, interruption, or multiple wait conditions. In a codebase where you're already using ReentrantLock, prefer it consistently — mixing the two styles adds cognitive overhead.

3. Senior-Level Practical Implementation

Complete Example: Deadlock-Resistant Account Transfer

The classic deadlock scenario: transferring money between two accounts requires locking both. tryLock() with timeout makes it deadlock-safe:

import java.util.concurrent.locks.*;

public class BankAccount {
    private double balance;
    private final Lock lock = new ReentrantLock();
    private final String id;

    public BankAccount(String id, double balance) {
        this.id = id;
        this.balance = balance;
    }

    /**
     * Transfer amount from this account to target.
     * Uses tryLock() with timeout on BOTH locks to prevent deadlock.
     * If either lock times out, releases all held locks and retries.
     */
    public boolean transfer(BankAccount target, double amount)
            throws InterruptedException {
        int attempts = 0;
        while (attempts < 3) {
            // Try to acquire THIS account's lock
            if (this.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
                try {
                    // Try to acquire TARGET account's lock
                    if (target.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
                        try {
                            if (this.balance < amount) {
                                System.out.println("Insufficient funds in " + this.id);
                                return false;
                            }
                            this.balance -= amount;
                            target.balance += amount;
                            System.out.printf("Transferred %.2f from %s to %s%n",
                                amount, this.id, target.id);
                            return true;
                        } finally {
                            target.lock.unlock();
                        }
                    }
                    // Target lock timed out — release source lock and retry
                } finally {
                    this.lock.unlock();
                }
            }
            // Both locks failed — back off before retry to reduce contention
            attempts++;
            Thread.sleep(10 * attempts);   // exponential backoff
        }
        System.out.println("Transfer failed after retries — deadlock avoidance triggered");
        return false;
    }

    public double getBalance() {
        lock.lock();
        try { return balance; }
        finally { lock.unlock(); }
    }
}

This pattern is impossible with synchronized — there is no way to attempt acquiring a synchronized lock with a timeout.

Real-World Use Case: Rate Limiter with tryLock()

import java.util.concurrent.locks.*;

/**
 * A simple rate limiter: allows at most `maxRequests` per time window.
 * tryLock() prevents request threads from blocking when the limiter is busy updating.
 */
public class RateLimiter {
    private final int maxRequests;
    private int requestCount = 0;
    private long windowStart = System.currentTimeMillis();
    private final long windowMs;
    private final Lock lock = new ReentrantLock();

    public RateLimiter(int maxRequests, long windowMs) {
        this.maxRequests = maxRequests;
        this.windowMs = windowMs;
    }

    /**
     * Returns true if the request is allowed, false if rate limit exceeded.
     * Never blocks — uses tryLock() so callers get an immediate decision.
     */
    public boolean allowRequest() {
        if (!lock.tryLock()) {
            return false;  // lock busy — reject immediately (conservative)
        }
        try {
            long now = System.currentTimeMillis();
            if (now - windowStart > windowMs) {
                windowStart = now;
                requestCount = 0;
            }
            if (requestCount < maxRequests) {
                requestCount++;
                return true;
            }
            return false;
        } finally {
            lock.unlock();
        }
    }
}

lockInterruptibly() — Responsive Shutdown

public class InterruptibleWorker implements Runnable {
    private final Lock lock;

    public InterruptibleWorker(Lock lock) { this.lock = lock; }

    @Override
    public void run() {
        try {
            // Unlike lock(), this CAN be interrupted while waiting
            lock.lockInterruptibly();
            try {
                performWork();
            } finally {
                lock.unlock();
            }
        } catch (InterruptedException e) {
            // Thread was interrupted while waiting for the lock
            // Clean exit — no lock acquired, no unlock needed
            System.out.println(Thread.currentThread().getName() + " interrupted — exiting cleanly");
            Thread.currentThread().interrupt();  // restore flag
        }
    }

    private void performWork() { /* ... */ }
}

With synchronized, a thread blocked waiting to acquire the lock cannot be interrupted — it's stuck until the lock is released by its holder. lockInterruptibly() is essential for responsive shutdown in services where threads may be waiting on locks when a shutdown signal arrives.

4. Deep Dive: Tradeoffs, Architecture & Performance

AQS Internals — Why ReentrantLock Performs Well

AbstractQueuedSynchronizer (AQS) uses a single volatile int state field for all synchronization state. Lock acquisition is a CAS (Compare-And-Swap) operation — a single atomic CPU instruction (no OS system call for uncontended acquisition). Parking waiting threads uses LockSupport.park(), which calls into the OS only when the thread genuinely needs to block. The result: uncontended ReentrantLock is essentially as fast as synchronized (both leverage CAS). Contended behavior differs: synchronized relies on the JVM's built-in monitor inflation mechanism; ReentrantLock uses the explicit AQS queue with predictable behavior.

Performance: Fair vs Unfair Lock

Fair locks maintain strict FIFO via the AQS queue. On every lock acquisition, the thread must check the queue and park itself at the end, even if the lock is briefly free. Unfair locks allow "barging" — if the lock becomes free exactly when a new thread calls lock(), that thread takes it without queuing. Empirically, unfair locks show 2–5x higher throughput for short critical sections because they avoid the overhead of queue management and thread wake-up cycles. Use fair locks only when starvation is a genuine risk (long-running, heterogeneous workloads) or when strict ordering is a correctness requirement.

Common Pitfalls

1. Forgetting unlock() in a finally Block

lock.lock();
process();         // throws RuntimeException
lock.unlock();     // NEVER REACHED — lock held permanently
// Every thread waiting for this lock is now deadlocked indefinitely

// CORRECT — always
lock.lock();
try {
    process();
} finally {
    lock.unlock();   // always runs, even on exception
}

2. Calling unlock() Without Holding the Lock

lock.unlock();   // throws IllegalMonitorStateException if this thread doesn't hold the lock

3. Acquiring Inside try Block (Leaving a Window)

try {
    lock.lock();   // if lock() itself throws (e.g., InterruptedException via lockInterruptibly)
    // ... work ...
} finally {
    lock.unlock(); // called even if lock() failed — IllegalMonitorStateException
}

// CORRECT: lock() outside the try block (lock() on ReentrantLock doesn't throw)
lock.lock();       // ← outside try
try {
    // ... work ...
} finally {
    lock.unlock();
}
// For lockInterruptibly(): catch InterruptedException before the finally block

4. Using Condition.await() Outside lock() Scope

Condition cond = lock.newCondition();
cond.await();   // IllegalMonitorStateException — must hold the lock

// CORRECT
lock.lock();
try {
    while (conditionNotMet) cond.await();
} finally {
    lock.unlock();
}

5. Calling signal() on the Wrong Condition

With multiple conditions, calling notFull.signal() when you want to wake a producer but a consumer is waiting on notEmpty means the consumer never wakes up. Keep condition semantics explicit: notFull = producers wait here / consumers signal here; notEmpty = consumers wait here / producers signal here.

5. Interview Questions & Answers

Basic

Q1: What is re-entrant synchronization and why is it necessary?

Re-entrant synchronization means a thread can acquire the same lock it already holds, without blocking on itself. Both synchronized and ReentrantLock are re-entrant. It's necessary because synchronized methods frequently call other synchronized methods on the same object — common in recursive algorithms, template method patterns, and any class where internal methods also need to be thread-safe. Without re-entrancy, a thread inside synchronized methodA() that calls synchronized methodB() on the same object would try to acquire the lock it already holds, see it as "owned by another thread" (itself), and block — permanently deadlocking with itself. ReentrantLock tracks re-entrancy via a hold count: each lock() call increments it, each unlock() decrements it, and the lock is only released when the count reaches zero.

Q2: What is a Condition object and how does it improve on wait()/notify()?

A Condition is obtained from a Lock via lock.newCondition() and provides its own independent queue of waiting threads, decoupled from the lock's general wait set. await() is equivalent to wait() — it releases the lock and parks the thread on this Condition's queue. signal() is equivalent to notify() — it wakes one thread from this Condition's queue. The improvement over wait()/notify() is that you can have multiple Condition objects per lock, each representing a different condition. In a producer-consumer scenario, producers wait on notFull and consumers wait on notEmpty. When a consumer finishes consuming, it calls notFull.signal() — guaranteed to wake a producer. With synchronized + notify(), you have one wait set and may accidentally wake a consumer instead of a producer, requiring notifyAll() and accepting wasted wakeups. Condition eliminates the non-determinism.

Q3: Why must unlock() always be called in a finally block?

ReentrantLock does not automatically release the lock when an exception is thrown — unlike synchronized, which compiles monitorexit into a synthetic finally block, ensuring release even on exception. If code between lock() and unlock() throws and unlock() is not in a finally block, the lock remains held permanently. Every thread subsequently calling lock() on that lock will block forever — a thread-level deadlock that silently freezes the application under load. The correct pattern is: call lock() outside the try block (so finally doesn't run if lock() itself fails), then wrap all work in try { ... } finally { lock.unlock(); }. This guarantees the lock is always released regardless of what happens inside the critical section.

Intermediate

Q4: tryLock() returns false in your code — what are the two distinct scenarios this represents and how should you handle each differently?

Scenario 1: Lock is held by another thread right now — tryLock() (no argument) returns false immediately if the lock is not free at the instant of the call, even if it becomes free a millisecond later. This is appropriate for cases where "if busy, do something else immediately" is the right behavior — rate limiters, non-blocking cache updates, optimistic UI refresh. Handling: proceed with a fallback path. Scenario 2: Lock was not acquired within the timeout — tryLock(timeout, unit) tried repeatedly for the timeout duration and the lock was never available. This indicates sustained contention. Handling: log a warning (contention at this level is a signal of a design problem), apply exponential backoff and retry, or fail-fast with an appropriate error response. In deadlock-avoidance code (two threads each trying tryLock() on two different locks), a false means "release what you have, back off, and retry the entire multi-lock acquisition sequence from scratch."

Q5: A service uses fair ReentrantLock and achieves 3x lower throughput than with unfair lock under load. Explain exactly why and when you'd accept that tradeoff.

Fair lock (new ReentrantLock(true)) maintains strict FIFO via the AQS queue. Every lock acquisition — even when the lock is momentarily free — requires checking the queue head, potentially queuing the current thread, and waiting for LockSupport.unpark() after the previous holder calls unlock(). This involves OS-level thread scheduling — a system call per acquisition. Unfair lock allows "barging" — if the lock becomes free exactly when a new thread calls lock(), that thread takes it without queuing, avoiding the park/unpark cycle entirely. Under high throughput with many threads and short critical sections, this difference compounds: fair lock incurs a system call per acquisition, unfair does not. The throughput gap is typically 2–5x. Accept the tradeoff when: (1) tasks are long-running and varied — a starvation scenario is realistic and its cost (a thread making no progress indefinitely) exceeds the throughput difference, (2) predictable latency is required — fair lock prevents one thread from experiencing unbounded wait time, (3) the critical section is long enough that the per-acquisition overhead is negligible compared to the time inside the critical section.

Advanced

Q6: Explain the AQS (AbstractQueuedSynchronizer) mechanism that underlies ReentrantLock, and why its CAS-based design makes uncontended locking essentially free.

AQS maintains a single volatile int state field and a CLH doubly-linked queue of waiting threads. For ReentrantLock, state = 0 means unlocked; state > 0 means locked, and the value is the re-entry hold count. Lock acquisition: the acquiring thread executes compareAndSet(0, 1) on the state field — a single atomic CPU instruction (x86: CMPXCHG, ARM: LDREX/STREX) that checks and updates the value atomically without any OS involvement. If the CAS succeeds (state was 0, now 1), the thread has the lock with zero kernel overhead — no system call, no thread context switch, just a single atomic instruction. If the CAS fails (another thread holds the lock), the thread is added to the CLH queue as a Node and calls LockSupport.park() — this invokes the OS to suspend the thread (a real system call, ~1–10μs). On unlock(): state is decremented. If it reaches 0, the head of the CLH queue is checked; if a waiter exists, LockSupport.unpark(waiter) is called — another system call to resume that thread. The key insight: for uncontended locks (the common case in well-designed code), the entire lock/unlock cycle is one CAS instruction — essentially as fast as a regular memory write. The OS overhead only appears when threads genuinely contend, which is exactly when blocking is correct behavior.

Q7: You have a producer-consumer system using synchronized + notifyAll(). Under high load with 20 producer threads and 20 consumer threads, CPU usage is 90% but throughput is low. Diagnose and redesign.

Diagnosis: notifyAll() wakes all 40 threads every time any state change occurs. With 40 threads competing for one monitor, 39 threads wake up, evaluate their conditions (most find conditions not met), and go back to wait() — each transition involves acquiring and releasing the monitor and OS scheduling overhead. This is a thundering herd problem: massive spurious wakeup overhead consuming CPU without useful work. Additionally, one wait set means producers and consumers both wake on every signal, doubling useless work. Redesign with ReentrantLock + two Condition objects: notFull (producers wait here) and notEmpty (consumers wait here). Producers call notEmpty.signal() (wakes exactly one consumer) after adding an item. Consumers call notFull.signal() (wakes exactly one producer) after removing an item. With 20 producers and 20 consumers, each signal wakes exactly one thread of the correct type — zero wasted wakeups. If capacity allows multiple concurrent producers or consumers, signalAll() on the specific condition still wakes only the relevant thread type, not the entire pool. Additional optimization: size the buffer to allow batching — each producer adds multiple items before signaling, reducing lock acquisition frequency per unit of work.

Q8: Design a ReadWriteLock-style scenario where you need multiple readers but exclusive writers. How would ReentrantLock alone fall short, and what does java.util.concurrent.locks.ReadWriteLock provide?

ReentrantLock provides mutual exclusion — only one thread at a time, whether reader or writer. For a read-heavy workload (e.g., an in-memory cache that's read 99% of the time and written 1% of the time), every read acquisition serializes all readers — even though concurrent reads are completely safe (reads don't modify state). Throughput is bounded by single-thread read access. ReadWriteLock (ReentrantReadWriteLock in Java) maintains two locks: a read lock (multiple concurrent holders allowed) and a write lock (exclusive — no readers or other writers while held). rwLock.readLock().lock() allows concurrent acquisition by multiple reader threads; rwLock.writeLock().lock() is exclusive. For a cache: readers call readLock().lock(), look up the entry, readLock().unlock() — all running in parallel. Writers call writeLock().lock(), update the cache, writeLock().unlock() — blocks until all readers finish, then runs exclusively. This is a pattern that ReentrantLock alone cannot express — it always excludes all threads regardless of operation type. Note: ReadWriteLock is not covered in this section of the course — this answer is included as context for where lock design goes next.
