Java Multithreading: wait(), notify() & Inter-Thread Communication

Principal Architect Notes — Production-Grade Mastery Series

Syllabus Coverage: Lectures 27–30 (wait/notify Mechanics · wait vs sleep · Producer-Consumer Pattern · while loop with wait)

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:
wait() and notify() are the JVM's built-in signaling mechanism that allows threads sharing the same intrinsic lock to communicate — one thread voluntarily releases the lock and parks itself to sleep, while another thread does its work and then wakes the parked thread back up.

The Architecture Analogy

Picture a single shared photocopier in an office, with a sign-out sheet (the intrinsic lock) you must sign before using it. Two employees: Alice (Producer) and Bob (Consumer).

Alice arrives first, signs the sheet, and loads 100 pages into the feeder tray (fills the buffer). The tray is now full. Alice puts the copier in "ready to copy" mode and writes on a sticky note: "Tray full — I'm stepping away. Wake me when it's empty." She puts the sticky note on the machine and goes to the break room — but critically, she first returns the sign-out sheet (releases the lock). This is wait().

Bob arrives, signs the sheet (acquires the lock), sees the tray full of loaded pages, and starts copying. When he's finished and the tray is empty, he writes on a sticky note: "Tray empty — Alice can come back." This is notify(). He then finishes whatever else he needs to do, returns the sign-out sheet, and leaves.

Alice is notified and returns to the copier, re-signs the sheet, checks the tray is indeed empty, and loads more pages. The cycle repeats.

The critical difference from sleep(): When Alice uses wait(), she completely surrenders the sign-out sheet (the lock). Any other employee can use the copier. When she uses sleep(), she keeps the sign-out sheet in her pocket while she's in the break room — no one else can use the copier even though she's doing nothing.

How the JVM Sees It

When a thread calls obj.wait():

  1. The thread must already hold obj's intrinsic lock (it must be inside synchronized(obj) or a synchronized method on obj). If not: IllegalMonitorStateException immediately.
  2. The thread atomically releases obj's monitor and moves itself into the wait set — a per-object data structure inside the JVM tracking threads waiting on that monitor.
  3. The thread enters WAITING state (or TIMED_WAITING if wait(timeout) was used) — zero CPU consumption.
  4. The thread stays in the wait set until another thread calls obj.notify() or obj.notifyAll() on the same object.

When another thread calls obj.notify():

  1. The calling thread must also hold obj's monitor.
  2. The JVM picks one thread from obj's wait set at random and marks it eligible to re-acquire the lock.
  3. The notifying thread does not immediately release the lock — it finishes executing its synchronized block first. Only when it exits the synchronized block does the monitor become available.
  4. The awakened thread competes to re-acquire the monitor. Once it does, it returns from wait() and continues.

The spurious wakeup: The Java specification explicitly permits a thread to wake up from wait() without being notified. This is a known JVM/OS-level phenomenon. This is exactly why condition checks must be in a while loop, not an if statement.

2. Course Content: Key Concepts & Definitions

wait() vs sleep() — The Fundamental Distinction

These two methods both pause a thread, but they are fundamentally different in purpose, contract, and behavior:

Dimension Object.wait() Thread.sleep(ms)
Called on Any Object instance (obj.wait()) The Thread class statically (Thread.sleep(ms))
Lock behavior Releases the intrinsic lock of the object it's called on Does NOT release any locks the thread holds
Requires synchronized? Yes — must be inside synchronized(obj) No — can be called anywhere
How to wake up obj.notify() or obj.notifyAll() from another thread Timer expires; or thread.interrupt()
Purpose Thread coordination — "I'm done, someone else take over" Pure delay — "pause for a fixed time, then resume"
Spurious wakeup Yes — must use while loop for condition check No — always sleeps the full duration (unless interrupted)
Exception InterruptedException (thread can be interrupted while waiting) InterruptedException (thread can be interrupted while sleeping) 
// sleep() — locks are HELD during pause
synchronized (lock) {
    Thread.sleep(5000);   // holds the lock for 5 full seconds
                          // ALL other threads waiting on 'lock' are blocked for 5s
}

// wait() — lock is RELEASED during pause
synchronized (lock) {
    lock.wait();   // releases the lock immediately
                   // other threads can now enter synchronized(lock) blocks
                   // this thread parks with zero CPU until notified
}

The wait() / notify() Contract — Three Rules

These are absolute requirements. Violating any of them throws IllegalMonitorStateException:

  1. Must be called on the same object that is locked: If you synchronized(lockObj), you must call lockObj.wait() and lockObj.notify() — not this.wait() or any other object.
  2. Must be called from within a synchronized block or method: wait() and notify() cannot be called outside of a synchronized context on the same object.
  3. notify() does not release the lock immediately: After calling notify(), the notifying thread completes the rest of its synchronized block before the monitor is released. The woken thread only gets to re-acquire the lock after the notifying thread exits the synchronized block.
// CORRECT: synchronized on 'this', wait/notify called on 'this'
public synchronized void produce() throws InterruptedException {
    wait();     // ← called on 'this' (implicit), inside synchronized method on 'this'
}
public synchronized void consume() throws InterruptedException {
    notify();   // ← called on 'this' (implicit), inside synchronized method on 'this'
}

// CORRECT: synchronized block form
synchronized (sharedObj) {
    sharedObj.wait();    // ← synchronized on sharedObj, wait called on sharedObj
}
synchronized (sharedObj) {
    sharedObj.notify();  // ← same object
}

// WRONG — throws IllegalMonitorStateException
notify();                        // not inside any synchronized block
synchronized (lockA) { lockB.notify(); }  // synchronized on A, notifying on B

The Producer-Consumer Pattern with synchronized + wait/notify

This is the canonical use case for wait()/notify(). A producer thread fills a buffer; a consumer thread drains it. They must cooperate so that:

  • The producer waits when the buffer is full
  • The consumer waits when the buffer is empty
import java.util.LinkedList;
import java.util.List;

public class SharedBuffer {
    private final List<Integer> buffer = new LinkedList<>();
    private final int capacity;

    public SharedBuffer(int capacity) {
        this.capacity = capacity;
    }

    // Called by the producer thread
    public synchronized void produce() throws InterruptedException {
        // WHILE (not if) — because of spurious wakeups and multi-consumer scenarios
        while (buffer.size() == capacity) {
            System.out.println("[Producer] Buffer full — waiting");
            wait();   // releases lock; parks until consumer calls notify()
            // When we return here, we do NOT know the buffer is definitely not full
            // — hence we re-check with the while loop
        }

        // Buffer has space — add items
        for (int i = 0; i < capacity; i++) {
            buffer.add(i);
            System.out.println("[Producer] Added: " + i);
        }

        notify();  // wake up a waiting consumer
    }

    // Called by the consumer thread
    public synchronized void consume() throws InterruptedException {
        // WHILE (not if) — guard against spurious wakeups
        while (buffer.size() < capacity) {
            System.out.println("[Consumer] Buffer not full yet — waiting");
            wait();   // releases lock; parks until producer calls notify()
        }

        // Buffer is full — drain it
        while (!buffer.isEmpty()) {
            int item = buffer.remove(0);
            System.out.println("[Consumer] Removed: " + item);
        }

        notify();  // wake up the waiting producer
    }
}
public class Main {
    public static void main(String[] args) {
        SharedBuffer buffer = new SharedBuffer(5);

        Thread producer = new Thread(() -> {
            while (true) {
                try {
                    buffer.produce();
                    Thread.sleep(500);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                }
            }
        }, "producer");

        Thread consumer = new Thread(() -> {
            while (true) {
                try {
                    buffer.consume();
                    Thread.sleep(500);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                }
            }
        }, "consumer");

        producer.start();
        consumer.start();
    }
}

Execution trace:

[Producer] Added: 0, 1, 2, 3, 4    → notify() wakes consumer
[Consumer] Removed: 0, 1, 2, 3, 4  → notify() wakes producer
[Producer] Added: 0, 1, 2, 3, 4    → cycle repeats

Why while Instead of if — Three Reasons

Joshua Bloch's Effective Java Item 81 mandates this. Here's exactly why:

Reason 1: Spurious Wakeups
The JVM specification explicitly allows threads to wake from wait() without any notify() being called. This is an OS/hardware-level phenomenon. With an if, a spuriously woken thread proceeds as if notified — which could be incorrect. With a while, it re-checks the condition and goes back to waiting if the condition isn't actually met.

Reason 2: Multi-Consumer / Multi-Producer Scenarios
With multiple consumer threads waiting:

  • Producer calls notifyAll() — all consumers wake up
  • All consumers compete for the lock
  • First consumer gets the lock, sees buffer is full, drains it
  • Second consumer gets the lock — but the buffer is now empty! With if, it proceeds incorrectly. With while, it re-checks and waits again.

Reason 3: Notify Called Before the Thread Reaches wait()
In poorly sequenced code, notify() can be called before the other thread reaches wait(). The while loop ensures the condition is always verified fresh, regardless of notification timing.

// WRONG — if statement
if (buffer.size() == capacity) {
    wait();   // spurious wakeup? proceeds with full buffer — bug
}
buffer.add(item);   // wrong: buffer may still be full!

// CORRECT — while loop (Joshua Bloch's recommendation)
while (buffer.size() == capacity) {
    wait();   // spurious wakeup? re-checks — safely goes back to waiting
}
buffer.add(item);   // guaranteed: buffer is NOT full when we reach here

notify() vs notifyAll()

notify() notifyAll()
What it does Wakes exactly one waiting thread (chosen randomly by JVM) Wakes all waiting threads on this object's monitor
Performance Better — only one thread wakes and competes Worse — all threads wake, only one gets the lock, rest go back to waiting
Safety Dangerous with multiple threads of different types Safe — each woken thread re-checks its condition
Use when All waiting threads are identical (pure producers or pure consumers) Multiple types of waiting threads — safer default

Rule of thumb: When in doubt, use notifyAll(). The performance difference is usually negligible, and it's much harder to reason about correctness bugs from under-notification than the overhead of extra wakeups.

Limitations of the synchronized + wait/notify Approach

Limitation Why It Matters
Non-deterministic thread selection notify() wakes a random waiting thread — no fairness. A thread can be indefinitely skipped (starved)
No tryLock() equivalent A thread trying to enter a synchronized block waits indefinitely — cannot specify a timeout or give up
Cannot interrupt a blocked thread A thread blocked waiting to acquire a synchronized lock cannot be interrupted — it's stuck until the lock is released
Single wait set per object All threads waiting on an object share one wait set — you cannot have "wait for condition A" vs "wait for condition B" on the same lock
No explicit unlock The lock is tied to the block scope — you cannot release it from a different method or code path

These limitations are exactly why ReentrantLock and Condition exist — they address each of these points explicitly.

3. Senior-Level Practical Implementation

Complete Producer-Consumer with notifyAll() and Multiple Workers

import java.util.ArrayDeque;
import java.util.Queue;

public class MultiWorkerBuffer {
    private final Queue<Integer> buffer = new ArrayDeque<>();
    private final int maxCapacity;
    private volatile boolean shuttingDown = false;

    public MultiWorkerBuffer(int maxCapacity) {
        this.maxCapacity = maxCapacity;
    }

    // Called by producer threads
    public synchronized void put(int item) throws InterruptedException {
        while (buffer.size() >= maxCapacity && !shuttingDown) {
            wait();   // release lock, wait until consumer drains some items
        }
        if (shuttingDown) return;

        buffer.offer(item);
        System.out.printf("[%s] PUT %d (size=%d)%n",
            Thread.currentThread().getName(), item, buffer.size());

        notifyAll();   // wake ALL waiting consumers (safer than notify() in multi-consumer)
    }

    // Called by consumer threads
    public synchronized Integer take() throws InterruptedException {
        while (buffer.isEmpty() && !shuttingDown) {
            wait();   // release lock, wait until producer adds items
        }
        if (buffer.isEmpty()) return null;   // shutdown path

        int item = buffer.poll();
        System.out.printf("[%s] TOOK %d (size=%d)%n",
            Thread.currentThread().getName(), item, buffer.size());

        notifyAll();   // wake ALL waiting producers (some may now have space)
        return item;
    }

    public synchronized void shutdown() {
        shuttingDown = true;
        notifyAll();   // wake all blocked threads so they can check shuttingDown and exit
    }

    public static void main(String[] args) throws InterruptedException {
        MultiWorkerBuffer buf = new MultiWorkerBuffer(3);

        // 2 producers
        Thread[] producers = new Thread[2];
        for (int i = 0; i < 2; i++) {
            final int producerId = i;
            producers[i] = new Thread(() -> {
                try {
                    for (int item = 0; item < 10; item++) {
                        buf.put(producerId * 100 + item);
                        Thread.sleep(100);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }, "producer-" + i);
        }

        // 3 consumers
        Thread[] consumers = new Thread[3];
        for (int i = 0; i < 3; i++) {
            consumers[i] = new Thread(() -> {
                try {
                    while (true) {
                        Integer item = buf.take();
                        if (item == null) break;   // shutdown signal
                        Thread.sleep(200);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }, "consumer-" + i);
        }

        for (Thread t : producers) t.start();
        for (Thread t : consumers) t.start();
        for (Thread t : producers) t.join();

        buf.shutdown();
        for (Thread t : consumers) t.join();
        System.out.println("All done.");
    }
}

The Diagnostic Pattern: Identifying wait()/notify() Bugs in Production

When wait()/notify() code misbehaves, the symptoms are:

  • Threads stuck in WAITING forever — notify() was called before wait(), or notify() woke the wrong thread and the correct waiting thread is never woken again
  • if instead of while — thread proceeds with invalid state after spurious wakeup
  • notify() vs notifyAll() — with multiple thread types, notify() can wake the wrong type, leaving a correct waiter permanently stuck

To diagnose: jstack <pid> and look for threads in java.lang.Object.wait(). Check which object's monitor they're waiting on. Then find the thread that holds that monitor and trace why it isn't calling notify().

4. Deep Dive: Tradeoffs, Architecture & Performance

Performance: When wait()/notify() Is the Right Tool

wait()/notify() has essentially zero overhead when threads are not contending. A thread parked in wait() consumes no CPU — it is removed from the OS scheduler's run queue entirely. The cost only appears at notify() time when the OS must wake the waiting thread (~1–10 microseconds for the kernel context switch back to RUNNABLE).

Compare to busy-waiting (the alternative without wait()):

// Busy-waiting — NEVER do this
while (buffer.isEmpty()) {
    // spin-checking continuously — 100% CPU usage while waiting
    // this is what you're AVOIDING with wait()
}

wait() is the correct way to block a thread that's waiting for a condition to change — zero CPU, OS manages the wakeup.

Thread Starvation Risk with notify()

With multiple threads waiting and notify() (single wakeup), the JVM picks randomly. In a scenario with 3 consumer threads, the same consumer can be woken repeatedly while the other two starve — they remain in the wait set forever. notifyAll() prevents starvation at the cost of extra wakeups (but threads that don't need to proceed simply re-evaluate their while condition and go back to waiting cheaply).

Common Pitfalls

1. wait() Outside synchronized Block

// Throws IllegalMonitorStateException immediately at runtime
obj.wait();   // not inside synchronized(obj) block

2. notify() on a Different Object Than wait()

synchronized (lockA) {
    lockA.wait();     // waiting on lockA's wait set
}
// ... another thread ...
synchronized (lockB) {
    lockB.notify();   // notifies lockB's wait set — WRONG, lockA's waiter is never woken
}

3. Using if Instead of while (Spurious Wakeup / Multi-Consumer Bug)

// WRONG
synchronized (this) {
    if (buffer.isEmpty()) wait();   // can wake spuriously and proceed to remove on empty buffer
    buffer.remove();                // NullPointerException / NoSuchElementException
}

// CORRECT
synchronized (this) {
    while (buffer.isEmpty()) wait(); // guaranteed: buffer not empty when we proceed
    buffer.remove();
}

4. Not Calling notifyAll() on Shutdown

// Without notifyAll() on shutdown:
// Consumer threads waiting in wait() are never woken up
// JVM hangs forever with user threads stuck in WAITING

public synchronized void shutdown() {
    this.running = false;
    notifyAll();   // wake all waiting threads so they can check 'running' and exit
}

5. Interview Questions & Answers

Basic

Q1: What is the purpose of wait() and notify() and why must they be called inside a synchronized block?

wait() and notify() enable two threads sharing the same intrinsic lock to coordinate their work — specifically, for one thread to signal the other that a condition has changed. wait() causes the calling thread to release the monitor and park itself with zero CPU consumption until another thread calls notify() or notifyAll() on the same object. notify() wakes one waiting thread so it can re-acquire the monitor and continue. They must be called inside a synchronized block because they operate on the object's monitor — a thread cannot release a lock it doesn't hold, and notifying a wait set on an unowned monitor makes no sense. Calling either outside synchronized throws IllegalMonitorStateException. The contract: always synchronized(obj) before obj.wait() or obj.notify().

Q2: What is the difference between wait() and sleep()?

Four key differences: (1) Called on: wait() is called on an Object (obj.wait()); sleep() is a static method on Thread (Thread.sleep(ms)). (2) Lock release: wait() releases the intrinsic lock of the object it's called on; sleep() holds all locks the thread owns for the entire sleep duration — it's a selfish pause. (3) Wake condition: wait() requires notify() or notifyAll() from another thread (or a timeout, or interrupt) to wake up; sleep() wakes after the specified duration elapses, or on interrupt(). (4) Context requirement: wait() must be called inside a synchronized block on the same object; sleep() has no such requirement. The conceptual distinction: sleep() means "pause for a fixed time, do not release resources"; wait() means "I'm blocked on a condition — let another thread proceed, and wake me when the condition changes."

Q3: Why should you always use a while loop instead of an if statement when calling wait()?

Three reasons: (1) Spurious wakeups — the Java specification explicitly permits threads to wake from wait() without any notify() being called. With an if, a spuriously woken thread proceeds as if the condition were met — producing incorrect behavior. With a while, it re-checks the condition and goes back to waiting if necessary. (2) Multiple waiters of the same type — if notifyAll() is called, every waiting thread wakes up. Only one gets the lock and consumes the resource; the others re-acquire the lock to find the condition no longer holds. With if, those threads proceed incorrectly; with while, they correctly re-wait. (3) Defensive correctness — a while loop makes the code correct regardless of notification source, spurious or genuine. This is Joshua Bloch's Item 81 in Effective Java: "always use the wait loop idiom to invoke the wait method; never invoke it outside of a loop."

Intermediate

Q4: The producer-consumer pattern works correctly with two threads but breaks with three consumer threads. What is the likely bug and how do you fix it?

The bug is almost certainly notify() instead of notifyAll() in the producer. With notify(), the producer wakes exactly one random waiting consumer. If that consumer completes its work and the buffer is now empty, it calls notify() to wake the producer. But the other two waiting consumers are still in the wait set — never notified. The producer produces more items and calls notify() again, potentially waking the same consumer (non-deterministic), leaving the other two starved indefinitely. Fix: replace every notify() call with notifyAll() to wake all waiting threads. Each woken thread re-checks its while condition — threads whose conditions aren't met simply go back to waiting. The overhead of extra wakeups is minimal compared to the correctness guarantee. Alternatively, restructure to use ReentrantLock with separate Condition objects for producers and consumers — allowing targeted signaling without the non-determinism problem.

Q5: Explain the sequence of events when wait() is called and then notify() is called from another thread.

Step 1: Thread A, inside synchronized(obj), calls obj.wait(). Atomically: Thread A releases obj's monitor, Thread A moves to obj's wait set, Thread A's state changes to WAITING. Step 2: Thread B is now able to acquire obj's monitor (it was waiting for it or now competes for it). Thread B enters its synchronized block and does its work. Step 3: Thread B calls obj.notify(). The JVM marks one thread from obj's wait set (chosen at random) as eligible to re-acquire the monitor. Thread B does not lose the lock at this point. Step 4: Thread B exits the synchronized block — monitorexit executes, releasing obj's monitor. Step 5: The awakened Thread A (and any other eligible threads) compete for obj's monitor. Thread A acquires it, exits wait(), and resumes execution at the line after wait(). Step 6: Thread A re-evaluates the while loop condition. If the condition is now satisfied, it proceeds; if not (spurious wakeup or condition not yet met), it calls wait() again.

Advanced

Q6: Two threads are stuck — both in WAITING state on the same object, and neither is ever woken. What are the three most likely root causes and how do you diagnose each?

Root cause 1: notify() called before both threads reached wait() — if the notifying thread called notify() when the wait set was empty, the notification was lost. No thread stored the notification — there's no memory of "a notify was sent." Threads that reach wait() afterward sleep forever. Diagnosis: add timestamps to notify() and wait() calls; if notify consistently precedes wait, restructure to use a boolean flag that is set before notify() and checked before wait(). Root cause 2: Exception swallowed in the notifying thread — the notifying thread threw an exception before reaching notify(), but the exception was swallowed (catch(Exception e) {}) and the notify() was never called. Diagnosis: add logging to exception handlers; jstack will show the notifying thread terminated or stuck. Root cause 3: Wrong object — wait() and notify() called on different objects (e.g., this.wait() in one method and otherObj.notify() in another). The notified wait set has no waiters; the actual waiters are on a different object's wait set and never woken. Diagnosis: jstack shows threads waiting on a specific object monitor address; verify the notify() is called on the exact same object address.

Q7: In the producer-consumer pattern, the consumer calls notify() after draining the buffer. Could this lead to the producer being starved if there are multiple consumers? Walk through the scenario precisely.

Yes. Scenario: 1 producer thread, 3 consumer threads. Buffer capacity = 5. State: buffer full. Producer has called wait() and is in the wait set. All 3 consumers compete for the lock. Consumer 1 acquires the lock, enters consume(), drains the buffer, calls notify(). notify() wakes exactly one of the other waiting threads — but who's waiting? The producer (now buffer is empty) and potentially consumers 2 and 3 (if they also called wait() when the buffer was empty before the producer filled it). If notify() wakes Consumer 2 or Consumer 3 instead of the Producer, that consumer re-checks its condition (while buffer.size() < capacity), finds it true (buffer IS empty), and goes back to wait(). Now Consumer 2 is in the wait set. Consumer 1's monitor is released. Consumer 2 or 3 was woken but went back to waiting. The producer is STILL in the wait set, never woken. Consumer 2 or 3's wait() call produces another wait(). Now producer + 2 consumers are all in the wait set, no thread running. The system deadlocks with no thread remaining to call notify(). This is the precise failure mode of notify() with multiple thread types. Fix: notifyAll() ensures the producer is always eventually woken; or use ReentrantLock with separate Condition objects (producerCondition.signal() from consumer, consumerCondition.signal() from producer) for perfectly targeted notification.
