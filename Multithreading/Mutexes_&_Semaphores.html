Java Multithreading: Mutexes & Semaphores

Principal Architect Notes — Production-Grade Mastery Series

Syllabus Coverage: Lectures 40–43 (What are Semaphores? · Mutexes and Semaphores · Semaphores Example · Mutex and Binary Semaphore)

1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:
A mutex is a synchronization construct that grants exclusive, ownership-bound access to a single critical section (which is exactly what synchronized and ReentrantLock already provide), while a semaphore generalizes this idea using a counter to allow a configurable number of threads to access a resource pool simultaneously, with no concept of ownership.

The Architecture Analogy

Picture a building with restricted-access rooms.

A mutex is a single-occupancy meeting room with one key. Only one person can be inside at a time, and critically — the room's policy states that only the person who took the key may return it. If someone else tries to return the key on your behalf, the building manager rejects it: an error. This ownership rule is the defining feature of a mutex.

A binary semaphore is the same single-occupancy room, but with no name attached to the key. Whoever picks it up can use the room, and — crucially — anyone can return the key to the box, not necessarily the person who took it. The room still only fits one person at a time (binary: 0 or 1 available), but there's no ownership enforcement.

A counting semaphore is a parking garage with, say, three available spots and a sign at the entrance showing the live count. Three cars can park simultaneously. The fourth car waits at the barrier until a spot opens. Critically — the garage attendant doesn't care which specific car uses which specific spot, or even which car leaves first — it only tracks how many spots are occupied, not who occupies them.

How the JVM Sees It

Dijkstra's semaphore (1965) predates Java by decades — it's an operating-systems-level concept that Java's java.util.concurrent.Semaphore class implements faithfully. Internally, Semaphore is built on AQS (AbstractQueuedSynchronizer) — the same foundation as ReentrantLock — but instead of a binary state (0 = unlocked, 1 = locked), it uses state as a non-negative integer permit count. acquire() atomically decrements the count via CAS (retrying if contended); if the count would go negative, the calling thread is parked in the AQS wait queue. release() atomically increments the count and, if threads are waiting, unparks one of them.

The crucial JVM-level distinction from ReentrantLock: AQS tracks an "exclusive owner thread" for ReentrantLock (used for re-entrancy counting and for throwing IllegalMonitorStateException if a non-owner calls unlock()). Semaphore does not track an owner thread at all — acquire() and release() can be called from completely different threads with no error, no ownership check. This is not a missing feature; it is the fundamental design distinction between the two constructs, faithfully reproduced from Dijkstra's original specification.

2. Course Content: Key Concepts & Definitions

Why Semaphores Exist — Historical Context

Edsger Dijkstra introduced semaphores in 1965 as part of his foundational work on synchronization in concurrent systems (the same Dijkstra known for the shortest-path algorithm). Before semaphores, there was no clean, formal, language-agnostic way to safely manage shared resources and critical sections among concurrent processes. The semaphore concept became — and remains — a fundamental building block underlying synchronization primitives in virtually every operating system and concurrent programming language, including Java's own synchronized and Lock implementations conceptually descend from this lineage.

A semaphore is fundamentally a counter with two atomic operations:

  • acquire() (historically "P", from Dutch proberen, "to test"): decrements the count; if the count would go below zero, the calling thread blocks until a permit becomes available
  • release() (historically "V", from Dutch verhogen, "to increment"): increments the count, potentially waking a blocked thread

Binary Semaphore vs. Counting Semaphore

Binary Semaphore Counting Semaphore
Permit range 0 or 1 only Any non-negative integer
Use case Enforce mutual exclusion (only one thread in the critical section) Limit concurrent access to N identical resources
Behaves like A lock A resource pool with a fixed capacity
Example Protecting a single shared counter Limiting concurrent database connections to 10 
// Binary semaphore — acts like a lock (mutual exclusion)
Semaphore binarySem = new Semaphore(1);   // exactly one permit

// Counting semaphore — limits concurrent access to a pool of resources
Semaphore connectionPool = new Semaphore(10);   // ten permits — e.g., 10 DB connections

Critical conceptual point from the course material: semaphores only track how many permits are available — they do not track which specific resources are free or in use. If you have a counting semaphore with 10 permits representing 10 database connections, the semaphore itself doesn't know or care which specific connection object a thread is using — that bookkeeping (matching threads to actual connection instances) is the application's responsibility, typically paired with a separate pool data structure (e.g., a BlockingQueue<Connection>).

Java's Semaphore Class

import java.util.concurrent.Semaphore;

// Constructor: number of permits, and optional fairness flag
Semaphore semaphore = new Semaphore(3, true);
// 3 permits available; fair = true means longest-waiting thread gets the next permit

Core methods:

Method Behavior
acquire() Blocks until a permit is available, then takes it (decrements count). Throws InterruptedException.
release() Returns a permit (increments count), potentially unblocking a waiting thread
tryAcquire() Non-blocking attempt; returns true/false immediately
tryAcquire(timeout, unit) Blocks up to the timeout, then gives up if no permit becomes available
availablePermits() Returns the current count of available permits

Fairness, exactly as with ReentrantLock: new Semaphore(n, true) guarantees FIFO ordering — the longest-waiting thread acquires the next available permit, preventing starvation. The default (false) allows "barging" — a newly arriving thread can acquire a permit that just became available even if other threads have been waiting longer.

Complete Worked Example: Limiting Concurrent Downloads

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Semaphore;

public class Downloader {

    // Singleton pattern via enum
    public enum Instance {
        INSTANCE;

        private final Semaphore semaphore = new Semaphore(3, true);  // only 3 concurrent downloads

        public void download() {
            try {
                semaphore.acquire();     // blocks if 3 downloads are already in progress
                try {
                    downloadData();
                } finally {
                    semaphore.release();  // ALWAYS release — even on exception
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }

        private void downloadData() {
            System.out.println(Thread.currentThread().getName() + " downloading data from the web");
            try {
                Thread.sleep(2000);   // simulate a 2-second download
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }

    public static void main(String[] args) {
        ExecutorService service = Executors.newCachedThreadPool();

        // 12 threads competing for only 3 download "slots"
        for (int i = 0; i < 12; i++) {
            service.execute(Downloader.Instance.INSTANCE::download);
        }
        service.shutdown();
    }
}

What happens at runtime: 12 threads are submitted, but only 3 can call downloadData() at any given moment — the other 9 block inside acquire() until a permit is released. As each download finishes and calls release(), the next waiting thread (chosen fairly, since fair=true) acquires the freed permit and proceeds. This pattern is exactly how production systems throttle concurrent access to constrained resources — database connection pools, rate-limited external APIs, limited worker slots for CPU-intensive tasks.

The mandatory try/finally pattern for release(): identical reasoning to ReentrantLock.unlock(). If downloadData() throws and release() isn't in a finally block, the permit is never returned — the pool of available slots silently shrinks by one forever (a "permit leak"), eventually starving all threads as every permit gets consumed and never returned.

Mutex vs. Semaphore — The Complete Comparison

Dimension Mutex Semaphore
Purpose Mutual exclusion — exactly one thread in the critical section Signaling and resource counting — N threads in the critical section
Permit count Always exactly 1 (locked/unlocked) Any non-negative integer
Ownership Has ownership — only the thread that locked it may unlock it No ownership — any thread can call acquire() or release(), regardless of which thread acquired which permit
What it protects A critical section / race condition Access to a limited pool of resources
Blocking condition Blocks if already locked Blocks if available permit count is 0
Release rule Must be released by the thread that acquired it Can be released by ANY thread
Java equivalents synchronized block/method, ReentrantLock java.util.concurrent.Semaphore 
// Mutex (via synchronized) — only the holding thread's own code path reaches the unlock point
synchronized (this) {
    // critical section — implicitly "released" only by the same thread exiting the block
}

// ReentrantLock — explicit ownership enforcement
lock.lock();
try { /* ... */ } finally { lock.unlock(); }
// If a DIFFERENT thread calls lock.unlock() without having called lock() first:
// → throws IllegalMonitorStateException — ownership is enforced

// Semaphore — NO ownership enforcement
Semaphore sem = new Semaphore(1);   // binary semaphore — acts like a mutex superficially
// Thread A:
sem.acquire();
// Thread B (a completely different thread!):
sem.release();    // ← perfectly legal! No exception, no ownership check

This last example is the single most important practical distinction: a binary semaphore can be acquired by one thread and released by a completely different thread with no error — this is by design and is sometimes exactly what you want (e.g., one thread produces a "permit" event that releases another waiting thread — a signaling pattern), but it is also a foot-gun if you intended mutual exclusion and accidentally allowed cross-thread release.

synchronized and ReentrantLock ARE Mutexes — Naming the Pattern

The course material makes an important conceptual unification: synchronized blocks and ReentrantLock are not "similar to" a mutex — they are mutex implementations, specifically. When you use synchronized(this), the object's intrinsic lock/monitor functions exactly as a mutex: only one thread at a time can execute the protected section, and the lock is conceptually "owned" by the entering thread (re-entrant for that same thread, blocking for any other thread). This naming exercise matters for interviews and cross-language conversations — "mutex" is the general computer science term; synchronized/ReentrantLock are Java's specific implementations of that general concept.

Mutex vs. Binary Semaphore — The Subtle but Critical Difference

These two are often used interchangeably in casual conversation because they behave almost identically in the simple case — both allow exactly one thread access at a time. But the ownership distinction is fundamental and not cosmetic:

"Ownership is the concept that when a task locks (acquires) a mutex, only that task is allowed to unlock (release) it." — if a synchronization construct lacks ownership semantics, it is not a true mutex, regardless of what it's called.

// Mutex (ReentrantLock) — ownership enforced
ReentrantLock mutex = new ReentrantLock();
mutex.lock();   // Thread A acquires
// ... if Thread B calls mutex.unlock() without ever calling lock() first:
// → IllegalMonitorStateException — the JVM REJECTS the unlock attempt
//   and CRITICALLY: the mutex remains locked. The error doesn't silently succeed.

// Binary Semaphore — no ownership
Semaphore binarySem = new Semaphore(1);
binarySem.acquire();   // Thread A acquires the single permit
// Thread B calls:
binarySem.release();   // succeeds with NO error — now 2 "permits" conceptually exist
                        // even though only 1 was ever defined — this can corrupt
                        // the intended invariant of "at most 1 thread in the critical section"

Why this matters in production: if you mistakenly use a binary semaphore where a true mutex was needed, a bug elsewhere in the code (a thread releasing a permit it never acquired, perhaps due to a logic error) will silently succeed and corrupt your mutual-exclusion invariant — potentially allowing 2 threads into a section meant to be strictly single-threaded, with no exception, no warning, no signal that anything went wrong. With a true mutex (ReentrantLock), the same bug throws IllegalMonitorStateException immediately, fails loudly, and — critically — the lock remains in its correct (locked) state, preserving the mutual exclusion invariant even in the presence of the bug.

When to use which:

  • Use a mutex (synchronized/ReentrantLock) when protecting a critical section where you want ownership enforcement as a correctness safety net
  • Use a binary semaphore for pure signaling or coordination between threads where ownership isn't conceptually meaningful — e.g., a producer thread "releasing" a permit to wake a consumer thread that never "acquired" anything itself in the traditional sense

3. Senior-Level Practical Implementation

Real-World Use Case 1: Database Connection Pool Throttling

import java.util.concurrent.Semaphore;
import java.util.ArrayDeque;
import java.util.Queue;

public class ConnectionPool {
    private final Semaphore availablePermits;
    private final Queue<Connection> connections = new ArrayDeque<>();
    private final Object poolLock = new Object();

    public ConnectionPool(int poolSize) {
        this.availablePermits = new Semaphore(poolSize, true);
        for (int i = 0; i < poolSize; i++) {
            connections.offer(new Connection("conn-" + i));
        }
    }

    public Connection acquire() throws InterruptedException {
        availablePermits.acquire();   // blocks if pool is fully checked out
        synchronized (poolLock) {
            return connections.poll();   // semaphore guarantees a connection IS available here
        }
    }

    public void release(Connection conn) {
        synchronized (poolLock) {
            connections.offer(conn);
        }
        availablePermits.release();   // signal that a permit (and connection) is available again
    }

    record Connection(String id) {}

    public static void main(String[] args) throws InterruptedException {
        ConnectionPool pool = new ConnectionPool(5);   // 5 connections max

        Runnable task = () -> {
            try {
                Connection conn = pool.acquire();
                System.out.println(Thread.currentThread().getName() + " using " + conn.id());
                Thread.sleep(500);   // simulate query execution
                pool.release(conn);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        };

        for (int i = 0; i < 20; i++) {
            new Thread(task, "client-" + i).start();
        }
        // Only 5 threads execute queries at any given moment; the other 15 wait their turn
    }
}

Note the design: the Semaphore controls how many threads may proceed, while a separate synchronized Queue tracks which specific connection objects are available — exactly matching the course material's point that semaphores count permits but don't track identity.

Real-World Use Case 2: API Rate Limiting with tryAcquire

import java.util.concurrent.Semaphore;
import java.util.concurrent.TimeUnit;

public class ApiRateLimiter {
    private final Semaphore rateLimitPermits;

    public ApiRateLimiter(int maxConcurrentCalls) {
        this.rateLimitPermits = new Semaphore(maxConcurrentCalls, true);
    }

    /**
     * Attempts the API call only if a permit is available within the timeout.
     * Returns a fallback response immediately if the system is overloaded —
     * does NOT make the caller wait indefinitely.
     */
    public String callExternalApi(String request) {
        try {
            if (rateLimitPermits.tryAcquire(200, TimeUnit.MILLISECONDS)) {
                try {
                    return performApiCall(request);
                } finally {
                    rateLimitPermits.release();
                }
            } else {
                return "FALLBACK: rate limit exceeded, try again later";
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return "ERROR: interrupted";
        }
    }

    private String performApiCall(String request) {
        // actual external HTTP call
        return "response for " + request;
    }
}

This pattern is exactly how production circuit-breaker and bulkhead patterns (in resilience libraries like Resilience4j) implement concurrency limiting under the hood — a semaphore-based bulkhead caps concurrent calls to a downstream dependency, protecting both the caller (no unbounded thread growth waiting on a slow dependency) and the callee (no overwhelming traffic).

4. Deep Dive: Tradeoffs, Architecture & Performance

Performance: Semaphore vs. ReentrantLock for the Binary Case

For the n=1 binary case, Semaphore and ReentrantLock have very similar performance characteristics since both are built on AQS. The meaningful difference is not raw performance but semantic safety: ReentrantLock's ownership enforcement converts certain classes of bugs (cross-thread unlock) from silent state corruption into loud, immediate exceptions. For genuine resource-pool throttling (n > 1), Semaphore is the only directly appropriate tool — there is no clean way to express "allow up to N concurrent threads" using ReentrantLock alone without manually building a counting mechanism on top of it (which is, in effect, reimplementing Semaphore).

Tradeoff Matrix: Mutex vs. Binary Semaphore vs. Counting Semaphore

Mutex (ReentrantLock) Binary Semaphore Counting Semaphore
Correctness safety net High — ownership violations throw immediately None — silent misuse possible None — silent misuse possible
Use case fit Protecting a critical section with a clear "owner" thread Cross-thread signaling without ownership semantics Throttling access to N identical resources
Risk of misuse Lower — JVM catches ownership errors Higher — a stray release() corrupts the invariant silently Higher — same risk, amplified across N permits
Idiomatic Java choice for mutual exclusion ✅ Preferred ❌ Avoid unless ownership-free signaling is the actual intent N/A — different use case entirely

Common Pitfalls (Senior Level)

1. Forgetting release() in a finally Block — Permit Leak

semaphore.acquire();
doWork();                 // throws an exception
semaphore.release();      // NEVER REACHED — permit is lost forever

// CORRECT
semaphore.acquire();
try {
    doWork();
} finally {
    semaphore.release();   // always executes
}

Over time, repeated permit leaks shrink the available pool toward zero, eventually causing every thread to block indefinitely — a slow-motion deadlock that's especially hard to diagnose because it doesn't manifest immediately; the system degrades gradually as permits leak away one exception at a time.

2. Calling release() More Times Than acquire() — Over-Releasing

semaphore.acquire();    // count: 2 → 1
semaphore.release();    // count: 1 → 2
semaphore.release();    // BUG: count: 2 → 3 — exceeds the intended maximum!
// Now MORE threads than intended can hold permits simultaneously,
// silently violating the resource limit the semaphore was meant to enforce

Unlike ReentrantLock, Semaphore does not throw an exception for this — release() can be called any number of times by any thread, and the permit count simply grows beyond its original maximum. This is a direct consequence of the no-ownership design and is a real production risk if release logic has any bugs (e.g., a retry path that calls release() twice).

3. Using a Semaphore When a True Mutex Was Needed for Correctness-Critical Code

If the code's correctness depends on strict ownership enforcement (e.g., you want a bug where Thread B incorrectly releases Thread A's lock to fail loudly rather than silently corrupt state), prefer ReentrantLock over Semaphore(1) even though they behave similarly in the happy path. The lack of an exception on misuse is a meaningful difference under bugs and edge cases, not just a stylistic one.

4. Confusing "Available Permits" with "Currently Executing Threads"

availablePermits() reflects the count, not which specific threads or resources are involved. In a connection pool scenario, availablePermits() == 3 tells you 3 connections are free, but says nothing about which 3 connections, or whether the application's separate connection-tracking data structure agrees with that count — a logic bug in the pairing code (like the example above) can desynchronize the semaphore's count from the actual resource pool's state.

5. Interview Questions & Answers

Basic

Q1: What is the fundamental difference between a mutex and a semaphore?

A mutex provides mutual exclusion for a single resource — exactly one thread may hold it at a time, and critically, it has ownership semantics: only the thread that acquired the mutex may release it; attempting to release from a different thread is rejected with an error, and the mutex remains correctly locked. A semaphore is a more general construct that maintains a count of available permits — any non-negative integer, not just 0 or 1 — allowing a configurable number of threads to access a resource concurrently, and crucially has no ownership: any thread can call acquire() or release(), regardless of which thread did which operation previously. In Java, synchronized and ReentrantLock are mutex implementations; java.util.concurrent.Semaphore is the semaphore implementation, usable in both binary mode (acts like a lock) and counting mode (throttles access to N resources).

Q2: What is a binary semaphore and how does it differ from a true mutex despite both allowing only one thread at a time?

A binary semaphore is a Semaphore initialized with exactly 1 permit — superficially it behaves like a mutex, allowing only one thread into the critical section at a time. The crucial difference is ownership: a true mutex enforces that only the thread that acquired the lock can release it (the JVM throws IllegalMonitorStateException if a different thread attempts to unlock), preserving the mutual exclusion invariant even in the presence of a bug. A binary semaphore has no such enforcement — release() can be called by any thread, including one that never called acquire(), with no error. This makes binary semaphores appropriate for cross-thread signaling (where one thread intentionally "releases" a permit for another thread that never held it) but inappropriate as a drop-in replacement for a mutex when correctness depends on ownership being enforced.

Q3: What does Semaphore.acquire() do if no permits are available, and what does release() do?

acquire() attempts to atomically decrement the semaphore's internal permit count. If the count is already at zero, the calling thread blocks (parks in the AQS wait queue, consuming zero CPU) until another thread calls release(), making a permit available. release() atomically increments the permit count and, if any threads are waiting, makes one of them eligible to proceed (the specific thread chosen depends on the fairness setting — FIFO if fair=true, unspecified otherwise). Both acquire() and release() can be called from any thread — Semaphore does not track which thread acquired which permit, unlike ReentrantLock's ownership tracking.

Intermediate

Q4: You need to limit concurrent access to an external API to at most 5 simultaneous calls across your entire application, regardless of which threads make the calls. Would you use ReentrantLock or Semaphore, and why?

Semaphore, specifically new Semaphore(5, true). ReentrantLock only models a binary state — locked or unlocked, exactly one holder at a time — there's no clean way to express "up to 5 concurrent holders" with a single ReentrantLock; you'd have to manually build a counter on top of it (essentially reimplementing what Semaphore already provides correctly). Semaphore is purpose-built for exactly this throttling scenario: initialize with 5 permits, have every API-calling code path wrap the call in acquire()/release() (with release() in a finally block to prevent permit leaks on exception), and the semaphore automatically ensures no more than 5 threads execute the protected section concurrently, queuing (or, with tryAcquire(timeout), timing out) the rest. The fairness flag (true) ensures threads are served in arrival order, preventing any caller from being starved indefinitely under sustained load.

Q5: A teammate proposes using Semaphore(1) instead of ReentrantLock to protect a critical section, arguing they're functionally equivalent. What risk are they introducing, and how would you explain it concretely?

They're introducing the loss of ownership enforcement. In the happy path, Semaphore(1) and ReentrantLock behave identically — exactly one thread in the critical section at a time. The risk surfaces under bugs: with ReentrantLock, if a coding error causes a thread to call unlock() without having called lock() (e.g., an exception handling path that unconditionally calls cleanup code, including unlock, even on a path where the lock was never acquired), the JVM throws IllegalMonitorStateException immediately — the bug is caught loudly, and critically, the lock state remains internally consistent (still correctly locked if it was locked). With Semaphore(1), the equivalent bug — a stray release() call from a thread that never called acquire() — succeeds silently with no exception. The permit count increments beyond its intended maximum of 1, meaning two threads can now simultaneously enter what was supposed to be a strictly single-threaded critical section — exactly the race condition the synchronization was meant to prevent, now silently reintroduced by a release-path bug. For correctness-critical mutual exclusion, the immediate, loud failure of ReentrantLock is a meaningful safety property that Semaphore(1) does not provide.

Advanced

Q6: Design a scenario where using a binary semaphore's lack-of-ownership property is actually the CORRECT design choice, not a bug risk.

The canonical case is cross-thread signaling, where the semantics genuinely require one thread to "produce" a permit for another thread to "consume," without the producing thread ever holding the resource itself. Example: a worker thread pool where a dedicated "initializer" thread performs a one-time setup task, and N worker threads must wait until setup is complete before proceeding. Using Semaphore(0) (starts with zero permits): all N worker threads call acquire() immediately on startup and block (since the count is 0). The initializer thread, upon completing setup, calls release(N) (or release() N times) — releasing N permits at once, unblocking all N waiting workers simultaneously. Critically, the initializer thread never called acquire() — it only ever calls release(). This is fundamentally NOT a mutual-exclusion scenario; it's a one-directional signal-and-proceed pattern, which is exactly what CountDownLatch is purpose-built for (a later topic), but demonstrates precisely why semaphores were designed without ownership: many legitimate concurrency patterns are not mutual exclusion at all, but signaling, where the very idea of "the thread that acquired must be the one to release" doesn't apply because no single thread ever "acquires" in the traditional sense — the relationship is inherently asymmetric (one producer of permits, many consumers).

Q7: A counting semaphore initialized with 10 permits is protecting a connection pool, but over several days of production operation, availablePermits() trends toward zero even though connection usage patterns haven't changed. What is the most likely root cause and how do you confirm it?

This is the signature of a slow permit leak — somewhere in the code, a code path acquires a permit (acquire()) but fails to release it (release()) under certain conditions, most commonly an exception path where release() was not placed in a finally block, or a conditional early-return that skips the release call. Each occurrence permanently reduces the effective pool size by one; over days of operation with intermittent exceptions, this compounds toward zero. Confirmation: (1) Instrument the code to log every acquire() and release() call with a correlation ID, and over a time window, count totals — a persistent imbalance (more acquires logged than releases) confirms the leak and its rate. (2) Audit every code path between acquire() and the corresponding release() for exception safety — specifically searching for any acquire() not immediately followed by a try { ... } finally { release(); } structure. (3) Use availablePermits() exposed via a metrics endpoint or JMX, graphed over time — a steady downward trend (rather than oscillation around a stable baseline) confirms systemic leakage rather than normal load variation. The fix is always structural: audit and correct every acquire()/release() pairing to guarantee the try/finally pattern, with no exceptions for "this path can't throw" assumptions — those assumptions are exactly where leaks originate as code evolves over time.

  • Design a legitimate ownership-free signaling pattern using Semaphore(0) where one thread produces permits and others consume them, recognizing this as fundamentally different from mutual exclusion
