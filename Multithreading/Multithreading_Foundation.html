1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:

Multithreading is the ability of a single JVM process to run multiple independent units of execution (threads) that share the same heap, enabling concurrent task processing that keeps applications responsive and CPUs fully utilized.

The Architecture Analogy

Think of a busy restaurant kitchen. The process is the kitchen itself — one physical space with shared resources (counter space, equipment, pantry). Threads are the individual chefs working inside it.

A single-threaded kitchen has one chef doing everything sequentially: prep → cook → plate → wash. The moment that chef is waiting for the oven to preheat (a network download, a file copy), the entire kitchen is frozen. Nobody else can cook.

A multi-threaded kitchen assigns different chefs to different stations. While Chef A waits for the oven (I/O wait), Chef B is plating and Chef C is prepping the next dish. The kitchen never freezes — but now you need house rules: who controls the one shared knife, who owns the stove at any given moment. Those house rules are what synchronization (coming in later lectures) will be all about.

The time-slicing algorithm is like having a single shared stove burner with a 30-second timer. Each chef gets the burner for 30 seconds, then hands it off. With one burner, the kitchen feels productive because chefs are always doing something. Add more burners (CPU cores) and chefs genuinely cook in parallel — no waiting.

How the JVM Sees It

At the JVM level, each Java thread maps 1:1 to a a native OS thread — on Linux this is a pthread, on Windows a Win32 thread. When you call thread.start(), the JVM internally calls pthread_create() (Linux) or CreateThread() (Windows), which asks the OS to schedule a new kernel thread.

What each thread owns exclusively (private per thread):

Resource What it holds
PC Register (Program Counter) The address of the current bytecode instruction being executed
JVM Stack Stack frames for every method call: local variables, operand stack, return addresses
Native Method Stack For any native (JNI) method calls

What all threads in the same process share:

Resource Why it matters
Heap All object instances live here — this shared space is the root cause of all concurrency bugs
Metaspace Class definitions, method bytecode, static variables
String Pool Interned string literals

The key mental model: stacks are private, heap is shared. A local variable inside a method (int x = 5) lives on the thread's private stack — completely safe. An object on the heap that two threads both hold a reference to — that's where bugs are born.

The Java Memory Model (JMM) — preview only, deep coverage in later lectures: The JMM governs visibility of writes across threads. Without synchronization, Thread A writing to a shared field is not guaranteed to be visible to Thread B. This is not a bug — it is defined, legal JVM behavior that enables JIT compiler and CPU optimizations. We'll handle this properly when we reach synchronized, volatile, and locks.

2. Course Content: Key Concepts & Definitions

Core Terminology

  • Sequential Execution: The default Java execution model. The JVM executes bytecode instructions one at a time, top to bottom. Each statement must complete before the next begins. Every Java application starts with a single thread: the main thread.
  • Process: An instance of a running program managed by the OS. The OS assigns it:
    • Its own registers
    • Its own stack memory
    • Its own heap memory
    • A unique PID (Process Identifier)

    Processes are isolated — one process cannot directly read another process's memory. Creating a process is expensive: the OS must allocate all the above from scratch, and every process must have a parent process (forming a directed graph in the OS process tree). In Java: ProcessBuilder lets you spawn OS processes, but this is rarely the right tool for within-JVM concurrency.

  • Thread: A lightweight execution unit that lives inside a process. A process can contain one or many threads. Threads within the same process share heap and other resources. Creating a thread is cheaper than creating a process because:
    • No new heap allocation needed (shared with the process)
    • No parent-child process graph management
    • Only a new stack and PC register must be allocated
  • Multithreading: Running multiple threads concurrently within a single process, either via time-slicing on one core or true simultaneous execution across multiple cores.
  • Concurrency vs. Parallelism:
    • Concurrency: Multiple threads make progress — but not necessarily at the exact same instant. On a single core with time-slicing, threads are concurrent but not parallel.
    • Parallelism: Multiple threads execute at the exact same instant on separate CPU cores. Parallelism is a subset of concurrency.
  • Context Switch: The CPU operation of suspending the current thread (saving its PC, stack pointer, and all registers) and loading the saved state of the next thread. This is the hidden cost of time-slicing — it takes real CPU cycles and pollutes CPU caches.

The Time-Slicing Algorithm

When there are more threads than CPU cores, the OS scheduler uses time-slicing to share the CPU:

Single CPU, Three Threads:

Time ─────────────────────────────────────────────────────►
      │  Thread 1  │  Thread 2  │  Thread 3  │  Thread 1  │ ...
      └────────────┴────────────┴────────────┴────────────┘
           ▲              ▲
           └── context    └── context
               switch         switch
               (overhead)     (overhead)

Each thread gets a time quantum (typically 10–100ms depending on the OS). When its quantum expires, the CPU saves its state, picks the next ready thread, loads its state, and resumes. This creates the illusion of parallel execution on a single core.

With multiple cores, the OS can assign one thread per core — no time-slicing needed for those threads. They execute truly simultaneously:

Core 0: │── Thread 1 ──────────────────────────────────────►
Core 1: │── Thread 2 ──────────────────────────────────────►
Core 2: │── Thread 3 ──────────────────────────────────────►

If you have more threads than cores, the OS combines both strategies: some threads run truly in parallel, and threads beyond the core count are time-sliced onto available cores.

Important: Java gives you no control over which core a thread runs on or exactly when it gets scheduled. The JVM delegates entirely to the OS scheduler. This non-determinism is a fundamental truth of multithreaded programming — never write code that depends on threads executing in a specific order without explicit coordination.

Advantages of Multithreading

  1. Responsive Applications (the primary motivation)

    In a single-threaded app, a time-consuming operation (downloading a 10GB dataset, copying a large directory, waiting for a database query) blocks everything. The UI freezes, no other logic runs, the user waits with no feedback.

    The fix: run the time-consuming operation on a worker thread, keeping the main thread free to handle user interactions. The OS uses time-slicing to interleave them — the main thread stays responsive even though the worker is busy.

    Single-threaded (broken):
Main: [init] ──── [downloading... 20 minutes ...] ──── [build model] ──── [predict]
                   ▲ UI is completely frozen here

Multi-threaded (correct):
Main:   [init] ──── [UI stays responsive] ──────────────────────────────────────────►
Worker:              [downloading...]──────────────[build model]──────[predict]

  2. Better CPU Utilization

    By default, a Java app has one thread → uses one CPU core → leaves all other cores idle. Even on a 16-core machine, a single-threaded app uses ~6% of available compute. Multiple threads let the OS distribute work across cores.

  3. True Parallelism on Multi-Core Hardware

    For problems that can be split into independent sub-problems (more on this in the parallelism and Fork-Join lectures later in the course), multiple threads can run on separate cores and complete the work in a fraction of the time.

Disadvantages of Multithreading

  1. Shared Memory → Synchronization Complexity

    All threads share the heap. If Thread A and Thread B both read and write the same variable without coordination, the result is undefined — the data can become inconsistent. This is called a race condition. Solving race conditions requires synchronization mechanisms (covered in upcoming lectures), which add complexity and their own failure modes.

  2. Hard to Design, Debug, and Test

    Multithreaded bugs are non-deterministic — they depend on the exact scheduling order the OS happens to pick. A bug may only appear under heavy load, disappear when you add a print statement (because println takes time and shifts timing), and never reproduce on your laptop but consistently crash in production. These are called heisenbugs.

  3. Context-Switch Overhead Makes More Threads ≠ More Speed

    Every thread switch has a cost. With too many threads, the CPU spends more time saving/restoring state than doing actual work. The relationship between thread count and performance looks like this:

    Performance
    ▲
    │         ●
    │      ●     ●
    │   ●            ●
    │ ●                  ●        ●
    │                        ●         ● (worse than single-threaded!)
    └─────────────────────────────────────►  Thread count
          ▲           ▲
     sweet spot    too many threads

    A rule of thumb: for small problems, single-threaded is faster. For parallelizable problems, a small number of threads (typically close to the number of CPU cores) gives maximum benefit. Beyond that, performance degrades. The exact optimal point is workload-dependent — the parallelism and performance chapters later in this course cover how to measure it.

State-by-state breakdown:

State How you enter it CPU consumption Notes
NEW new Thread(task) — object created, start() not yet called None Thread exists as a Java object but has no OS thread yet
RUNNABLE thread.start() called Minimal OS thread created; ready to run but may be waiting its turn in the scheduler queue
RUNNING OS scheduler picks this thread Active Executing bytecode instructions; this is the only state where real work happens
BLOCKED / WAITING sleep(), wait(), join(), or blocked on a lock Zero Thread is parked; OS scheduler skips it entirely until the condition resolves
TERMINATED run() returns normally or throws an unhandled exception None Thread cannot be restarted; the Thread object becomes eligible for GC

The most important insight about BLOCKED/WAITING: a thread in this state consumes zero CPU cycles. The OS scheduler simply doesn't schedule it. This is what makes it possible for one thread to wait on a slow network call without degrading the performance of other threads — the waiting thread is invisible to the CPU until it has work to do again.

State transitions that appear in later lectures (noted here for context):

  • sleep(ms) → WAITING → back to RUNNABLE after the duration (lecture 27)
  • wait() → WAITING → back to RUNNABLE after notify() (lecture 27)
  • join() → WAITING until target thread terminates → back to RUNNABLE (lecture 14)
  • Blocked on synchronized lock → BLOCKED → RUNNABLE when lock is released (lecture 22)

3. Senior-Level Practical Implementation

Real-World Use Case 1: Stock Market App — Keeping the UI Alive

A trading application downloads a 2GB historical dataset from Yahoo Finance for model training. On the main (UI) thread, this freezes the entire app. The correct architecture:

public class StockApp {

    public static void main(String[] args) {
        System.out.println("App starting on: " + Thread.currentThread().getName());

        // Worker thread handles the slow operation
        Thread dataWorker = new Thread(() -> {
            System.out.println("Download starting on: " + Thread.currentThread().getName());
            downloadDataset("https://finance.yahoo.com/api/historical");   // slow — 2-3 min
            buildModel();
            makePredictions();
            System.out.println("Worker done.");
        });

        // Give the thread a meaningful name — critical for debugging thread dumps in production
        dataWorker.setName("data-download-worker");
        dataWorker.start();  // returns immediately — main thread is NOT blocked

        // Main thread stays responsive — can handle UI events, heartbeats, etc.
        System.out.println("Main thread free. Showing loading indicator...");
        showLoadingIndicator();   // runs concurrently with the download
    }

    static void downloadDataset(String url)  { /* network I/O — slow */ }
    static void buildModel()                 { /* computation */ }
    static void makePredictions()            { /* inference */ }
    static void showLoadingIndicator()       { /* UI update loop */ }
}

What happens at the OS level: dataWorker.start() causes the JVM to call pthread_create(), creating a second kernel thread. The OS time-slices both threads on available cores. The main thread runs its loading indicator loop; the worker thread independently progresses through the download. If you have 2+ cores, they may run simultaneously.

Real-World Use Case 2: Parallel Directory Copy — OS Behavior in Action

The course example of copying a directory without freezing the OS explorer is a perfect real-world illustration. Here's the programmatic equivalent — a file copy that runs without blocking the calling context:

import java.io.*;
import java.nio.file.*;

public class NonBlockingFileCopy {

    // Returns immediately; copy proceeds on a background thread
    public static void startCopyAsync(Path source, Path destination) {
        Thread copyThread = new Thread(() -> {
            try {
                System.out.println("[" + Thread.currentThread().getName() + "] Copy started");
                copyDirectory(source, destination);
                System.out.println("[" + Thread.currentThread().getName() + "] Copy complete");
            } catch (IOException e) {
                // In production: publish to an error queue or callback
                System.err.println("Copy failed: " + e.getMessage());
            }
        }, "file-copy-worker");    // named thread — always name your threads

        copyThread.start();        // fire and continue; caller is not blocked
    }

    private static void copyDirectory(Path src, Path dst) throws IOException {
        Files.walk(src).forEach(sourcePath -> {
            try {
                Path targetPath = dst.resolve(src.relativize(sourcePath));
                Files.copy(sourcePath, targetPath, StandardCopyOption.REPLACE_EXISTING);
            } catch (IOException e) {
                throw new UncheckedIOException(e);
            }
        });
    }

    public static void main(String[] args) {
        startCopyAsync(Path.of("/data/source"), Path.of("/data/dest"));

        // Main thread continues immediately — the copy is happening in the background
        System.out.println("Copy running in background. Main thread free.");
        // ... handle other work here ...
    }
}

What NOT to Do: The Single-Threaded Anti-Pattern

// ANTI-PATTERN: Everything on the main thread
public class BadStockApp {
    public static void main(String[] args) {
        initArrays();              // fast
        downloadDataset();         // ← blocks for 20 minutes. UI frozen. User panics.
        buildModel();              // only runs after download completes
        makePredictions();         // only runs after model build completes
        // The user has been staring at a frozen screen this entire time.
    }
}

The sequential version isn't just slower from a UX perspective — it's architecturally fragile. A network timeout during the download crashes the entire app with no opportunity for the main thread to handle errors gracefully or show meaningful feedback.

Always Name Your Threads

This is a small practice with outsized production impact. When you take a thread dump (jstack <pid>) during a production incident, named threads immediately tell you what's happening:

// Bad thread dump output:
"Thread-0" RUNNABLE
"Thread-1" WAITING
"Thread-2" BLOCKED

// Good thread dump output:
"data-download-worker" RUNNABLE
"model-training-worker" WAITING
"prediction-engine" BLOCKED   ← instantly clear which subsystem has the problem

// Always provide a name when constructing a Thread
Thread t = new Thread(task, "descriptive-name-here");

// Or set it after construction
t.setName("order-processor-" + orderId);

4. Deep Dive: Tradeoffs, Architecture & Performance

Performance Implications of Multithreading

  • Context-Switch Cost: On Linux with HotSpot JVM, each context switch costs ~1–10 microseconds for register save/restore, plus indirect costs from CPU pipeline flushing and L1/L2 cache invalidation (the new thread's working data must be loaded into cache). For tight loops doing microsecond-level work, this overhead can dominate.
  • Thread Stack Memory: Each thread's stack defaults to 512KB–1MB (configurable via -Xss). This is native memory (outside the Java heap). With 1,000 threads: 500MB–1GB of native memory just for stacks, before any heap object allocation. This is a hard operational ceiling for raw thread-per-task models.

The Performance Curve — Why More Threads ≠ More Speed

The relationship is not linear. It has three zones:

Zone Thread Count What's Happening
Under-utilized 1 to ~nCores Extra threads fill idle cores — genuine speedup
Sweet spot ~nCores to 2×nCores All cores busy, minimal context switching
Degraded Much more than nCores Context-switch overhead dominates; GC pressure from Thread objects rises; may be slower than single-threaded

The exact sweet spot depends on whether your work is CPU-bound (tight computations — thread count ≈ nCores) or I/O-bound (waiting on network/disk — threads spend most time in WAITING, so more threads are tolerable). Measuring is always better than guessing.

Tradeoff Analysis: Process vs. Thread

Dimension Process Thread
Creation cost High (full memory allocation, OS bookkeeping) Low (new stack + PC only)
Memory isolation Complete (crash in one process doesn't touch others) None (crash in one thread can corrupt shared heap)
Communication Expensive (IPC: pipes, sockets, shared memory segments) Free (direct heap access — but needs synchronization)
Use case OS-level isolation, different runtimes, security boundaries Concurrent tasks within one application
Java API ProcessBuilder new Thread(...), Runnable

For in-JVM concurrency, threads are always the right tool. Processes are for when you need crash isolation or are calling code in a different language/runtime.

Common Pitfalls at This Stage

1. Calling run() instead of start()

This is the most common beginner mistake and one of the most important to internalize early:

Thread t = new Thread(() -> System.out.println("Thread: " + Thread.currentThread().getName()));

t.run();    // WRONG: executes run() on the CURRENT thread — no new thread is created
            // Output: "Thread: main"

t.start();  // CORRECT: asks the JVM to create a new OS thread and execute run() on it
            // Output: "Thread: Thread-0"

When you call run() directly, you're just calling a method. No new thread. No OS thread creation. The start() method is what triggers pthread_create() internally.

2. Assuming Thread Execution Order

Thread t1 = new Thread(() -> System.out.println("T1"));
Thread t2 = new Thread(() -> System.out.println("T2"));
t1.start();
t2.start();
// Output is non-deterministic: could be T1 then T2, or T2 then T1
// NEVER rely on this ordering without explicit synchronization

The OS scheduler decides execution order. It can — and will — vary between runs, between machines, and between JVM versions. Code that accidentally depends on a particular ordering is a latent bug waiting for production conditions to reveal it.

3. Forgetting That the Main Thread Can End Before Worker Threads

public static void main(String[] args) {
    Thread worker = new Thread(() -> {
        try { Thread.sleep(2000); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        System.out.println("Worker done");  // May or may not print
    });
    worker.start();
    System.out.println("Main done");
    // main() returns here — but the JVM keeps running because 'worker' is still alive
    // (User threads keep the JVM alive; this is covered in the Daemon Threads lecture)
}

The JVM does not exit when main() returns — it exits when all user threads finish. (Daemon threads are an exception — covered in lecture 18.)

4. Not Naming Threads

Anonymous threads named Thread-0, Thread-1 in production thread dumps during incidents waste critical debugging time. Always name threads with their purpose.

Architectural Impact: Why This Foundation Matters for Everything That Follows

Every advanced concurrency topic in this course — synchronization, locks, volatile, thread pools, Fork-Join, reactive streams, virtual threads — exists because of exactly two facts established in this section:

  1. Threads share the heap → you need synchronization to protect shared data
  2. Creating threads has cost → you need thread pools and higher abstractions to manage lifecycle efficiently

Understanding why these problems exist (shared memory + context-switch overhead) is what separates engineers who cargo-cult synchronized from those who design correct, performant concurrent systems. Every tool introduced in later lectures is a direct solution to a specific consequence of these two facts.

5. Interview Questions & Answers

Basic

Q1: What is the difference between a process and a thread?
A process is an OS-managed instance of a running program with its own isolated memory space — separate registers, stack, and heap. No two processes share memory by default. A thread is a lightweight execution unit that lives inside a process, sharing the process's heap and resources with sibling threads. Creating a process is expensive because the OS must allocate all resources from scratch and manage parent-child relationships. Creating a thread is cheap — only a new stack and program counter register are needed. The shared heap is what makes threads powerful for communication but requires careful synchronization to prevent data corruption.
Q2: What are the thread lifecycle states in Java and what triggers each transition?
Java threads have four phases. NEW: the Thread object is instantiated but start() hasn't been called — no OS thread exists yet. RUNNABLE (with two sub-states): after start() is called, the OS thread is created; the thread is either waiting in the scheduler queue (runnable) or actively executing on a CPU (running) — Java's Thread.State API exposes these as one RUNNABLE state. BLOCKED/WAITING: entered via sleep(), wait(), join(), or lock contention — the thread parks, consuming zero CPU, until the condition resolves. TERMINATED: run() has returned or an uncaught exception was thrown — the OS thread is destroyed, and the Thread Java object becomes GC-eligible. A thread cannot be restarted once terminated; you must create a new Thread instance.
Q3: What is the time-slicing algorithm and when does the OS use it?
Time-slicing is the OS scheduler's strategy for sharing a single CPU core among multiple ready threads. Each thread gets a short burst of CPU time (a time quantum, typically 10–100ms), then the CPU saves that thread's state (registers, PC, stack pointer) and loads the state of the next thread in the ready queue. This creates the illusion of simultaneous execution on a single core. The OS uses time-slicing when there are more runnable threads than available CPU cores. When threads ≤ cores, the OS can assign one thread per core for true parallel execution. In Java, this distinction matters because you have no control over which mode is in effect — the JVM never exposes CPU affinity, and thread scheduling is entirely delegated to the OS.

Intermediate

Q4: A colleague says "I added 50 threads to my algorithm and it got slower." What are the two most likely causes?
First, context-switch saturation: with 50 threads competing on, say, 8 cores, the CPU is spending significant time saving and restoring thread state (1–10µs per switch, polluting L1/L2 caches) instead of executing the algorithm. The more threads than cores, the worse this gets. Second, the algorithm may not be sufficiently parallelizable: if most of the algorithm is sequential (data must be processed in order, or one step produces input for the next), adding threads just adds overhead without reducing the critical path. The fix is first to profile — find out whether threads are spending time computing or context-switching — then size the thread count to roughly match the number of CPU cores for CPU-bound work, and validate with benchmarks. (Formal analysis of parallelizability is covered later in Amdahl's Law.)
Q5: Why does calling run() directly on a Thread object work without throwing an exception, yet produce completely wrong behavior?
Because Thread implements Runnable, and run() is just a regular method. Calling t.run() directly invokes that method on the calling thread — identical to calling any other method. No OS thread is created, no JVM thread is registered, and Thread.currentThread() inside run() returns the caller's thread, not t. The program appears to work (the code inside run() executes), but it executes sequentially on the current thread with zero concurrency benefit. This is a silent bug — no exception, no warning, just wrong behavior that can pass unit tests and only reveal itself as a performance problem under load. start() is what triggers pthread_create() under the hood and registers the thread with the JVM's thread scheduler.
Q6: The main thread's main() method returns, but the JVM keeps running. Why?
The JVM's exit policy is: terminate when all user threads have finished, regardless of whether main() has returned. If main() starts worker threads and then returns, the JVM stays alive until all those workers complete. This is important to understand because it means resource leaks in worker threads (open connections, locks held, infinite loops) will prevent JVM shutdown indefinitely. The complement of this — daemon threads — are threads the JVM will not wait for. If only daemon threads remain when main() exits, the JVM terminates immediately, potentially killing those threads mid-execution. (Daemon threads are covered in lecture 18.)

Advanced

Q7: Explain why thread execution order is non-deterministic in Java and what production risks this creates.
Java deliberately does not specify a thread scheduling algorithm — the JVM delegates to the host OS scheduler. On Linux, this is the Completely Fair Scheduler (CFS); on Windows, a priority-based preemptive scheduler; each with different behaviors. Thread priorities in Java (setPriority()) are hints only — some OS schedulers honor them, others ignore them. The result: the order in which threads execute, and the exact interleaving of their instructions, is undefined and can differ between runs, machines, OS versions, and JVM implementations. Production risk: code with hidden ordering dependencies (e.g., Thread A assumes Thread B has already written a value to a shared variable) will pass all tests on a developer's laptop (where timing happens to produce the "right" order) and fail intermittently in production under different load or on different hardware. These bugs — called heisenbugs — are extraordinarily expensive to diagnose. The engineering response is: never assume execution order; always use explicit synchronization mechanisms to establish ordering guarantees. (What those mechanisms are is the entire subject of the next section of this course.)
Q8: Walk through what happens at the OS and JVM level when thread.start() is called on a Java thread.
When thread.start() is called: (1) The JVM checks that the thread is in NEW state — calling start() on an already-started thread throws IllegalThreadStateException. (2) The JVM calls the native method Thread.start0(), which is implemented in the JVM's C++ layer. (3) In HotSpot on Linux, this calls pthread_create(), creating a new POSIX kernel thread and registering it with the OS scheduler. The OS assigns the new thread a kernel thread ID and places it in the READY queue. (4) The JVM allocates a new JVM stack for the thread (default 512KB–1MB depending on platform and -Xss setting) and sets its Program Counter to the entry point of the run() method. (5) start() returns immediately on the calling thread — it does not wait for the new thread to begin executing. (6) The OS scheduler, at its discretion, begins executing the new thread's run() method. From this point, two threads are running concurrently. The key insight: there is a window between start() returning and the new thread's first instruction executing — code in that window in the calling thread happens-before the new thread's run() body, per the JMM, so any data set up before start() is safely visible to the new thread without additional synchronization.
Q9: A junior engineer on your team says: "I profiled our app and saw that our 16-thread pool is spending 40% of CPU time on context switching. Should we reduce threads to 1?" How do you respond?
Reducing to 1 thread would eliminate context-switch overhead but also eliminate all concurrency benefits — and the root cause is likely not thread count but thread behavior. First, diagnose what those 16 threads are actually doing: if they are CPU-bound tasks (computation, sorting, mathematical operations), 16 threads on, say, an 8-core machine means 2 threads per core constantly competing, and reducing to 8 threads would likely cut context-switch overhead by ~50% while preserving almost all throughput. If the threads are I/O-bound (waiting on network or database), the 40% context-switching figure is suspicious — I/O threads spend most time in WAITING state and should not be generating high context-switch rates. That pattern instead suggests threads are spinning (busy-waiting in a loop rather than blocking) — a separate and more serious bug. The correct response is: profile with OS-level tools (vmstat, perf stat -e context-switches) to measure actual switch rate, correlate with jstack thread states, identify whether threads are CPU-bound or I/O-bound, and then size the pool to nCores for CPU-bound work, or investigate the spinning pattern for I/O-bound work. Guessing thread count without measurement is how you trade one performance problem for another.

6. Mastery Checklist

After internalizing these notes, you should be able to:

  • Explain to a junior engineer the exact memory layout difference between a process and a thread — what each owns exclusively (stack, PC register) versus what threads share (heap, Metaspace) — and why the shared heap is the root of all concurrency complexity
  • Draw the complete Java thread lifecycle from NEW to TERMINATED, correctly describe what triggers each transition, and explain why BLOCKED/WAITING threads consume zero CPU
  • Describe the time-slicing algorithm precisely: what the OS saves/restores during a context switch, when time-slicing is needed vs. when true parallelism is possible, and why Java gives you no control over which mode is in effect
