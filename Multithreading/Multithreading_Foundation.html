1. Core Concept & Simplified Theory (The "Elevator Pitch")

One-Sentence Definition:

Multithreading is the ability of a single JVM process to run multiple independent units of execution (threads) that share the same heap, enabling concurrent task processing that keeps applications responsive and CPUs fully utilized.

The Architecture Analogy

Think of a busy restaurant kitchen. The process is the kitchen itself — one physical space with shared resources (counter space, equipment, pantry). Threads are the individual chefs working inside it.

A single-threaded kitchen has one chef doing everything sequentially: prep → cook → plate → wash. The moment that chef is waiting for the oven to preheat (a network download, a file copy), the entire kitchen is frozen. Nobody else can cook.

A multi-threaded kitchen assigns different chefs to different stations. While Chef A waits for the oven (I/O wait), Chef B is plating and Chef C is prepping the next dish. The kitchen never freezes — but now you need house rules: who controls the one shared knife, who owns the stove at any given moment. Those house rules are what synchronization (coming in later lectures) will be all about.

The time-slicing algorithm is like having a single shared stove burner with a 30-second timer. Each chef gets the burner for 30 seconds, then hands it off. With one burner, the kitchen feels productive because chefs are always doing something. Add more burners (CPU cores) and chefs genuinely cook in parallel — no waiting.

How the JVM Sees It

At the JVM level, each Java thread maps 1:1 to a a native OS thread — on Linux this is a pthread, on Windows a Win32 thread. When you call thread.start(), the JVM internally calls pthread_create() (Linux) or CreateThread() (Windows), which asks the OS to schedule a new kernel thread.

What each thread owns exclusively (private per thread):

Resource What it holds
PC Register (Program Counter) The address of the current bytecode instruction being executed
JVM Stack Stack frames for every method call: local variables, operand stack, return addresses
Native Method Stack For any native (JNI) method calls

What all threads in the same process share:

Resource Why it matters
Heap All object instances live here — this shared space is the root cause of all concurrency bugs
Metaspace Class definitions, method bytecode, static variables
String Pool Interned string literals

The key mental model: stacks are private, heap is shared. A local variable inside a method (int x = 5) lives on the thread's private stack — completely safe. An object on the heap that two threads both hold a reference to — that's where bugs are born.

The Java Memory Model (JMM) — preview only, deep coverage in later lectures: The JMM governs visibility of writes across threads. Without synchronization, Thread A writing to a shared field is not guaranteed to be visible to Thread B. This is not a bug — it is defined, legal JVM behavior that enables JIT compiler and CPU optimizations. We'll handle this properly when we reach synchronized, volatile, and locks.

2. Course Content: Key Concepts & Definitions

Core Terminology

  • Sequential Execution: The default Java execution model. The JVM executes bytecode instructions one at a time, top to bottom. Each statement must complete before the next begins. Every Java application starts with a single thread: the main thread.
  • Process: An instance of a running program managed by the OS. The OS assigns it:
    • Its own registers
    • Its own stack memory
    • Its own heap memory
    • A unique PID (Process Identifier)

    Processes are isolated — one process cannot directly read another process's memory. Creating a process is expensive: the OS must allocate all the above from scratch, and every process must have a parent process (forming a directed graph in the OS process tree). In Java: ProcessBuilder lets you spawn OS processes, but this is rarely the right tool for within-JVM concurrency.

  • Thread: A lightweight execution unit that lives inside a process. A process can contain one or many threads. Threads within the same process share heap and other resources. Creating a thread is cheaper than creating a process because:
    • No new heap allocation needed (shared with the process)
    • No parent-child process graph management
    • Only a new stack and PC register must be allocated
  • Multithreading: Running multiple threads concurrently within a single process, either via time-slicing on one core or true simultaneous execution across multiple cores.
  • Concurrency vs. Parallelism:
    • Concurrency: Multiple threads make progress — but not necessarily at the exact same instant. On a single core with time-slicing, threads are concurrent but not parallel.
    • Parallelism: Multiple threads execute at the exact same instant on separate CPU cores. Parallelism is a subset of concurrency.
  • Context Switch: The CPU operation of suspending the current thread (saving its PC, stack pointer, and all registers) and loading the saved state of the next thread. This is the hidden cost of time-slicing — it takes real CPU cycles and pollutes CPU caches.

The Time-Slicing Algorithm

When there are more threads than CPU cores, the OS scheduler uses time-slicing to share the CPU:

Single CPU, Three Threads:

Time ─────────────────────────────────────────────────────►
      │  Thread 1  │  Thread 2  │  Thread 3  │  Thread 1  │ ...
      └────────────┴────────────┴────────────┴────────────┘
           ▲              ▲
           └── context    └── context
               switch         switch
               (overhead)     (overhead)

Each thread gets a time quantum (typically 10–100ms depending on the OS). When its quantum expires, the CPU saves its state, picks the next ready thread, loads its state, and resumes. This creates the illusion of parallel execution on a single core.

With multiple cores, the OS can assign one thread per core — no time-slicing needed for those threads. They execute truly simultaneously:

Core 0: │── Thread 1 ──────────────────────────────────────►
Core 1: │── Thread 2 ──────────────────────────────────────►
Core 2: │── Thread 3 ──────────────────────────────────────►

If you have more threads than cores, the OS combines both strategies: some threads run truly in parallel, and threads beyond the core count are time-sliced onto available cores.

Important: Java gives you no control over which core a thread runs on or exactly when it gets scheduled. The JVM delegates entirely to the OS scheduler. This non-determinism is a fundamental truth of multithreaded programming — never write code that depends on threads executing in a specific order without explicit coordination.

Advantages of Multithreading

  1. Responsive Applications (the primary motivation)

    In a single-threaded app, a time-consuming operation (downloading a 10GB dataset, copying a large directory, waiting for a database query) blocks everything. The UI freezes, no other logic runs, the user waits with no feedback.

    The fix: run the time-consuming operation on a worker thread, keeping the main thread free to handle user interactions. The OS uses time-slicing to interleave them — the main thread stays responsive even though the worker is busy.

    Single-threaded (broken):
Main: [init] ──── [downloading... 20 minutes ...] ──── [build model] ──── [predict]
                   ▲ UI is completely frozen here

Multi-threaded (correct):
Main:   [init] ──── [UI stays responsive] ──────────────────────────────────────────►
Worker:              [downloading...]──────────────[build model]──────[predict]

  2. Better CPU Utilization

    By default, a Java app has one thread → uses one CPU core → leaves all other cores idle. Even on a 16-core machine, a single-threaded app uses ~6% of available compute. Multiple threads let the OS distribute work across cores.

  3. True Parallelism on Multi-Core Hardware

    For problems that can be split into independent sub-problems (more on this in the parallelism and Fork-Join lectures later in the course), multiple threads can run on separate cores and complete the work in a fraction of the time.

Disadvantages of Multithreading

  1. Shared Memory → Synchronization Complexity

    All threads share the heap. If Thread A and Thread B both read and write the same variable without coordination, the result is undefined — the data can become inconsistent. This is called a race condition. Solving race conditions requires synchronization mechanisms (covered in upcoming lectures), which add complexity and their own failure modes.

  2. Hard to Design, Debug, and Test

    Multithreaded bugs are non-deterministic — they depend on the exact scheduling order the OS happens to pick. A bug may only appear under heavy load, disappear when you add a print statement (because println takes time and shifts timing), and never reproduce on your laptop but consistently crash in production. These are called heisenbugs.

  3. Context-Switch Overhead Makes More Threads ≠ More Speed

    Every thread switch has a cost. With too many threads, the CPU spends more time saving/restoring state than doing actual work. The relationship between thread count and performance looks like this:

    Performance
    ▲
    │         ●
    │      ●     ●
    │   ●            ●
    │ ●                  ●        ●
    │                        ●         ● (worse than single-threaded!)
    └─────────────────────────────────────►  Thread count
          ▲           ▲
     sweet spot    too many threads

    A rule of thumb: for small problems, single-threaded is faster. For parallelizable problems, a small number of threads (typically close to the number of CPU cores) gives maximum benefit. Beyond that, performance degrades. The exact optimal point is workload-dependent — the parallelism and performance chapters later in this course cover how to measure it.

State-by-state breakdown:

State How you enter it CPU consumption Notes
NEW new Thread(task) — object created, start() not yet called None Thread exists as a Java object but has no OS thread yet
RUNNABLE thread.start() called Minimal OS thread created; ready to run but may be waiting its turn in the scheduler queue
RUNNING OS scheduler picks this thread Active Executing bytecode instructions; this is the only state where real work happens
BLOCKED / WAITING sleep(), wait(), join(), or blocked on a lock Zero Thread is parked; OS scheduler skips it entirely until the condition resolves
TERMINATED run() returns normally or throws an unhandled exception None Thread cannot be restarted; the Thread object becomes eligible for GC

The most important insight about BLOCKED/WAITING: a thread in this state consumes zero CPU cycles. The OS scheduler simply doesn't schedule it. This is what makes it possible for one thread to wait on a slow network call without degrading the performance of other threads — the waiting thread is invisible to the CPU until it has work to do again.

State transitions that appear in later lectures (noted here for context):

  • sleep(ms) → WAITING → back to RUNNABLE after the duration (lecture 27)
  • wait() → WAITING → back to RUNNABLE after notify() (lecture 27)
  • join() → WAITING until target thread terminates → back to RUNNABLE (lecture 14)
  • Blocked on synchronized lock → BLOCKED → RUNNABLE when lock is released (lecture 22)
